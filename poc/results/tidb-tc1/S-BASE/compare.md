### VM 部署（Variant: vm）

```
172.24.40.31 (client)
     │  go-tpc → :4000
     ▼
172.24.40.34  HAProxy :4000
     │  roundrobin
     ├─▶ 172.24.40.32  TiDB + PD + TiKV
     └─▶ 172.24.40.33  TiDB + PD + TiKV
         172.24.40.34            PD + TiKV

processes: 直接跑於 OS，無容器層
TiKV RF=3，每節點 30 GB data dir
```

### K8s 部署（Variant: k8s-unlimit / k8s-limit）

```
172.24.40.31 (client)
     │  go-tpc → :30004
     ▼
172.24.40.32  NodePort :30004
     │
     │  k3s cluster
     ├─ poc-1  k3s server  ┐
     ├─ poc-2  k3s agent   ├─ TiDB Operator v1.6.5
     └─ poc-3  k3s agent   ┘
          │
          ├─ PD   × 3  req: 500m/1Gi   limit: 1c/2Gi   PV: 10Gi  (local-path)
          ├─ TiDB × 2  req: 500m/1Gi   limit: 1c/3Gi
          └─ TiKV × 3  req:   1c/4Gi   limit: 2c/8Gi   PV: 100Gi (local-path)

k8s-unlimit: 不設 limits（移除 limit 欄位）
k8s-limit  : limits 如上表（反映實際資源上限）
overhead   : k3s control plane 跑於 poc-1，佔約 1 vCPU / 1~2 GB RAM（納入比較）
```

| 128     | —                            | —              | —    |

---

### 量測二：K8s-unlimit → K8s-limit（resource limits 影響）

> 引入 CPU/Memory limits（TiKV: 2c/8Gi、TiDB: 1c/3Gi、PD: 1c/2Gi）。
> 預期：低 thread 數無感；高 thread 數（≥ 64）若 TiKV CPU 觸頂則 tpmC 下降、p99 上升。

| threads | k8s-unlimit tpmC | k8s-limit tpmC | 比值   | tpmC 損耗 | NO p99 delta (ms) | 判定 |
|---------|-----------------|----------------|--------|-----------|-------------------|------|
| 16      | 13,668          | 11,156         | 0.816  | **-18.4%** | +39.9            | ⚠️ 超預期 |
| 32      | 16,992          | 11,508         | 0.677  | **-32.3%** | +96.4            | ❌ 嚴重 |
| 64      | 18,456          | 11,729         | 0.635  | **-36.5%** | +134.2           | ❌ 嚴重 |
| 128     | 18,842          | 11,823         | 0.628  | **-37.2%** | +268.4           | ❌ 嚴重 |

**結論**：limits 設定過嚴，TiKV 2c 上限已成瓶頸。tpmC 在 limit 模式下從 16t 起即出現 18% 損耗，
64t 以上穩定在 -37% 且 tpmC 幾乎不增長（11,729 → 11,823），確認 CPU ceiling 已觸頂。
k8s-limit 配置不適合此工作負載，生產部署需調高 TiKV limits 或改用 Guaranteed QoS。

## 綜合結論

| 比較組 | tpmC 損耗 | p99 影響 | 備註 |
|--------|----------|---------|------|
| VM → K8s-unlimit | 待補 | 待補 | 純容器層開銷 |
| K8s-unlimit → K8s-limit | **-18% ~ -37%** | +40 ~ +268ms | TiKV 2c limit 觸頂 |
| VM → K8s-limit | 待補 | 待補 | 實際生產部署總損耗 |
