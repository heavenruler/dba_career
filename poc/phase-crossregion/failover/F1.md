> 實跑啟用需另開 PR + DBA review。

## Scenario

IDC 側目前負責 leader 的 DB process 被刻意停掉（計畫性 / admin-triggered），
觀察 GCP follower 是否能在 raft 重新選舉後接手 leader，
並把第一筆 NEW_ORDER 寫入完成所需 wall-clock 時間量化為 RTO。
RPO 則量 kill 前 5 秒已 commit 的 transaction 是否在切換後仍可在 GCP-side 查到。

非範圍：
- 不含非預期 failure（屬 chaos C1/C4/C7）
- 不含 GCP → IDC 反向（屬另一支 F1 反向 case；同 spec 結構）
- 不含 sweep / thread 變化（F1 專做 sub-second precision）

## 量測指標

### RTO (Recovery Time Objective)
- 起點 `t_kill`：planner 印出的 kill 指令**若實際執行**的本機 wall-clock
- 終點 `t_first_write_gcp`：自 GCP-side client 透過 GCP haproxy / GCP endpoint 送出
  第一筆 NEW_ORDER 並收到 commit ack 的時間
- `rto_sec = t_first_write_gcp - t_kill`
- 期望（假設 lab mode；assumption）：
  - TiDB ~10-30s
  - CRDB ~5-15s
  - YBDB ~10-20s

### RPO (Recovery Point Objective)
- 取 kill 前 5 秒間 IDC-side 已收到 commit ack 的 NEW_ORDER `(w_id, d_id, o_id)` 清單為 `S_pre`
- failover 完成後，從 GCP-side 用同一組 key 查詢 `o_id` 是否仍存在為 `S_post`
- `rpo_lost_tx_count = |S_pre - S_post|`
- 期望（lab mode；assumption）：raft 同步 commit 模型下三 DB 應為 0；如非 0 即代表 replication lag exceeded RPO 5s window

## 三 DB 觸發機制（會印出的指令，**目前不執行**）

| DB | leader 停機指令（dry-run 印出用） | 適用 process |
|---|---|---|
| TiDB | `tiup ctl:<ver> pd -u http://<pd-host>:2379 member leader resign` <br>+ `ssh <idc-dbhost-X> 'sudo systemctl stop tidb-server@<port>'` <br>（更小範圍：`tiup cluster restart <cluster> -N idc-dbhost-1` 重啟單節點） | tidb-server / pd-server |
| CRDB | `ssh <idc-dbhost-X> 'cockroach quit --insecure --host=localhost:26257'` <br>（或更輕量 `cockroach node drain <node-id>`） | cockroach |
| YBDB | `yb-admin --master_addresses=<masters> master_leader_stepdown` <br>+ `ssh <idc-dbhost-X> 'sudo systemctl stop yb-master'`（如要連 tserver 一併停：`sudo systemctl stop yb-tserver`） | yb-master / yb-tserver |

備註：
- 上述指令版本 / flag 可能因 release 微調（assumption — F1 實跑前由 DBA 用對應 cluster 版本 confirm）。
- `--kill-target leader-auto` 由 planner 印出「應先用對應 admin CLI 找出當前 leader host」的 lookup 指令，
  並非由 planner 直接呼叫 cluster。

## artifact schema

每次 plan 對應一個 ts；實跑時 artifact 應落在：

```
runs/F1/<ts>/
├── kill.log                  # 實際 kill 指令文字 + 觸發 wall-clock (RFC3339, ms)
├── leader-handover.log       # 5s polling timeline，記錄 (poll_ts, current_leader_host, raft_term)
├── rto-rpo.json              # {"db_kind": "tidb|crdb|ybdb",
│                             #  "rto_sec": <float>,
│                             #  "rpo_lost_tx_count": <int>,
│                             #  "kill_target": "<host>",
│                             #  "kill_cmd": "<verbatim>",
│                             #  "t_kill": "<RFC3339>",
│                             #  "t_first_write_gcp": "<RFC3339>" }
└── db-config-snapshot/
    ├── pre-kill/             # kill 前 cluster config dump (placement / replicas / leader map)
    └── post-handover/        # GCP 接管後同一組 dump，供 diff
```

`leader-handover.log` polling 範例（5s interval）：

```
2026-06-15T10:00:00Z leader=idc-dbhost-1 term=42
2026-06-15T10:00:05Z leader=<none>       term=42  # 觸發選舉中
2026-06-15T10:00:10Z leader=gcp-dbhost-2 term=43
```

## 監測流程（dry-run 階段只印，不跑）

1. **t-30s** 起 IDC client 跑穩態 TPC-C 100 TPS（A-S profile），記錄 commit 序列
2. **t=0** 印出（不執行）對應 DB 的 kill 指令
3. **t+0..60s** 每 5s 對 admin endpoint 查 leader（指令依 DB family；見上表）
4. **t+0..60s** GCP-side client 持續嘗試送 NEW_ORDER；第一筆成功時間記為 `t_first_write_gcp`
5. **t+60s** dump db-config-snapshot/post-handover
6. **t+60s** 對 `S_pre`（kill 前 5s commit 過的 tx）走 GCP-side query 計算 `rpo_lost_tx_count`
7. 輸出 `rto-rpo.json`
