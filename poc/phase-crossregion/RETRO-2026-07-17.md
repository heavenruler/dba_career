# phase-crossregion 實驗設計獨立復盤（2026-07-17）

> 產出方式：由未參與專案的獨立審查 agent 閱讀結案報告、SESSION-HISTORY、
> decisions、gitbook 05/09/10/16 章與 manifest 後產出；經 grilling 對話
> 逐項審議，採納範圍見 decisions-2026-06-08.md 同日拍板節。

**結論：設計紀律罕見地高，但 gate 屬「事故驅動補洞」、批次效應未受控是兩大結構性風險。**

## A. 驗收 gate 體系的完備性

**現況評價**：三重驗證（placement gate / GCP replica gate / GCP probe 斷言）+ fail-closed 已是高水準，但演化模式是「每次事故補一層」——tpcc 表→系統層、設定面→資料面、probe 自身失效，三次都是漏網後才補。缺一張「應驗物件 × 驗證面 × 時點」的覆蓋矩陣，無法預先看出下一個洞。

**建議**：
- 【高/S】建立 gate 覆蓋矩陣文件：列（tpcc 表、系統層物件、client 主機、量測工具自身）×（設定面、資料面、執行面）×（prepare 前、run 中、unfreeze 後）。已發生的七個根因全可映射到空格；P-B 前先掃一輪空格。
- 【高/S】`summary-from-stdout.py` 加 fail-closed 斷言：rounds==5、threads 檔位齊全、每輪 stdout 非空。目前缺輪行為未定義，靜默以 4 輪算 mean 將是下一個「量測工具自身失效」。
- 【高/M】三家系統層物件一次性盤點（TiDB 系統 region、CRDB system ranges、YBDB status/master tablets），決定各自「納入 gate 或明文豁免」——O1 只是這類的第一例，逐案補會再漏。
- 【中/S】unfreeze 後加 after-dump 並與 before diff（現只有 before；TiDB 連 before 都缺，O4）；量測期間 leader 分布至少 run 中抽查一次，不只 prepare 前 + post-run 兩端。
- 【中/S】IDC 端 probe 加對稱的 `fail_count==0` static-check 斷言——目前只斷言 GCP 檔，IDC probe fail-quiet 的劇本與 `.15` 事件完全同構。
- 【中/S】error_count 進 static-check（率超閾值 FAIL 或強制標 caveat）——309 筆錯誤靠 user 肉眼看 raw log 才發現，監看 filter 已證明會漏。
- 【低/S】metrics 檔案完整性斷言：每 host × round × 工具（mpstat/sar）檔案存在且非空，GCP 端尤其（sar-net 曾是零副本的關鍵旁證）。

## B. 統計效度

**現況評價**：R1-R5 mean 主口徑一致性優先，可接受；但 range/mean 對單一 outlier 極敏感——5 輪中 1 輪 dip 就把 CV 從 <10% 撐到 28-42%，「合格判定」實質被單輪雜訊主導。dip 集中在 R2（TiDB t32、YBDB t32）且 TiDB#1 是 R1 偏高後回落，形狀更像「檔位切換後穩態重建」而非 round 太短（warmup 固定 threads=64，與量測檔位 concurrency regime 不同）。

**建議**：
- 【高/S】summary.json 增列 secondary estimator：median、trimmed mean（去頭去尾）、stdev/mean，主口徑不動。P-B 前一次 schema 改動即可，讓 dip 輪的影響可量化而非只能敘述。
- 【高/S】自動 outlier flag：|round−median|/median >20% 即標注，取代人工發現 dip。
- 【中/M】針對 dip 檔位（t32）做 targeted N=3 重跑（每家約 1.5h），一次回答「結構性 vs 偶發」——O2 現在的答案是「時間允許時」，但 dip 已三家四次出現，值得升級。
- 【中/S】round 間加 60-120s idle（讓 compaction/checkpoint 消化），並將檔位切換後首輪標記 `post-transition`，供 secondary 口徑排除分析。
- 【低/M】dip-prone 檔位 rounds 5→7；成本 +20min/檔，僅在 targeted 重跑證實偶發後考慮。

## C. 變因控制鏈（G6）

**現況評價**：四步單變數的鏈式設計本身正確，但隱含「批次不是變數」的假設已被自家證據反駁——TiDB 同參數跨批差 21%（O6），量級可能**大於 placement 效應本身**。若 P-A×A-S 錨點沿用 07-12/07-14 舊 cell 而 P-B 用新批，鏈上第一步的 delta 就無法歸因。現行採用 cell 本身已橫跨兩批（O3）。

**建議**：
- 【高/M】「同批」升為硬性要求（新 decision）：G6 鏈四步同一 DB 必須在同一 VM 生命週期序列內連跑；至少 P-A×A-S 錨點與 P-B×A-S 必須同批，否則 placement 效應與批次效應混淆。
- 【高/S】若全鏈同批成本不可行，每批強制重跑一個 P-A×A-S anchor cell，鏈上結論一律以 within-batch delta 表達，不引跨批絕對值。
- 【中/S】先用既有三個 TiDB cell（07-03 / 07-12 / 07-11 備查）量化跨批 spread，寫進報告當 error bar 下限——這決定「多大的 placement delta 才有意義」。
- 【中/S】A-A-RO 起雙 client 引入 `.15` 為新變因：GCP client 主機納入環境 gate（chrony、client 版本、資源快照）——`.15` 已有「宣稱裝好、實際沒裝」前科。

## D. 工程可靠性

**現況評價**：detached 指揮鏈、lock+pgrep preflight、fail-closed gate 把三類已知單點都收掉了。剩餘風險集中在資料持久性：`.31` artifact 落在 `/tmp`，重開機即滅、fetch 前是唯一份；Mac 端 pipeline log 已實際遺失過一次證據（leader gate console 輸出）；vSphere 夜間斷線連三晚同模式，destroy 欠帳會污染下批 preflight。

**建議**：
- 【高/S】`.31` artifact 路徑搬離 `/tmp`（`/var/lib` 或專用掛載），且每 cell `.suite.done` 後立即 rsync 第三份（不等 phase9 fetch）。
- 【高/S】driver 與 gate 的 console 輸出一律 tee 進 suite 目錄——遺失的「首次輪詢即 100%」證據就是這個洞。
- 【中/S】phase9-destroy 加 retry + 避開夜間維護窗排程；phase1 preflight 加「殘留 VM 偵測」fail-closed（destroy 欠帳直接擋新批）。
- 【中/M】.31 driver 心跳檔 + 簡單監看（driver 死亡數小時無人知的劇本尚未發生，但 detached 化之後這是新暴露面）。
- 【低/S】fetch-receipt 加雙端 checksum 比對。

## E. 報告與證據紀律

**現況評價**：六標籤制、證據矩陣、備查隔離的成本已被七次事故證明合理，總體不算過度工程；SESSION-HISTORY 的誠實揭露（含「已知證據缺口」段）是稀有優點。主要弱點是 §N 引用：報告 07-15 已重構過一次資訊架構，gitbook 09 引 `§2/§5`、報告內大量 `§8 O3` 互引，下次重構必漂移。

**建議**：
- 【高/S】§N 引用改穩定錨點：每節/每 O-item 給不變 ID（O1-O8 已是好例子——ID 化的引用不怕重排），外部文件禁止裸引節號。
- 【中/S】static-check 加報告 link-lint：掃描 gitbook/報告內所有相對連結與 §/O 引用，檔案不存在或錨點消失即 FAIL。
- 【中/S】報告頂部加「引用規則一頁卡」：哪些數字可引、必附哪些 caveat——防下游只抄 §5 表格。
- 【低】標籤制不必擴到 SESSION-HISTORY（會是過度工程），維持現狀即可。

## F. 範圍蔓延風險

**現況評價**：原估全矩陣 42 工作天；實際五週產出 3/18 cells，且大半時間花在 framework debug（有價值，但 P-B/雙 client 是全新路徑，可預期同量級踩坑）。18-cell 矩陣以目前速率不現實；failover/chaos 尚未起步。

**建議**：
- 【高/S】正式把 18 cells 降為 G6 鏈的 12 cells（P-A×A-A-RO、P-A×A-A 共 6 cells 明文砍除）。
- 【高/M】P-B 鏈先單家打通（建議 CRDB：0 error、gate 最成熟）走完 4 步，再決定另兩家跑多深。
- 【中/M】F1 failover 保留且優先級應高於 A-A：RTO/RPO 是跨區 PoC 對業務最有決策價值的產出。chaos 砍 C4/C7，C1 併入 F1 觀察。
- 【中/L】IDC-only 6-node paired control：若「WAN penalty」不需對外宣稱則砍；若導入審查需要此數字，它應排在 A-A 之前。
- 【低】A-A max-contention 可降為單家單檔位示範，不跑全 sweep。

## 若只能做三件事

1. **【C】「同批」硬性化 + anchor cell 規則**——批次效應 21% 已大於待測效應，不控它，P-B 對照鏈跑完也無法歸因。
2. **【A】gate 覆蓋矩陣 + summary parser 缺輪斷言**——把 gate 演化從「事故驅動」轉為「矩陣驅動」；parser 是目前唯一沒有自證機制的量測工具。
3. **【D】artifact 離開 /tmp + console log 落檔**——證據已實際遺失過一次；所有紀律的前提是原始檔還在。
