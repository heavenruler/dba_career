# 分散式資料庫架構 PoC 管理階層報告

> [ITDBA-3596](https://104corp.atlassian.net/browse/ITDBA-3596)

---

## 01　我們在解決什麼問題

**目標：停機維護是 104 的事，不應影響到客戶使用。**

- 現有傳統 HA 架構（單點 Primary / Standby）在維護視窗期間仍有服務中斷風險
- 分散式資料庫可實現「任一節點故障或維護，服務持續」
- PoC 目的：用數據驗證分散式架構是否真的可行及選型建議

---

## 02　評估標的

| 系統 | 相容性 | 核心機制 |
|------|--------|---------|
| **TiDB** | MySQL-compatible | Percolator + PD TSO |
| **YugabyteDB** | PostgreSQL-compatible | DocDB + HLC |

- 同時評估兩條技術路線，避免單一選型受既有習慣限制

**補充：為什麼現在評估分散式資料庫？**

分散式資料庫的導入動機常被誤解為架構冗餘，但對原目標的驅動力是 **產品持續營運的可用性**：

- 現有 Primary/Standby 架構在計劃性維護時仍需停機視窗，影響用戶體驗
- 分散式架構讓 DBA 可逐節點線上維護，業務不感知
- PoC 的前提假設已鎖定：**高併發 OLTP 場景（熱點寫入、多站點讀寫）**，並非泛用性評估

**本次 PoC 未納入協作單位的已知風險**

| 協作單位 | 未協作的影響範圍 | 後續必要動作 |
|----------|----------------|------------|
| **NET** | 專線 IDC↔EDC 實際規格與 QoS 未確認；PoC 延遲數字基於 SD IDC↔GCP 代替，等效性尚未正式背書 | 正式導入前需 NET 確認跨站專線延遲、頻寬上限與 QoS 策略 |
| **HVM** | IDC 側 VM 資源（CPU / RAM / Disk）未協調；PoC 使用現有測試資源，不代表生產規格可直接取用 | 落地計畫需提前確認 IDC 可供分配的裸機或 VM 資源與排程時程 |
| **SE (WEB)** | 流量切換層（LB / DNS / 反向代理）未納入驗證；S2 情境（計劃性零中斷切換）僅驗證 DB 層行為，端對端路徑尚未閉環 | S2 情境的完整驗收需 SE(WEB) 協作確認應用層切換行為 |

## 03　為什麼用 SD IDC ↔ GCP 模擬 IDC ↔ EDC

**短答：環境充分且協作成本最低。**

- Y25 環境測試結論：SD IDC（172.24.\*.\*）↔ GCP（10.160.\*.\*）的線路規格與延遲特性，足以模擬 IDC ↔ EDC 站點間行為
- 改用 IDC ↔ EDC 需額外協調網路、資安、基礎設施等跨單位設定，延長前置時間
- 現有 GCP 環境兼具雲端費用競爭力評估的附加價值（備份策略節費可行性）

## 04　測試基準：TPC-C

**TPC-C 是 OLTP 領域的業界標準壓測基準，與產品業務場景直接對應。**

- 模擬高併發訂單交易：建立訂單、付款、查詢、庫存更新
- 直接量測：吞吐量（tpmC）、交易延遲（p95/p99）、衝突率、故障恢復時間
- 選用理由：結果可與業界 case study 直接對照，管理階層決策有外部參照依據

**補充：為什麼不用 sysbench？**

| | sysbench oltp | TPC-C |
|--|--------------|-------|
| 交易模型 | 單表單行操作（point read / write） | 五種跨表關聯交易（New Order、Payment 等） |
| 讀寫混合 | 可設定但人工，非場景驅動 | 固定比例（45% New Order），反映真實 OLTP 負載 |
| 分散式場景代表性 | 易被單一 region 或 cache 最佳化，數字失真 | 交易依賴關係強制跨節點協調，行為難以迴避 |
| 外部對照 | 原廠少用 sysbench 發布正式 benchmark | TiDB / YugabyteDB 原廠均有 TPC-C 公開數字，可直接對照 |

結論：sysbench 適合快速健康檢查，不適合作為架構選型的決策依據。

## 07　VM 測試結果能否代表未來 K8s 部署？

**本次 PoC 驗證的是「架構行為」，與部署平台無關。**

- PoC 關注：一致性保證、quorum 行為、故障切換時間、split-brain 防護
- 這些特性由資料庫引擎決定，VM / K8s 部署均適用
- VM → K8s 的差異在於：資源排程效率與單叢集效能數字，不影響架構正確性結論
- **結論：VM PoC 結果有效；K8s 部署為後續落地評估的獨立項目**

## 08　後續行動

### 開發工程 → 工程大學

- 分散式資料庫的開發邏輯與傳統單機不同（retry 策略、transaction 設計、hotspot 避免）
- 無論最終選 MySQL-compatible 或 PostgreSQL-compatible，開發端都需要學習曲線抹平
- 建議以工程大學課程形式，提前佈局開發端知識準備

### DBA Team → DBaaS 承接能力建立

- 分散式資料庫的運維模式與傳統 DBA 操作有本質差異（Raft、placement policy、rolling upgrade）
- DBA 週會持續同步 PoC 進度，同步建立操作與故障排查能力
- 目標：PoC 結論出爐時，DBA 已具備基礎承接能力，縮短導入落地時間
