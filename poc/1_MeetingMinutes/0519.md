# 分散式資料庫 PoC 週報 #0519

## 1. 週報摘要

本週重點是收斂 PoC 設計口徑，確認哪些參數會直接影響 TiDB / CockroachDB / YugabyteDB 的 TPC-C-derived stress benchmark 結果。這些參數若遺漏或未對齊，會讓 tpmC、P95/P99、retry/abort rate、scale-out ratio 失去客觀比較基礎。

---

## 2. 關鍵參數

### 2.1 新討論

- **隔離級**：主對標固定 READ COMMITTED；TiDB 需 pessimistic mode，YBDB 需 RC 真正生效，CRDB 需 RC setting。若未生效，retry/abort 行為會污染引擎吞吐觀察。見 `results/PoC-DESIGN.md:106`。
- **RF / shard / replica**：vm-1node=RF1，vm-3node=RF3；vm-3node 必須手動鎖 shard 數，否則無法拆解 sharding 與 replication 成本。見 `results/PoC-DESIGN.md:206`。
- **WAL durable**：三家必須 fsync-on-commit；若某家 WAL 不 durable，tpmC 會偏高但不具可比性。見 `results/PoC-DESIGN.md:50`。
- **Cold reset / round 策略**：每組只做一次 cold reset，5 round 間不重啟；丟 round 1，取 round 2-5 median，避免 cold cache 與 round restart 低估。見 `results/PoC-DESIGN.md:748`。
- **conn-params 控制 isolation**：不使用 go-tpc `--isolation`，避免 driver TxOptions 覆蓋 session default；所有 isolation 走 connection string。見 `results/PoC-DESIGN.md:716`。

### 2.2 基礎對齊

- **Memory budget**：三家統一 DB process 約 11 GB（16 GB * 70%），避免某一家因 cache / heap 額外放大而取得不公平優勢。見 `results/PoC-DESIGN.md:49`。
- **硬體 / OS**：因 YugabyteDB 2025.2 LTS 相依性無法在 AlmaLinux 10.1 安裝，已從 AlmaLinux 10.1 調整為 AlmaLinux 8.10；每節點固定 4 vCPU / 16 GB RAM / 100 GB disk。若規格不同，吞吐與 latency 不可橫向比較。見 `results/PoC-DESIGN.md:43`。
- **工作負載與併發**：固定 128 warehouses，threads 為 16 / 32 / 64 / 128，每水位 5 round；少跑或改 threads 會改變飽和曲線。見 `results/PoC-DESIGN.md:46`。

### 2.3 已知控制項

- **Auto-statistics / ANALYZE**：auto-stats 關閉，prepare 後強制 ANALYZE 並 dump EXPLAIN；避免背景統計或 stale plan 影響結果。見 `results/PoC-DESIGN.md:51`。
- **Warmup**：正式數據維持 20m、warmup_threads=64；5m 或 128 threads 僅能作 quick-run / exploratory，不可混入正式 median。見 `results/PoC-DESIGN.md:770`、`results/PoC-DESIGN.md:780`。

---

## 3. 風險控管

- 若 isolation gate、shard hard gate、OS/chrony/disk gate 檢查未通過，該組應通知後重測。
- 若 warmup 時間、warmup threads、threads list、RF、shard 數、WAL durable、auto-stats 任一項偏離，結果需標記並複測驗證重現性，必要時呈現問題，再確認問題方向及調整策略。
- 報告用語必須維持 stress benchmark / TPC-C-derived workload，不宣稱 audited TPC-C。見 `results/PoC-DESIGN.md:35`。

---

## 4. 本週結論

PoC 的客觀性核心不是單次 tpmC 數字，而是在以下條件一致時產生的可追溯 artifact：

- 同硬體
- 同 workload（已完成重構）
- 同 isolation
- 同 warmup
- 同 durability
- 同統計策略
- 同 shard / replica 控制

後續執行時每組都必須保留 gate / prepare / run / collect marker 與 TPCC_TS lineage，避免設定遺漏後產生無法解釋的數據偏差。
