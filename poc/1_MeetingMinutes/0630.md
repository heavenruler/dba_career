# 2026-06-30 TiDB 資源隔離與跨區就近存取研究

## 1. 目的

整理 TiDB 在多應用、資料熱點與 IDC / GCP 跨專線架構下的資源隔離手段，並釐清下列問題：

- Resource Group、Placement Policy、cgroup 與獨立 Cluster 各自解決什麼問題。
- Placement Policy 能否保證 IDC Request 只在 IDC、GCP Request 只在 GCP。
- PD Leader 固定在 IDC 時，GCP TiDB 查詢 Region metadata 與 TSO 的實際路徑。
- 跨專線是否能只承載 TiKV Raft 資料同步。

> 本文區分「官方機制」與「架構建議」。實際上線前仍須依 TiDB 版本、網路 RTT、RPO / RTO 與故障演練結果拍板。

---

## 2. 先釐清 Region 名詞

| 名詞 | 意義 |
|---|---|
| TiKV Region | TiKV 的資料分片與 Raft Group；依 key range 切分 |
| 地理 Region | IDC、GCP 或不同城市、雲端區域 |

跨專線部署同時涉及兩者：大量 TiKV Region 的 Leader / Follower 分散在不同地理 Region。

---

## 3. 資源隔離分層

| 層級 | 主要機制 | 可控制內容 | 不能保證的內容 |
|---|---|---|---|
| SQL workload | Resource Group、Runaway Query | RU、優先級、流量等待、異常查詢 | 固定 CPU core、磁碟 IOPS、網路頻寬 |
| TiDB process | Memory quota、SQL concurrency、cgroup | 單一查詢 / session 記憶體及 process 上限 | TiKV 資料實體位置 |
| TiKV storage | Label + Placement Policy | Leader / Follower 與資料放置位置 | Request 一定走本地、硬體資源獨占 |
| Cluster | 獨立 TiDB Cluster | 故障域、硬體、網路、維運完整隔離 | 單一 Cluster 的同步強一致體驗 |

核心判讀：

1. Resource Group 解決「誰可以使用多少」。
2. Placement Policy 解決「資料副本放在哪裡」。
3. Schema / key 設計解決「資料是否形成熱點」。
4. cgroup、獨立磁碟與獨立 Cluster 解決硬資源及故障範圍。

### 3.1 Resource Group

TiDB Resource Control 在 TiDB 層使用 token bucket 限流，在 TiKV 層依 Resource Group priority 排程請求。RU 是 CPU、IOPS、IO bandwidth 的抽象成本。

```sql
CREATE RESOURCE GROUP app_critical
RU_PER_SEC = 5000
PRIORITY = HIGH;

CREATE RESOURCE GROUP app_batch
RU_PER_SEC = 500
PRIORITY = LOW;

ALTER USER 'app1'@'%' RESOURCE GROUP app_critical;
ALTER USER 'batch'@'%' RESOURCE GROUP app_batch;
```

注意事項：

- 未設定 `BURSTABLE` 時，RU 不足會等待並可能 timeout。
- 設定 `BURSTABLE` 後可使用剩餘容量，不再是嚴格上限。
- `default` Resource Group 預設為 `UNLIMITED`、`BURSTABLE`；未綁定帳號可能繞過隔離設計。
- `ALTER USER ... RESOURCE GROUP` 只影響新連線，既有 connection pool 需要重建。
- Resource Group 是 QoS 與排程隔離，不是 CPU、IOPS 或 bandwidth 的硬保留。

官方參考：[Resource Control](https://docs.pingcap.com/tidb/stable/tidb-resource-control-ru-groups/)

### 3.2 Runaway Query

```sql
ALTER RESOURCE GROUP app_batch
QUERY_LIMIT=(
  EXEC_ELAPSED='60s',
  ACTION=KILL,
  WATCH=SIMILAR DURATION='10m'
);
```

建議先使用 `DRYRUN` 蒐集實際 SQL，再決定改成 `COOLDOWN` 或 `KILL`，避免正常長查詢被誤殺。

官方參考：[Runaway Queries](https://docs.pingcap.com/zh/tidb/stable/tidb-resource-control-runaway-queries/)

### 3.3 TiDB process 保護

```sql
SET GLOBAL tidb_mem_quota_query = 1073741824;
SET GLOBAL tidb_mem_oom_action = 'CANCEL';
SET GLOBAL tidb_distsql_scan_concurrency = 8;
```

需要同步管控：

- 單一 query / session memory。
- DistSQL scan 與 executor concurrency。
- 大型 transaction。
- ANALYZE / statistics memory。
- Prepared statement 數量及 connection pool 生命週期。

若同一台主機混部 TiDB、PD、TiKV 或其他服務，應使用 cgroup / container limit 限制 CPU 與 memory，並分離資料盤或提供 IOPS 保證。

官方參考：[TiDB OOM 排查](https://docs.pingcap.com/tidb/stable/troubleshoot-tidb-oom/)、[System Variables](https://docs.pingcap.com/tidb/stable/system-variables)、[Background Task Control](https://docs.pingcap.com/tidb/stable/tidb-resource-control-background-tasks/)

### 3.4 TiKV 資料實體隔離

可用 TiKV label 與 Placement Policy 將 Database、Table 或 Partition 放到指定 TiKV Store Group：

```yaml
tikv_servers:
  - host: tikv-ssd-1
    config:
      server.labels:
        app: level1
        storage: ssd
        region: idc

  - host: tikv-hdd-1
    config:
      server.labels:
        app: archive
        storage: hdd
        region: idc
```

```sql
CREATE PLACEMENT POLICY app1_ssd
CONSTRAINTS='[+app=level1,+storage=ssd]'
FOLLOWERS=2;

ALTER DATABASE app1 PLACEMENT POLICY=app1_ssd;
```

限制：

- 這些 TiKV 仍屬同一個 Cluster，共用 TiDB、PD、控制面與部分網路。
- 每個 Store Group 都要有足夠節點滿足 RF，否則 Placement 長期無法收斂。
- Placement Policy 不會預留 CPU、memory 或 IOPS。
- HDD 與 SSD 不宜混在同一份 Raft replica set，commit latency 會受慢速磁碟與 quorum 路徑影響。
- SLA、故障域或硬體等級要求嚴格時，SSD 與 HDD 應拆成不同 TiDB Cluster。

官方參考：[Placement Rules in SQL](https://docs.pingcap.com/tidb/stable/placement-rules-in-sql/)

## 4. 需要隔離或處理的熱點情境

| 情境 | 問題 | 建議處理 |
|---|---|---|
| `AUTO_INCREMENT` / 隱含 RowID | 寫入集中於 key range 尾端 | `AUTO_RANDOM`、`SHARD_ROW_ID_BITS`、pre-split |
| 單調 Secondary Index | 時間、流水號集中寫入少數 Region | 重設 key / index、拆表、觀察 index hot Region |
| 小表高頻讀取 | 少數 Region Leader 形成讀熱點 | Cache、Follower Read、應用快取 |
| Tenant skew | 單一 tenant 壓滿特定 Region | tenant Resource Group + key 分散 |
| 大型 SQL / 大交易 | TiDB memory、TiKV scan 與 lock 放大 | memory quota、concurrency、拆小交易 |
| BR / import / DDL / stats | 背景 CPU、IO 與網路污染線上負載 | Background Control、離峰、獨立執行窗口 |
| Scale-out / 故障恢復 | Region move、snapshot、補副本占用專線 | 等待收斂後再測；另列 rebalance 測項 |

官方參考：[Troubleshoot Hotspot Issues](https://docs.pingcap.com/tidb/stable/troubleshoot-hot-spot-issues/)

## 5. Placement Policy 與「就近存取」

### 5.1 必須修正的觀念

> Placement Policy 控制 Leader / Follower 放置位置，不控制每筆 SQL Request 的路由。

例如：

```sql
CREATE PLACEMENT POLICY geo_policy
PRIMARY_REGION='idc'
REGIONS='idc,gcp'
SCHEDULE='MAJORITY_IN_PRIMARY'
FOLLOWERS=4;
```

代表：

- Region Leader 優先位於 IDC。
- Follower 分布在 IDC / GCP。
- 多數 replica 位於 IDC。

但不代表：

- IDC Request 一定只存取 IDC。
- GCP Request 一定只存取 GCP。
- GCP TiDB 不會連到 IDC TiKV。
- 專線只承載 Raft log。

### 5.2 完整的就近存取條件

```text
IDC Client -> IDC HAProxy -> IDC TiDB [zone=idc]
GCP Client -> GCP HAProxy -> GCP TiDB [zone=gcp]

IDC TiKV [zone=idc]
GCP TiKV [zone=gcp]
```

PD topology 必須包含 `zone`：

```yaml
replication.location-labels:
  - zone
  - host
```

TiDB 與 TiKV 必須有一致的 `zone` label，讀取 Session 再設定：

```sql
SET SESSION tidb_replica_read = 'closest-replicas';
```

官方參考：[Schedule Replicas by Topology Labels](https://docs.pingcap.com/tidb/stable/schedule-replicas-by-topology-labels)、[Follower Read](https://docs.pingcap.com/tidb/stable/follower-read/)

### 5.3 讀取路徑

預設 `tidb_replica_read=leader` 時，即使 GCP 有 Follower，GCP TiDB 仍會存取 IDC Leader：

```text
GCP Client -> GCP TiDB -> IDC TiKV Region Leader
```

設定 `closest-replicas` 後，TiDB 優先選擇相同 `zone` 的 replica：

```text
IDC Client -> IDC TiDB -> IDC TiKV replica
GCP Client -> GCP TiDB -> GCP TiKV replica
```

但這不是完全本地化：

- 強一致 Follower Read 需要 `ReadIndex`，Follower 仍須與 Region Leader 確認 commit progress。
- 本地 replica 不可用時會 fallback 到 Leader。
- Region metadata 過期或 KV retry 時也可能跨區重新選路。
- `closest-replicas` 是優先策略，不是 fail-closed 規則。

若允許資料稍舊，可用 Stale Read 搭配 `closest-replicas`，降低跨區讀取延遲，但必須明確定義可接受的 staleness。

官方參考：[三資料中心就近讀取](https://docs.pingcap.com/zh/tidb/stable/three-dc-local-read)

### 5.4 寫入路徑

寫入一定由該 TiKV Region 的唯一 Leader 處理。若 Leader 全在 IDC：

```text
IDC Client
  -> IDC TiDB
  -> IDC TiKV Leader
  -> Raft replicate to GCP

GCP Client
  -> GCP TiDB
  -> 跨專線到 IDC TiKV Leader
  -> Raft replicate to GCP
```

因此「GCP Request 只在 GCP 內存取」不能適用於同一份資料的強一致寫入。可選方案只有：

1. 依地理 Partition 配置 Leader，讓 GCP 資料的 Leader 位於 GCP。
2. IDC / GCP 使用不同 Cluster，再做非同步資料複寫。
3. 接受 GCP write 經專線送往 IDC Primary Region。

跨 Partition transaction、global index 或跨區查詢仍可能產生跨專線流量。

## 6. PD Leader 固定 IDC 時的實際路徑

### 6.1 TiDB 不會每筆 Row Request 都查 PD

TiDB 會維護 Region Cache：

```text
SQL Request
  -> 查 TiDB 本地 Region Cache
  -> 直接送 TiKV

cache miss / Region split / Leader change / epoch mismatch
  -> 重新查 PD
  -> 更新 Region Cache
  -> retry TiKV request
```

因此 GCP TiDB 不是每筆資料都跨專線詢問 PD，但 cache miss 或 metadata 失效時仍需要 PD。

### 6.2 Region metadata

預設：

```sql
SHOW GLOBAL VARIABLES LIKE 'pd_enable_follower_handle_region';
```

若為 `OFF`，TiDB 只向 PD Leader 取得 Region information；PD Leader 位於 IDC 時，GCP TiDB 的 metadata miss 會跨專線回 IDC。

TiDB 8.5 可啟用 Active PD Follower：

```sql
SET GLOBAL pd_enable_follower_handle_region = ON;
```

啟用後：

- PD Follower 可直接處理 Region information request。
- 若 GCP TiDB 連到 GCP PD Follower，該次 metadata 查詢可由 GCP 回覆。
- Follower metadata 過期或不可用時，TiDB 會 fallback 到 PD Leader。
- 官方機制是將 Region information request 分散到所有 PD，不是嚴格的 zone-local routing。

TiDB 8.5.0 起 Active PD Follower 已為 GA。

官方參考：[Tune Region Performance](https://docs.pingcap.com/tidb/stable/tune-region-performance/)、[TiDB 8.5 Release Notes](https://docs.pingcap.com/tidb/stable/release-8.5.0/)

### 6.3 TSO 是另一條路徑

Transaction 可能需要向 PD 取得全域 TSO。預設情況下，PD Leader 在 IDC 時：

```text
GCP TiDB -> IDC PD Leader -> TSO
```

可評估：

```sql
SET GLOBAL tidb_enable_tso_follower_proxy = ON;
```

這讓 TSO request 分散送到 PD servers，由 PD Follower 接收或代理，但不會在 GCP 產生獨立 TSO。整個 Cluster 仍須維持全域時間順序，因此不能視為完全本地服務。

若 TSO latency 原因是 WAN，而不是 PD Leader CPU，啟用 Follower Proxy 可能增加延遲。應先比較：

- PD TSO Wait Duration。
- PD TSO RPC Duration。
- IDC / GCP TiDB 到各 PD 的 RTT。
- 開啟前後 transaction p95 / p99。

官方參考：[TiDB System Variables](https://docs.pingcap.com/tidb/stable/system-variables)

### 6.4 與「就近存取」是否衝突

不構成功能衝突，但會形成架構目標的缺口：

- Placement / Follower Read 處理 data plane 的副本與讀取路徑。
- PD Region metadata / TSO 屬 control plane。
- 即使資料讀取就近，control plane 仍可能跨專線。

所以「就近存取」應定義為「資料請求優先本地」，不能寫成「所有 Request 與控制流量均不離開當地」。

## 7. 跨專線實際承載流量

「IDC / GCP 專線只處理 TiKV Raft 資料同步」不成立。專線可能承載：

- TiKV Raft log replication。
- Raft snapshot、補副本、Region rebalance。
- GCP TiDB 到 IDC Region Leader 的讀寫。
- Follower Read 的 `ReadIndex`。
- PD Region metadata request / fallback。
- TSO request 或 PD Follower proxy。
- PD 自身 metadata consensus。
- TiKV heartbeat、Store / Region 狀態。
- DDL、GC safepoint 與 scheduling control traffic。
- BR、CDC、監控與維運流量。

因此 WAN QoS、bandwidth 與 failure test 必須獨立設計，Resource Group 不會直接限制專線頻寬。

## 8. 建議架構

```text
Application
  |-- Critical user -> High priority Resource Group
  |-- Normal user   -> Fixed RU Resource Group
  `-- Batch user    -> Low priority + Runaway control

IDC
  IDC Client -> IDC HAProxy -> IDC TiDB [zone=idc]
                           -> IDC TiKV leader / replica

GCP
  GCP Client -> GCP HAProxy -> GCP TiDB [zone=gcp]
                           -> GCP TiKV replica

PD
  IDC PD Leader + GCP PD Follower
  Active PD Follower ON

WAN
  Raft + ReadIndex + PD/TSO + fallback + maintenance traffic
```

建議初始設定：

```sql
SET GLOBAL tidb_replica_read = 'closest-replicas';
SET GLOBAL pd_enable_follower_handle_region = ON;
```

`tidb_enable_tso_follower_proxy` 應先測量再決定，不直接視為必要設定。

## 9. 驗證程序

### 9.1 Placement 是否完成

```sql
SHOW PLACEMENT;
SHOW PLACEMENT LABELS;
SELECT * FROM information_schema.placement_policies;
```

套用到資料物件的 `Scheduling_State` 必須為 `SCHEDULED`，不能只確認 Policy 已建立。

### 9.2 Leader / replica 實際位置

```sql
SELECT
    r.DB_NAME,
    r.TABLE_NAME,
    s.ADDRESS,
    s.LABEL,
    SUM(p.IS_LEADER) AS leader_count,
    COUNT(*) AS replica_count
FROM information_schema.TIKV_REGION_STATUS r
JOIN information_schema.TIKV_REGION_PEERS p
  ON p.REGION_ID = r.REGION_ID
JOIN information_schema.TIKV_STORE_STATUS s
  ON s.STORE_ID = p.STORE_ID
WHERE r.DB_NAME = 'app_db'
GROUP BY r.DB_NAME, r.TABLE_NAME, s.ADDRESS, s.LABEL
ORDER BY r.TABLE_NAME, s.ADDRESS;
```

官方參考：[TIKV_REGION_PEERS](https://docs.pingcap.com/tidb/stable/information-schema-tikv-region-peers/)、[TIKV_STORE_STATUS](https://docs.pingcap.com/tidb/stable/information-schema-tikv-store-status/)

### 9.3 就近讀取是否真的生效

Placement metadata 只能證明資料位置，不能證明 Request path。還要檢查：

- IDC / GCP Client 是否進入當地 HAProxy 與 TiDB。
- Grafana TiDB `KV Request / Read Req Traffic`。
- IDC / GCP TiDB 到 TiKV `20160` 的 destination IP。
- 專線 NetFlow 依 `2379`、`20160` 等 port 分類。
- 固定 SQL 在 IDC / GCP 執行時的 latency 與 remote traffic。
- 本地 replica 故障時是否 fallback 到另一區。

### 9.4 必須保留的設定快照

```sql
SHOW GLOBAL VARIABLES LIKE 'tidb_replica_read';
SHOW GLOBAL VARIABLES LIKE 'pd_enable_follower_handle_region';
SHOW GLOBAL VARIABLES LIKE 'tidb_enable_tso_follower_proxy';
SHOW PLACEMENT;
SHOW PLACEMENT LABELS;
```

另保留：

- PD Leader / Follower 位置。
- Region Leader 分布。
- TiDB / TiKV `zone` label。
- WAN RTT、jitter、packet loss 與 bandwidth。
- TSO RPC、ReadIndex、Raft append、snapshot traffic。

## 10. 最終結論

合理的目標敘述應為：

> IDC / GCP Client 先進入當地 TiDB；讀取優先使用當地 TiKV replica；寫入送往資料唯一的 Region Leader；跨專線承載 Raft、PD / TSO control traffic 與必要 fallback。

若需求是：

> IDC Request 絕不離開 IDC，GCP Request 絕不離開 GCP。

單一且強一致的 TiDB Cluster 無法嚴格保證。此需求必須改用：

- IDC / GCP 獨立 Cluster + 非同步 replication；或
- 地理資料 Partition + 對應區域 Leader；或
- 接受 Stale Read 與明確的資料新鮮度界線。
