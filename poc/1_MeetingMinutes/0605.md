## 0. 討論項目（會議優先）

### #1 — 壓力測試 CPU ~= 80% 條件下，調整這些參數的目的為何？

**背景觀察**（取自 [`../results/tidb-tc1/S-BASE/pipeline-log.md`](../results/tidb-tc1/S-BASE/pipeline-log.md)、[`../results/yuga-tc1/S-BASE/pipeline-log.md`](../results/yuga-tc1/S-BASE/pipeline-log.md)）：

| DB | iso | 代表點 | %user | %sys | %iowait | %idle | 推論 |
|---|---|---|---:|---:|---:|---:|---|
| TiDB | rc | vm-1node t128 | 79.6% | ~10% | <5% | 4.5% | **CPU 飽和 ~95.5%** |
| TiDB | rr | vm-1node t128 | 80.8% | ~10% | <5% | 4.5% | **CPU 飽和 ~95.5%** |
| YBDB | rc | vm-1node t32 | 74% | 18% | 0.25% | 1.89% | **CPU 飽和 ~92%** |
| CRDB | rc | vm-1node t128 | 68% | ~9% | 18% | 5% | **IO-bound**（非 CPU-bound）|
| CRDB | strict | vm-1node t128 | ~70% | ~13% | 17% | 10% | scale with thread（仍有 CPU headroom）|

→ TiDB / YBDB 已 CPU-bound；CRDB rc 是 IO-bound 例外。

**§3–§5 列的調參按目的分 4 類**：

| 類 | 範例 | CPU 飽和下的 ROI 與目的 |
|---|---|---|
| **A. Concurrency / pool size** | `tidb_executor_concurrency`、`readpool.unified.max-thread-count`、`grpc-concurrency`、`--rpc_workers_limit`、`--num_reactor_threads` | **throughput 影響弱**（資源已耗盡）；但改變 queueing 與 **p99 tail**；目的：identify queueing 行為，不是求 tpmC 增量 |
| **B. Admission control** | CRDB `admission.kv.enabled`、`admission.sql_kv_response.enabled`、`admission.sql_sql_response.enabled` | **CPU 飽和下最有效**：直接拒絕 / 排隊保護 tail latency；CRDB 強項；TiDB/YBDB 無等價 |
| **C. Memory / cache budget** | CRDB `--cache`、`--max-sql-memory`、`--max-go-memory`；TiDB block cache | 改變 cache hit rate → **間接改變 CPU profile**（cache miss = decode/parse/syscall 開銷）；目的：isolate from working set 變化 |
| **D. Split/rebalance guardrails** | `coprocessor.region-split-size=128GB`、`--enable_automatic_tablet_splitting=false`、`kv.range_split.by_load_enabled=false` | **與 throughput 無關**；避免測試中 split/rebalance 觸發污染 baseline；目的：controlled experiment 完整性 |

**核心論述**：

1. **本 PoC 在 CPU 飽和下，A / C / D 三類調參主要服務「controlled experiment 可重現性 + 公平對比」**，不是 throughput tuning。
2. **只有 B（admission control）在 CPU 飽和下直接改變 throughput**；但本 PoC 不調 B（接受三家 default admission 行為作為對比變數的一部分）。
3. **跨硬體假設**：所有結論均條件於「4 vCPU 飽和」；硬體升級（8/16 vCPU）會解鎖 A 類調參的真實 ROI。

- (b) 在 README 主表加註此 caveat？
- (c) 是否要在 pipeline-log template `db-config-dump` 段標明「dump 出來只供 controlled experiment 驗證，非 tuning lever」？

### #3 — 三家參數不對等對 cross-DB comparison 的影響（caveat 寫法）

**問題**：

| 控制維度 | TiDB | YBDB | CRDB |
|---|---|---|---|
| 顯式 thread pool 數 | ✓（readpool / grpc / scheduler）| ✓（rpc_workers / reactor）| ✗（單 process scheduler）|
| Admission control | 部分（pessimistic lock-wait）| 部分（queue length）| ✓ 完整（KV / SQL_KV / SQL_SQL）|
| Memory budget knob | TiKV block cache 等 | tablet_operation_memory_limit | `--cache` / `--max-sql-memory` |
| Process 隔離 | 三 process（TiDB Server / TiKV / PD）| 二 process（master / tserver）| 一 process |

**論述**：三家在 PoC 不能「同維度」對標 thread pool；CRDB 沒等價 knob → 強行對比會出現「CRDB 沒調 thread pool 所以可能比較差 / 比較好」的誤判。

**待決**：

- (a) README 主表 caveat 加一句：「process/thread/admission 三家不對等，跨家 tpmC 結論限同硬體 + same workload 下成立」？
- (b) 或在跨家對比段落顯式列三家「不可調」項目的清單？
- (c) 或不做 caveat，當作 PoC 的 implicit 假設？

## 1. 目的

盤點 TiDB / YugabyteDB / CockroachDB 在 PoC 架構中的主要 process、thread、concurrency 與 admission control 控制點，確認哪些參數會影響 CPU 使用、SQL 執行併發、KV/RPC thread pool、讀寫排隊與 benchmark 可比性。

本紀錄用於後續 vm-3node、HAProxy、Kubernetes 與跨專線測項規劃。正式測試時，這些參數若未對齊或未 dump，會讓 tpmC、p99、error rate 與 scale-out 效益難以歸因。

---

## 2. 總結

| DB | 架構特性 | Process / thread 控制重點 | PoC 判讀 |
|---|---|---|---|
| TiDB | SQL 層、TiKV、PD 分離 | TiDB SQL concurrency + TiKV gRPC / read pool / scheduler pool | 控制點多，需分 SQL 層與 KV 層判讀 |
| YugabyteDB | YB-Master + YB-TServer；YSQL 與 DocDB 同 tserver | YSQL connection/backend + tserver RPC / reactor / tablet operation | tserver 一體化，CPU 飽和時需分辨 SQL 層或 DocDB 層 |
| CockroachDB | 單一 `cockroach` process 內含 SQL/KV/Raft | 啟動資源限制 + admission control + DistSQL 行為 | 較少直接 thread knob，主要靠資源與 admission 控制 |

結論：三家不能只用「threads=128」解讀。go-tpc threads 是 client 併發，DB 內部仍會受到 SQL worker、RPC worker、read pool、Raft / KV queue、admission control 與 memory budget 影響。

## 3. TiDB 控制點

### 3.1 架構要件

| 元件 | 角色 | 主要影響 |
|---|---|---|
| TiDB Server | SQL parser / optimizer / executor | SQL 執行併發、connection、DistSQL scan |
| TiKV | KV store / Raft / RocksDB | gRPC thread、read pool、scheduler worker、block cache |
| PD | metadata / scheduling / placement | replica、leader、region schedule |
| TiFlash | columnar replica | 本輪 TPC-C-derived workload 不作主觀察點 |

### 3.2 Process / thread 相關參數

| 層級 | 參數 | 用途 | 官方說明 |
|---|---|---|---|
| TiDB Server | `max_connections` / `instance.max_connections` | 控制 SQL client connection 上限 | [TiDB config: max-server-connections / max_connections](https://docs.pingcap.com/tidb/stable/tidb-configuration-file/#max-server-connections) |
| TiDB Server | `tidb_executor_concurrency` | 控制 executor operator 併發 | [tidb_executor_concurrency](https://docs.pingcap.com/tidb/stable/system-variables/#tidb_executor_concurrency) |
| TiDB Server | `tidb_distsql_scan_concurrency` | 控制 DistSQL scan 併發 | [tidb_distsql_scan_concurrency](https://docs.pingcap.com/tidb/stable/system-variables/#tidb_distsql_scan_concurrency) |
| TiKV | `server.grpc-concurrency` | gRPC worker thread 數 | [TiKV grpc-concurrency](https://docs.pingcap.com/tidb/stable/tikv-configuration-file/#grpc-concurrency) |
| TiKV | `readpool.unified.max-thread-count` | read pool 最大 worker 數 | [TiKV readpool.unified.max-thread-count](https://docs.pingcap.com/tidb/stable/tikv-configuration-file/#max-thread-count) |
| TiKV | `readpool.unified.auto-adjust-pool-size` | 是否依 CPU 自動調整 read pool | [TiKV auto-adjust-pool-size](https://docs.pingcap.com/tidb/stable/tikv-configuration-file/#auto-adjust-pool-size-new-in-v630) |
| TiKV | `storage.scheduler-worker-pool-size` | storage scheduler worker pool | [TiKV scheduler-worker-pool-size](https://docs.pingcap.com/tidb/stable/tikv-configuration-file/#scheduler-worker-pool-size) |
| TiKV | `coprocessor.region-split-size` | region 自動分裂 guardrail | [TiKV region-split-size](https://docs.pingcap.com/tidb/stable/tikv-configuration-file/#region-split-size) |

### 3.3 PoC 注意事項

- TiDB 需同時 dump TiDB Server system variables 與 TiKV config，否則無法判斷瓶頸在 SQL executor 還是 TiKV read / scheduler pool。
- vm-3node 若 TiKV thread pool 或 auto-adjust 行為不同，會污染 shard / replica 對照。
- 本 PoC 以手動 split 鎖 shard 數；`region-split-size` 屬於避免背景自然分裂的 guardrail，不是正式 shard 控制點。

## 4. YugabyteDB 控制點

### 4.1 架構要件

| 元件 | 角色 | 主要影響 |
|---|---|---|
| YB-Master | metadata / placement / tablet 管理 | universe config、RF、tablet placement |
| YB-TServer | YSQL endpoint + DocDB tablet server | SQL connection、RPC worker、tablet operation、DocDB read/write |
| YSQL backend | PostgreSQL-compatible SQL session | connection / session / transaction 行為 |
| DocDB / Tablet | 分散式 KV / Raft tablet | shard/tablet、replica、quorum commit |

### 4.2 Process / thread 相關參數

| 層級 | 參數 | 用途 | 官方說明 |
|---|---|---|---|
| YB-TServer | `--ysql_max_connections` | 控制 YSQL connection 上限 | [yb-tserver configuration](https://docs.yugabyte.com/stable/reference/configuration/yb-tserver/) |
| YB-TServer | `--rpc_workers_limit` | 控制 RPC worker 上限 | [All yb-tserver flags](https://docs.yugabyte.com/stable/reference/configuration/all-flags-yb-tserver/) |
| YB-TServer | `--num_reactor_threads` | 控制 RPC reactor thread 數 | [All yb-tserver flags](https://docs.yugabyte.com/stable/reference/configuration/all-flags-yb-tserver/) |
| YB-TServer | `--tablet_server_svc_queue_length` | tablet server service queue 長度 | [All yb-tserver flags](https://docs.yugabyte.com/stable/reference/configuration/all-flags-yb-tserver/) |
| YB-TServer | `--tablet_operation_memory_limit_mb` | tablet operation memory limit | [All yb-tserver flags](https://docs.yugabyte.com/stable/reference/configuration/all-flags-yb-tserver/) |
| YB-TServer | `--ysql_num_shards_per_tserver` | 預設 tablet/shard 行為 guardrail | [yb-tserver configuration](https://docs.yugabyte.com/stable/reference/configuration/yb-tserver/) |
| YB-TServer | `--enable_automatic_tablet_splitting` | 是否啟用自動 tablet splitting | [yb-tserver configuration](https://docs.yugabyte.com/stable/reference/configuration/yb-tserver/) |
| YB-Master | `--num_reactor_threads` | master RPC reactor thread | [All yb-master flags](https://docs.yugabyte.com/stable/reference/configuration/all-flags-yb-master/) |

### 4.3 PoC 注意事項

- YugabyteDB 的 YSQL 與 DocDB 都落在 YB-TServer，CPU 飽和時不能只看 SQL session 數，還要看 RPC、tablet queue 與 DocDB operation。
- `--ysql_num_shards_per_tserver` 不是本 PoC 的正式 shard 控制點；正式 shard 數由 prepare 前 schema pre-create `SPLIT INTO N TABLETS` 鎖定。
- `--enable_automatic_tablet_splitting=false` 是避免背景 auto split 影響 benchmark，不代表 shard 數已被正確設定。
- 若後續做水平擴充測試，tablet rebalancing / leader movement 需獨立記錄，不能混入 steady-state baseline。

## 5. CockroachDB 控制點

### 5.1 架構要件

| 元件 | 角色 | 主要影響 |
|---|---|---|
| `cockroach` process | SQL / KV / Raft / storage 同 process | CPU、memory、SQL/KV queue、Raft activity 共用 |
| SQL layer | session / optimizer / DistSQL | connection、query distribution、SQL admission |
| KV layer | range / replica / leaseholder | range split、replication、raft commit |
| Pebble | local storage engine | block cache、write stall、compaction |

### 5.2 Process / thread 相關參數

CockroachDB 對使用者暴露的直接 thread pool 參數較少，主要透過 process memory、cache 與 admission control 管理資源。

| 層級 | 參數 / setting | 用途 | 官方說明 |
|---|---|---|---|
| Process start | `--cache` | Pebble block cache | [cockroach start: --cache](https://www.cockroachlabs.com/docs/stable/cockroach-start.html#general) |
| Process start | `--max-sql-memory` | SQL query memory budget | [cockroach start: --max-sql-memory](https://www.cockroachlabs.com/docs/stable/cockroach-start.html#general) |
| Process start | `--max-go-memory` | Go runtime soft memory limit | [cockroach start: --max-go-memory](https://www.cockroachlabs.com/docs/stable/cockroach-start.html#general) |
| Cluster setting | `server.max_connections_per_gateway` | gateway node connection 上限 | [Cluster settings](https://www.cockroachlabs.com/docs/stable/cluster-settings) |
| Cluster setting | `admission.kv.enabled` | KV admission control | [Admission control](https://www.cockroachlabs.com/docs/stable/admission-control.html) |
| Cluster setting | `admission.sql_kv_response.enabled` | SQL/KV response admission | [Admission control](https://www.cockroachlabs.com/docs/stable/admission-control.html) |
| Cluster setting | `admission.sql_sql_response.enabled` | SQL response admission | [Admission control](https://www.cockroachlabs.com/docs/stable/admission-control.html) |
| Session / optimizer | `distsql` / `sql.defaults.distsql` | 控制 distributed SQL planning 行為 | [Session variables](https://www.cockroachlabs.com/docs/stable/set-vars) |

### 5.3 PoC 注意事項

- CockroachDB 不適合用「thread pool 參數」單點對標 TiDB / YugabyteDB；它更偏向單 process 內部 scheduler + admission control。
- 若 `--cache`、`--max-sql-memory` 或 admission control 不一致，會改變 queueing 與 p99 表現。
- vm-3node shard / replica 測試需同時檢查 range 數、replica 數、leaseholder 分布與 admission 設定，否則無法判斷瓶頸來自 SQL、KV 還是 Raft。

## 6. 對 PoC 的落地建議

### 6.1 需要納入 config dump

| DB | 建議固定 dump |
|---|---|
| TiDB | TiDB system variables、TiKV config、PD schedule / replication config |
| YugabyteDB | yb-tserver flags、yb-master flags、universe config、tablet count、tablet placement |
| CockroachDB | `cockroach start` flags、cluster settings、session variables、range / replica / leaseholder 分布 |

### 6.2 需要標記的風險

- go-tpc client threads 不是 DB server thread 數；不可把 `threads=128` 直接解讀成 DB 內部 128 worker。
- TiDB / YugabyteDB 有較多顯式 thread / RPC / pool 參數；CockroachDB 主要靠 process memory 與 admission control。
- 若 auto thread pool、auto split、auto rebalance 在測試期間啟動，會讓 round 間差異放大，需標 caveat 或獨立成 scale-out / rebalance 測項。
- 跨專線測項若 IDC / GCP 使用不同 process/thread 參數，只能作異質部署觀察，不可與同硬體 baseline 直接比較。

### 6.3 建議下一步

1. 在 `pipeline-log-template.md` 新增「process/thread/admission config dump」欄位。
2. 在每個 suite 的 `gate` 或 `prepare` 階段保存三家對應 config snapshot。
3. 將「steady-state benchmark」與「scale-out / rebalance benchmark」分開報告。
4. 若要調整 thread / worker / admission 參數，需另開 tuning track，不混入 baseline。
