## 2. 三家 placement 機制與不跨區規則

> 「不跨區存取」適用於**應用層**（TPCC client → haproxy → DB）；**DB 內 raft replication 跨 WAN 是必然且為測量對象**。

| DB | placement 設定 | 跨區 leader-affinity 控制 |
|---|---|---|
| **TiDB** | PD `replication.location-labels=region,zone` + tikv start `--labels region=idc/gcp,zone=z1/z2/z3` | `label-property reject-leader region=gcp`（強制 leader 留 IDC）；或 PD `leader-priority`、`leader-schedule-limit` 配 region |
| **CRDB** | per-node `--locality region=idc,zone=z1/z2/z3` + `ALTER ... CONFIGURE ZONE USING lease_preferences='[[+region=idc]]', voter_constraints='{+region=idc:2, +region=gcp:1}'` | 跨區語義最完整；可 `voter_constraints` 精控每 shard voter 分布 |
| **YBDB** | `yugabyted start --placement_region=idc --placement_zone=z1` + `yb-admin modify_placement_info` + `set_preferred_zones` | `set_preferred_zones` 鎖 leader region |

**[repo 推論]** 目前 ansible playbook（`tidb-vm3.yml`、`cockroach-vm3.yml`、`yugabyte-vm3.yml`）**未支援 6-node 跨區拓樸**：

- locality / placement_region label 全未注入
- inventory `vm3.ini` 只有 IDC 三 db node + .20 haproxy + .31 client
- GCP 5 IPs `10.162.0.x` 在 `iac-gcp/` 目錄存在路徑但配對 ansible playbook 未實作（**[missing source]**）

→ **Pre-P0 必須補 playbook 6-node 變體**（§9 排程已含）。

## 3. Test 1 — IDC 獨立 TPCC（測 raft 跨 WAN 對單側寫的損耗）

**目的**：跨專線後，**只 IDC 側發起 workload**，量化 raft 寫 quorum 因 GCP follower 存在的延遲增加。

| 項 | 設定 |
|---|---|
| 拓樸 | 6-node single cluster，placement = **P-A**（每 shard 2 IDC voter + 1 GCP voter；leader 鎖 IDC） |
| Workload | 只 IDC client (172.24.40.31) → IDC haproxy (172.24.47.20) → IDC nodes (.32/.33/.34) |
| GCP 側 | haproxy/client idle；GCP 三 nodes 純當 follower voter |
| Iso | rc |
| W | 128 |
| Thread sweep | 同現行 16/32/64/128（或 user 指定） |
| Rounds | 5（與 vm-3node 對齊） |
| 度量 vs baseline | tpmC、p99、commit latency、raft propose latency、inter-region bytes |
| Baseline 對照 | 現行 CRDB/TiDB/YBDB `vm-3node-haproxy-3s3r-rc` 主表（**[已實作事實]** `results/{db}-tc1/S-BASE/vm-3node-haproxy-3s3r-rc/`） |
| 預期 | iso=rc + raft 寫需 quorum；majority 在 IDC → 兩 IDC voter ACK 即可 commit，**GCP voter 不擋 critical path**；tpmC ↓ 10–30%（端 RTT、CPU competition）|

**Critical**：若 placement actual 沒落地（GCP 變 leader） → write critical path 跨 WAN → tpmC 預期跌至 < 10% baseline。**[下一步建議]** dry-run-confirm 必須新增 placement actual gate。

## 4. Test 2 — IDC + GCP 同時 TPCC（測互相干擾 + 跨區並發）

**目的**：兩區同時跑 workload，量化 WAN 競爭 + 跨區 conflict。

| 項 | 設定 |
|---|---|
| 拓樸 | 6-node single cluster，placement = **P-B**（leader 平衡：1 IDC + 1 GCP + 1 arbiter；或 per-shard leader 各區散）|
| Workload | IDC client + GCP client **並行** |
| 度量 | 兩側獨立 tpmC、合計 throughput、相互 p99 影響、WAN BW 使用率、cross-region key conflict |
| 預期 | WAN BW 競爭 + raft 跨區 leader → **兩側都比 Test 1 差**；若 TPCC W 區分（IDC W=1–64, GCP W=65–128）→ 減少 key conflict |
| 衝突隔離 | 建議 W 區分（避免兩 client 寫同 warehouse），讓「WAN 互擾」與「key conflict」可分開歸因 |

## 5. Test 3 — Chaos / Failover 設計

| # | 場景 | 注入方式 | 度量 | 預期 |
|---|---|---|---|---|
| **C1** | IDC cluster node1 down | `systemctl stop <db>` on .32 | tpmC 谷期、error rate、leader 重選 RTO、recovery 後 throughput | TiDB ~20–30s / CRDB ~9s / YBDB ~5–15s |
| **C2** | IDC haproxy down | `systemctl stop haproxy` on .20 | client 連線 fail / fallback 行為（無 fallback → 應接受 error）| 純應用層失效；DB layer 不受影響 |
| **C3** | WAN 全斷（IDC ⇄ GCP） | `iptables -A INPUT -s 10.162.0.0/24 -j DROP` × 雙向 | majority/minority 行為；split-brain 防護；恢復後 catch-up | P-A 下：IDC 仍有 majority → 可寫；GCP minority → read-only fallback；P-B 下：兩區皆 minority → 都寫不通 |
| **C4** | WAN 注入 +200ms 延遲 | `tc qdisc add dev <iface> root netem delay 200ms` | tpmC 漸降、raft heartbeat 失敗閾值（CRDB 默認 3s，TiDB raft tick 1s × election=10） | tpmC 線性下降至 RTT 主導 |
| **C5** | WAN packet loss 5% | `tc qdisc add ... netem loss 5%` | retransmit ↑、retry rate、tail latency、application error | — |
| **C6** | 慢 disk (IDC node1) | `cgroup io.max` throttle 或 `fio` 背景 inflate iops | 該 node leader 主動卸下、tpmC 受擊、tail 飆 | TiDB/CRDB/YBDB 各有 hot-spot 偵測 — 觀察 settle 時間 |
| **C7** | IDC 全 3 node down（region majority loss） | `systemctl stop` × 3 同時 | P-A 下 → GCP 只剩 1 voter，全 cluster read-only / 寫拒；P-B 下 → 同 |
| **C8** (可選) | clock skew injection | NTP off + `date -s` ±2s on .32 | CRDB hybrid logical clock 嚴測；TiDB/YBDB 較不敏感 | CRDB 可能拒 transaction；其餘 tpmC 可能微降 |

**Chaos 共同 metric artifact**：
- `chaos/<C-id>/timeline.json`（注入 t0、谷期 t1、recover t2、recovery RTO=t2−t0）
- `chaos/<C-id>/tpmc-1s.txt`（per-second tpmC 取樣）
- `chaos/<C-id>/error-rate-by-sec.txt`（區分 conn-refused / timeout / app-retry-success）
- `chaos/<C-id>/leader-redist-trace.txt`（leader 重選歷程）

## 7. Critical Risks

| # | 風險 | 緩解 |
|---|---|---|
| **R1** | iac/ansible 全未支援 6-node 跨區（GCP playbook 未實作） | Pre-P0 預估 **2–3 天** 補 playbook + inventory + locality label 注入 |
| **R2** | WAN 變數不可控（GCP region 選擇 / IDC 出口 / 路由跳數 / 運營商 QoS） | per-test 起前/起末各測 60s baseline，作 caveat 寫主表 |
| **R3** | iso=rc + 跨 WAN raft：平均寫 latency = round-trip 跨區；若 RTT > 50ms，Test 1 tpmC 預期 < 30% baseline | 預先估算 RTT × 寫 % → 評估是否值得跑；若數據過低，重審 placement P-A 改 3 IDC voter（GCP 純 learner） |
| **R4** | go-tpc retry 行為：fail-over 期 error 統計不準（不知是「拒絕」還是「重試成功」） | chaos 場景額外採 `error-rate-by-sec.txt` 區分 |
| **R5** | cost 爆炸：6-node × 3 DB × 2 placement × 3 test + chaos = 太多 | 先 1 DB（CRDB）全套；TiDB/YBDB 範圍依結果決定 |
| **R6** | data 一致性語義差異：CRDB serializable default / TiDB pessimistic / YBDB snapshot；iso=rc 下三家行為差異需先 align | 沿用現行 isolation gate（`dry-run-confirm.sh §3`，**[已實作事實]**）|
| **R7** | placement actual ≠ config（PD/allocator 遲遲不調度）| 新 hard gate `placement/leader-affinity-gate` fail-closed；§6 已列 |
| **R8** | TPCC W=128 太大 / 太小：跨區寫放大下，W 分布 affect key conflict 程度 | Test 2 W 分區（IDC W=1-64, GCP W=65-128）讓互擾與 conflict 可分離 |
