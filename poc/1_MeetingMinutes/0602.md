## 3. 問題回答

### Q1 IDC / GCP 能否拆不同設定？

**TiDB**
- **[已實作事實]** PD/TiKV/TiDB-server 可分機架部署，PD `replication.location-labels` + tikv labels 可做 zone-aware placement，但本 PoC 三 TiKV 同 spec（`tidb-vm3.yml:198-210` 無 per-store 差異），未啟用 zone label。
- **[repo 推論]** 若 IDC ↔ GCP 一邊 `region-split-size=128GB`、另一邊 default 96MB，rebalance / split 行為**不可比**，會把跨環境差異混入 shard 變數 → **應禁止異參**。

**CockroachDB**
- **[已實作事實]** `--locality=region=...,zone=...` 是 per-node start flag，CRDB 會依 zone config 自動 zone-aware；本 PoC `cockroach-vm3.yml:205` 只動 cluster `num_replicas`，未設 locality。
- **[repo 推論]** IDC vs GCP 異 locality 會啟動跨地域 placement → 違反「controlled experiment shard 鎖定」原則；只能作 exploratory 拓樸（e.g. multi-region）。

**YugabyteDB**
- **[已實作事實]** tserver gflags 可 per-node 設，但 placement 由 yugabyted/yb-master 統一決定；本 PoC `yugabyte-vm3.yml:38,173,225` 三 tserver flags 一致。
- **[repo 推論]** 跨 IDC tserver 異 `ysql_num_shards_per_tserver` 會直接破壞 tablet 預期數計算（§7.5.3 公式 `ysql_num_shards_per_tserver × num_tservers_in_placement`）；**禁止異參**。

**結論**：三家**可拆**（DB 機制本來就提供），但對 PoC 對照**不應拆**，否則 tpmC/p99 來源無法 attribute。

### Q3 cluster 設 3 shard 不做 manual split 可行？

**TiDB**
- **[repo 推論]** 沒有 cluster-level「預設 N regions per table」設定；regions 由 TiKV 依 `region-split-size`（128MB default）自然 split。在 `region-split-size=128GB`（本 PoC 設定）下，128W 的 TPC-C 表（最大 order_line ~3GB）不會自然 split → 預設只 1 region/表。若改 `region-split-size=1MB` 強制自然分裂 → 數量不可控（隨資料 byte size），且會在 run 期間繼續 split → 污染。
- **不可行**作 controlled baseline；只能作 exploratory。

**CockroachDB**
- **[repo 推論]** `range_max_bytes=137438953472`（128GB，PoC-DESIGN §7.5.2 line 634）下 TPC-C 表也不會自然 split。開 `kv.range_split.by_load_enabled=true` + load-based split 可能在 run 期間因 QPS 高的 hot range 動態切分 → 同 TiDB 不可控。
- **不可行**作 controlled baseline；只能作 exploratory。

**YugabyteDB**
- **[已實作事實]** §7.5.3 line 684 註明：「3 tservers × ysql_num_shards_per_tserver=1 = 3 自然 tablets」**錯**，2026-05-23 cell 3 dispatch 實測 RF=1 placement 只覆蓋 1 tserver → 自然 1 tablet。即使 RF=3 下「自然 3 tablets」也只在 cluster default 路徑成立，但這仍是 cluster gflag 結果，不是「不 split」的零控制。
- **不可行**作 controlled baseline；3s\*r 須 pre-create + SPLIT INTO。

**結論**：**三家皆不可行**作 controlled baseline。理由：自然 shard 數隨資料量/負載動態變化，跨 cell/cross-DB 不同步。

### Q4 auto rebalance / auto sharding 對 TPC-C 影響？

**共同影響（[repo 推論]）**：
- run 期間若觸發 split → 寫熱點 range/region/tablet 進入 split → tail latency p99 飆升、temporary unavailability、retry rate 上升
- 若 rebalance（leader / leaseholder / tablet leader move）→ followers re-snapshot、cache thrashing → tpmC 下降
- benchmark 期間任一 trigger → 該組 N=1 結果**不可作正式 median**（README §註N10 已寫 shard/replica 沒鎖死則不可比）

**TiDB 特例**：
- **[已實作事實]** `tidb-vm3.yml:208-210` 故意保留 `leader-schedule-limit=4` + `replica-schedule-limit=4`、只關 `region-schedule-limit=0`。意圖（Fix #11 D10 註）：region 數凍結（鎖 shard 變數），但 leader/replica 允許搬移（RF=3 部署需要 PD 主動 make-up-replica）。
- **[repo 推論]** 這實際上**允許 run 期間 leader balance** — 是「**part-controlled**」狀態，不是純 steady-state。dispatch records `2026-05-31-tidb-schedule-limit-0-vs-4` 即比較 l0r0 vs l4r4 兩種 PD 設定 → 顯示 leader balance ON 對 tpmC 有 +5~10% 影響。

**結論**：auto rebalance / split **必須**在正式 baseline 期間全關；TiDB l4r4 屬於「**已知 caveat 的混合狀態**」，需在主表標明。

## 9. 水平擴充與 re-sharding / re-balance 規劃

| DB | scale-out 觸發行為 | auto re-sharding / rebalance 常見行為 | 對 TPC-C 指標影響（推論） | 正式 benchmark 措施 | exploratory 測項 |
|---|---|---|---|---|---|
| **TiDB** | add TiKV → PD detect → start make-up-replica + leader balance | 受 `region-schedule-limit` / `leader-schedule-limit` / `replica-schedule-limit` 控制；本 PoC l4r4 模式下 leader/replica 持續搬，region 數凍結 | leader move → 新 leaseholder cache 冷啟 → 短期 p99 飆升、tpmC 下降 5-15%；replica add → 寫 raft 多一份 quorum cost | **設 0** 後跑（l0r0），run 後再復原為 l4r4；或於 add-node 後等 `under_replicated==0` settle 至少 N min | l4r4 下 add-node、觀察 leader move 對 t=128 p99 影響 |
| **CockroachDB** | add node → allocator detect → start replica rebalance + lease move | 預設開（無 cluster setting 全關 rebalance）；可 `SET CLUSTER SETTING kv.allocator.load_based_lease_rebalancing.enabled=false` 部分關 | leaseholder move → 該 range 短期 read latency double、write retry 上升；replica add → quorum cost +33% per range during settle | run 期間需 settle 至 `under_replicated==0` + leaseholder stddev < 閾值；不允許 add-node 在 run 中 | add-node 在 run 第 3 round 後注入，觀察 retry/p99 |
