# 数据库异常智能分析与诊断

作者: 金龙

## 1 现状与问题

### 1.1 规模增长与运维能力发展之间的不平衡问题凸显
伴随着最近几年美团业务的快速发展，数据库的规模也保持着高速增长。作为整个业务系统的“神经末梢”，数据库一旦出现问题，对业务造成的损失就会非常大。同时，因数据库规模的快速增长，出现问题的数量也大大增加，完全依靠人力的被动分析与定位已经不堪重负。  
图1 数据库实例增长趋势

### 1.2 理想很丰满，现实很骨感
美团数据库团队当前面临的主要矛盾是：实例规模增长与运维能力发展之间的不平衡，主要体现在数据库稳定性要求较高与关键数据缺失。由于产品能力不足，只能依赖专业 DBA 手动排查问题，异常处理时间较长。因此，我们决定补齐关键信息，提供自助或自动定位问题的能力，缩短处理时长。

我们复盘了过去一段时间内的故障和告警，深入分析这些问题的根因，发现任何一个异常其实都可以按时间拆分为异常预防、异常处理和异常复盘三阶段。针对这三阶段，结合 MTTR 的定义，并调研了美团内部及业界的解决方案，我们做了一张涵盖数据库异常处理方案的全景图。  
图2 运维能力的现状

通过对比，我们发现：
- 每个环节我们都有相关的工具支撑，但能力又不够强，相比头部云厂商大概只有 20%～30% 的能力，短板明显。
- 自助化和自动化能力不足，工具虽多，但整个链条没有打通，未形成合力。

团队成员经过深入分析和讨论后，提出了一种比较符合当前发展阶段的解决思路。

## 2 解决的思路

### 2.1 既解决短期矛盾，也立足长远发展
从对历史故障的复盘来看，80% 的故障中 80% 的时间都花在分析和定位上。解决异常分析和定位效率短期的 ROI 最高。长期来看，只有完善能力版图，才能持续不断地提升数据库的稳定性及保障能力。因此，我们的想法是既要解决短期矛盾，又要立足长远发展（Think Big Picture, Think Long Term）。新的方案要为未来留足发展空间，不能只是“头痛医头、脚痛医脚”。

在宏观层面，我们希望将更多功能做到自动定位，基于自动定位来自助或自动地处理变更，从而提高异常恢复的效率，最终提升用户体验。将异常处理效率提高和用户体验提升后，运维人员（主要是 DBA）的沟通成本将大幅降低，运维人员就有更多时间进行技术投入，把更多“人肉处理”的异常变成自助或自动处理，形成“飞轮效应”，最终达成高效的稳定性保障目标。

在微观层面，我们基于已有的数据，通过结构化的信息输出提升可观测性，补齐关键数据缺失的短板。同时，基于完善的信息输出，通过规则（专家经验）和 AI 配合，提供自助或自动定位能力，缩短处理时长。  
图3 宏观和微观

### 2.2 夯实基础能力，赋能上层业务，实现数据库自治
有了明确的指导思想，我们需要采取怎样的发展策略和路径？就当时团队的人力情况看，没有同学有过类似异常自治的开发经验，甚至对数据库异常分析的能力还不具备，人才结构不能满足产品的终极目标。所谓“天下大事必作于细，天下难事必作于易”。我们的思路是从小功能和容易的地方入手，先完善指标监控、慢查询、活跃会话这些简单功能，再逐步深入到全量 SQL、异常根因分析和慢查询优化建议等复杂功能。通过这些基础工作来“借假修真”，不断提升团队攻坚克难的能力，同时为智能化打下良好基础。

### 2.3 建立科学的评估体系，持续跟踪产品质量
卡普兰说过：“没有度量就没有管理”。只有建立科学的评估体系，才能推动产品持续改进。我们曾考虑根因定位结果指标（准确率、召回率），但结果指标不可控，难以指导日常工作。需要找可控因素并持续改善。

参考亚马逊的可控输入/输出指标方法论，我们设计了根因定位的可控输入与输出指标，并规划了技术实现思路：在模拟环境不断提升可控部分，最终改善线上实际效果。主要包括“根因定位可控输入和输出指标设计思路”和“根因定位可控输入指标获取的技术实现思路”。

## 3 技术方案

### 3.1 技术架构的顶层设计
在顶层设计上，我们秉承平台化、自助化、智能化和自动化四步走的演进策略。首先完善可观测能力，通过关键信息展示构建易用的数据库监控平台。然后根据这些关键信息为变更（如数据变更和索引变更）提供赋能，将部分高频运维工作通过这些结构化关键信息让用户自主决策（自助化）。接着加入智能元素（专家经验 + AI），进一步覆盖自助化场景，并逐步将部分低风险功能自动化，最终走向高级或完全自动化。

我们将自动化放在智能化之后，因为智能化是自动化的前提，只有不断提升智能化才能达到高级或完全自动化。下图为顶层架构设计（左侧为演进策略，右侧为技术架构顶层设计及 2021 年底的现状）：

### 3.2 数据采集层的设计
数据采集层是所有链路的最底层和最重要环节，采集数据的质量直接决定整个系统能力。它与数据库实例直接打交道，任何设计缺陷都可能导致大规模故障。因此，技术方案必须兼顾采集质量和实例稳定性；在二者无法平衡时，宁可牺牲采集质量也要保证数据库稳定性。

业界通常采用基于内核的采集方式，但美团自研内核较晚且部署周期长。短期我们采用抓包方式作为过渡，等基于内核的采集部署到一定规模后再逐步切换。调研如下：

| 方案     | 性能 | 通用性 | 备注                          |
|---------|------|--------|-------------------------------|
| pcap    | 低   | 高     | 美团酒旅团队已线上实践         |
| pf_ring | 中   | 中     | 需要改造 MySQL                |
| dpdk    | 高   | 低     | 需要重新编译网卡驱动          |

基于 pf_ring 和 dpdk 的方案依赖较大、短期难以实现，且团队无相关经验。基于 pcap 的方式依赖少、我们已有经验（美团酒旅团队基于抓包做过全量 SQL 数据采集工具并经过一年验证），因此最终选择基于 pcap 的抓包方案。以下为采集层方案的架构图及采集质量与对数据库性能影响情况。

### 3.3 计算存储层的设计
考虑到美团数据库实例数量和流量巨大且快速增长，设计需满足当前并兼顾未来 5 年及更久的需求。对故障分析而言，数据的实时性和完备性是快速高效定位问题的关键，而保证实时性和完备性需要的容量成本也不可忽视。结合这些要求，我们提出设计原则：
- 全内存计算：确保所有计算在单线程或单进程内做纯内存操作，追求性能与吞吐量极致。
- 上报原始数据：MySQL 实例上报的数据尽量维持原始状态，尽量少做数据加工。
- 数据压缩：上报量巨大，需对上报数据做极致压缩。
- 内存消耗可控：通过理论和实际压测确保内存溢出几乎不可能。
- 最小化对 MySQL 实例的影响：计算尽量后置，不在 Agent 上做复杂计算，确保对 RDS 实例影响最小。

全量 SQL（所有访问数据库的 SQL）是系统最具挑战且最重要的信息，因此对全量 SQL 的聚合方式、压缩效果和补偿设计进行了重点设计。

#### 3.3.1 全量 SQL 的聚合方式
由于明细数据巨大，我们采用聚合方式。消费线程会对相同模板 SQL 的消息按分钟粒度进行聚合计算，以 “RDSIP + DBName + SQL 模板 ID + SQL 查询结束时间所在分钟数” 为聚合键。聚合键计算公式为：  
Aggkey = RDS_IP_DBName_SQL_Template_ID_Time_Minute（Time_Minute 取自 SQL 查询结束时间所在的 年-月-日-时-分钟）

#### 3.3.2 全量 SQL 数据聚合和压缩的效果
在数据压缩方面，遵循层层减流原则，使用消息压缩、预聚合、字典化、分钟级聚合等手段，保证流量在经过每个组件时逐级递减，最终节省带宽和存储量。相关压缩环节和测试数据表现如下（敏感数据已脱敏）：  
图13 全量 SQL 压缩设计与效果

#### 3.3.3 全量 SQL 数据补偿机制
聚合端按分钟进行聚合，并允许额外一分钟的消息延迟；如果消息延迟超过 1 分钟会被直接丢弃。在业务高峰期延迟严重时会丢失大量数据，影响后续异常分析准确性。因此，我们增加了延迟消息补偿机制：对过期数据发入补偿队列（采用美团消息队列服务 Mafka），通过过期数据补偿保证延迟久的消息也能正常存储，以最终一致性保证后续数据库异常分析的准确性。

### 3.4 分析决策层的设计
在获取较完整数据后，基于数据进行决策推断根因。这部分我们使用专家经验结合 AI 的方式，演进路径分为四个阶段：
1. 第一阶段：完全以规则为主，积累领域经验，探索可行路径。  
2. 第二阶段：探索 AI 场景，但以专家经验为主，在少量低频场景使用 AI 算法验证能力。  
3. 第三阶段：专家经验和 AI 齐头并进，专家经验在已有场景迭代，AI 在新场景落地，通过双轨制保证能力不退化。  
4. 第四阶段：AI 替换大部分专家经验，以 AI 为主、专家经验为辅，极致发挥 AI 能力。

#### 3.4.2 基于 AI 算法的方式
异常数据库指标检测：数据库核心指标的异常检测依赖于对指标历史数据的合理建模，通过将离线定期建模与实时流检测结合，有助于在实例存在故障或风险时有效定位根本问题，及时解决问题。

建模过程分为三步：先对指标历史数据做预处理（缺失值填充、数据平滑与聚合等）；随后通过分类模块创建不同分支，针对不同类型指标采用不同建模手段；最终将模型序列化存储，供 Flink 任务读取实现流检测。  
图17 基于 AI 的异常检测设计

根因诊断（构建中）：订阅告警消息（基于规则或异常检测触发），触发诊断流程采集、分析数据，推断根因并筛选出有效信息辅助用户解决。诊断结果通过内部 IM（大象）通知用户，并提供诊断详情页面，用户可通过标注来优化诊断准确性。  
图18 根因诊断设计

主要模块包括：
- 数据采集：采集数据库性能指标、数据库状态抓取、系统指标、硬件问题、日志、记录等数据。
- 特征提取：从各类数据中提取时序特征、文本特征以及基于数据库知识的领域特征。
- 根因分类：特征预处理、特征筛选、算法分类、根因排序等。
- 根因扩展：基于根因类别深入挖掘相关信息，提高用户处理效率，包括 SQL 行为分析、专家规则、指标关联、维度下钻、日志分析等功能模块。

## 5 总结与未来展望
数据库自治服务经过约 2 年发展，已基本夯实基础能力，并在部分业务场景完成初步赋能（例如：问题 SQL 在业务服务上线发布前自动识别提示潜在风险；索引误变更工单执行前自动检测索引近期访问流量以阻断误变更等）。接下来目标除了在已完成工作上继续深耕与提升能力外，重点瞄准数据库自治能力，规划围绕以下三点展开：

1. 计算存储能力增强：随着实例和业务流量持续高速增长及采集信息的丰富，需增强现有数据通道能力，保障未来 3–5 年的处理能力。
2. 自治能力在少部分场景落地：采取三步走策略实现数据库自治：
   - 第一步：建立根因诊断与 SOP 文档的关联，将诊断和处理透明化；
   - 第二步：SOP 文档平台化，将诊断和处理流程化；
   - 第三步：部分低风险场景无人干预，实现诊断和处理自动化，逐步实现数据库自治。
3. 更灵活的异常回溯系统：场景根因定位算法在上线前或改进后需要验证，我们将完善验证体系，建立灵活的异常回溯系统，通过现场信息回放不断优化提升定位准确率。
