# Redis 性能刺客：大Key

作者：竹子爱揍功夫熊猫  
日期：2023-11-09

本文介绍了 Redis 中“大Key”的定义、可能引发的问题、快速定位方法以及常见的优化方案，帮助提升系统性能与稳定性。

## 一、大Key 的定义

在 Redis 中，大Key 指占用较多内存空间的键（key）。大Key 的定义是相对的，通常以 key 的总大小和 key 中成员的数量来综合判断，例如：

- String 类型的 key，其值为 5MB（示例值，仅供参考）。
- ZSET 类型的 key，成员数量为 10k 个（示例值）。
- Hash 类型的 key，成员数量虽然只有 1k 个，但这些成员的 value 总大小为 100MB（示例值）。

具体阈值应结合业务场景和系统容量进行判断。

## 二、大Key 引发的问题

大量或单个大Key 会对 Redis 产生多方面负面影响，包括但不限于：

- 客户端执行命令变慢（单个命令处理时间增加）。
- Redis 内存达到 maxmemory 限制，导致操作阻塞、重要 key 被逐出，甚至 OOM（内存溢出）。
- 集群中某个分片的内存使用远高于其他分片，无法实现内存均衡。
- 对大Key 的读请求会占满实例带宽，使服务变慢并波及相关服务。
- 对大Key 的删除或修改操作可能导致主库长时间阻塞，引发同步中断或主从切换。

总体可归纳为三类影响：内存占用、网络/传输延迟、持久化与复制延迟。

## 三、大Key 产生的原因

常见原因包括：

- 存储大量数据：在不适合的场景下使用单个 key 存储大量数据，例如用 String 存放大型二进制文件或将大量条目累积在同一个 Hash/List/Set 中。
- 缺乏业务规划：上线前未对数据结构或成员拆分进行合理设计，导致个别 key 成员数量过多。
- 无效数据堆积：未定期清理无用或过期的数据，产生长期增长。
- 序列化格式选择不当：序列化导致占用空间过大。
- 数据结构选择不当：选择的数据类型不适合该场景，导致单个 key 体积异常增长。
- 代码故障：生产端消费或更新逻辑异常，造成某些 key 只增不减。

## 四、如何快速找出大Key

可以使用 Redis 自带命令和第三方工具进行扫描和分析，常用方法：

- 使用 SCAN 命令遍历键并结合 `MEMORY USAGE <key>` 获取键大小。示例流程：用 SCAN 迭代遍历所有 key（或按模式匹配），对每个 key 调用 `MEMORY USAGE` 并记录较大的 key。
- 使用 Redis 的内存分析工具（如 Redis 官方的 Redis Memory Analyzer，RMA）生成实例内存快照，分析快照中每个 key 的大小及结构。
- 编写 Lua 脚本：在 Redis 中用 Lua 脚本结合遍历逻辑和 `MEMORY USAGE` 对 key 进行筛选与统计。
- 使用云 Redis 的内置诊断/分析工具：多数云服务（如腾讯云、阿里云等）的 Redis 产品自带大 key/内存分析功能，参考对应文档即可。

## 五、大Key 的优化方案

针对大Key 可以采取以下优化措施：

### 5.1 拆分大Key
将包含数万成员的单个 key 拆分为多个 key，确保单个 key 的成员数量在合理范围内。特别是在集群模式下，拆分有助于实现数据分片间的内存均衡。

### 5.2 清理与迁移
- 对不适合放在 Redis 的数据，迁移到其他持久化存储（如对象存储、数据库），并在 Redis 中删除对应数据。
- 对于删除大Key，Redis 4.0 及以上版本提供 `UNLINK` 命令，可以异步、非阻塞地删除大Key（底层会异步释放内存）。
- 在 Redis 4.0 之前的版本，建议使用 SCAN 分批读取并删除数据，避免一次性删除大量数据导致 Redis 阻塞。

### 5.3 定期清理过期或冗余数据
- 通过定时任务定期清理失效或不再需要的数据，防止过期数据堆积。
- 对 Hash 类型的数据，建议使用 `HSCAN` 配合 `HDEL` 分批删除失效字段，避免一次性删除大量字段引起阻塞。

### 5.4 其他注意事项
- 选择合适的数据结构和序列化格式，避免因序列化膨胀导致的内存问题。
- 在云 Redis 环境下，关注云厂商提供的大Key 防护和自动诊断方案并按需启用。

## 六、总结

大Key 会对 Redis 的性能、内存和可用性带来显著影响。识别大Key、定位根因并采取相应的拆分、清理或迁移策略，可以显著提升系统的稳定性与用户体验。实践中应结合具体业务场景制定合适的阈值、清理策略和监控告警。
