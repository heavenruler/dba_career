## 01 前言

从去年开始京东广告投放系统做了一次以领域驱动设计为思想内核的架构升级。在深入理解 DDD 思想的同时，我们基于广告投放业务的本质特征融入了自己的理解和改造。新架构从设计思想到落地框架都进行了彻底革新，涉及内容较多，因此我们希望通过一系列文章循序渐进地阐述本次架构升级的始末。新架构并不是一日而成，而是经过多次架构升级的演进。本文作为该系列的第一篇，先通过广告投放平台的架构演进历程来介绍新架构的设计初衷。

如前言所述，本文主要聚焦于广告投放系统历代代码架构的演进历程。为避免篇幅过长，对新架构中具体框架及 API 的说明将浅尝辄止，后续文章会逐步给出更具象的描述。

## 02 什么是好的代码架构

当前我们面临的主要矛盾是“越来越多的多场景化复杂业务需求与有限的研发人力之间的矛盾”。要解决这一矛盾，系统应具备：设计易拓展、代码易复用、逻辑易传承、运行更稳定。每个特性都有具体要求：

- 设计易拓展  
  好的架构应实现业务与技术组件的分离，使设计者专注业务流程，能够开箱即用地套用组件、框架和解决方案，避免重复设计。好的架构还能引导设计者完成最小子问题的正交分解，降低需求复杂度和理解成本。

- 代码易复用  
  好的架构应有良好分层，强调正交子模块的拆分与封装，同层原子模块之间应避免互相依赖和耦合，让上层系统轻松组合复用底层逻辑，以 O(n1+n2+...+nm) 的实现复杂度支撑 O(n1* n2* ... *nm) 的业务复杂度。另外，业务逻辑基于数据之上，封装良好的架构应有健全的数据模型维护和共享机制，避免同一数据对象的重复查询，并能通过批量操作降低系统 I/O 负载。

- 逻辑易传承  
  我们曾尝试通过维护文档建立业务知识库，但多数失败了。业务功能由代码承接，代码天然具备业务知识库的功能。因此好的代码架构不仅实现业务功能，还应承担传递业务知识的职责：新成员通过代码快速建立对业务的宏观认知，并能按图索骥定位改动点和深入了解业务细节。

- 运行更稳定  
  好的架构在多场景化需求下能做到场景隔离，避免不同场景特有逻辑互相耦合干扰，防止一个场景上线影响其他业务。此外，应通过设计良好的框架和标准模板在有益程度上约束开发者编码行为，把规范框架化，而不是过度依赖人治和 code review。

## 03 架构演进之路

下面将从最早期代码架构开始，逐代剖析架构演进的历程，说明每次改进背后的设计动机与思路，以便更好理解新架构设计，并为推动下一代演进打下基础。

### 第一代：没有架构的架构

最初的架构非常常见：简单、平铺直叙，数据查询和业务逻辑交织，是典型的面向数据库编程案例。在早期场景单一、需求简单时可以快速实现功能，但随着服务场景增加，这套架构逐渐暴露问题，主要体现在两个方面：

1. 频繁“打补丁”式开发。新增业务通常在现有流程中增加 if...else 分支并直接实现逻辑，随着流程变长容易忽视前置流程已查询的数据，导致数据重复查询。为复用逻辑，我们把常用逻辑封装为方法，但常常把数据获取逻辑也封装在内，进一步加剧数据重复查询，尤其在循环调用场景中问题更突出。此外，这种复用方式造成数据库访问碎片化，难以利用批量操作优化性能。最近大促中暴露出的几处性能问题大多由此导致。

2. 逻辑耦合与复杂度上升。不同业务场景逻辑互相交织，if 分支层层嵌套，圈复杂度飙升，通用逻辑对不同场景的适配性下降。新增需求开发变得困难：试图复用相似逻辑时会遇到很多纠结，代码执行流程不再清晰。为避免影响旧流程，我们不断增加 if 分支，反而进一步恶化，代码充斥重复代码和多层嵌套。

为了缓解这些问题，我们引入了上下文机制，尝试将数据查询逻辑与业务流程分离，进而进入第二代架构。

### 第二代：略有改善的上下文机制

上下文机制为了解决数据重复查询问题而引入。思路朴素：在方法一开始就把完整业务流程所需的全部数据提前查询并校验，查询出来的数据对象保存到上下文对象中，业务逻辑需要底层数据实体时统一从上下文获取。

所有数据集中在“上下文构造”步骤中查询，整个业务流程运行在上下文对象中。通过上下文引入，我们基本上解决了数据重复查询问题，且集中查询有助于采用数据库批量查询提升性能。上下文构造过程也成为数据校验过程，可以提前发现异常数据，避免写入脏数据和不必要的回滚。

但上下文机制引入了新的痛点：数据模型中对象较多且关系复杂，导致上下文构造逻辑冗长。同一业务域内不同接口使用的上下文属性有较大重叠但也有差异，出现大量重复编码或混乱封装。例如询量单的新建接口与修改接口对应的上下文有 80% 属性相同，这些属性的查询和关联逻辑造成重复编码。此外，上下文机制并未从根本上解决多场景下业务流程差异复杂度高的问题。

### 第三代：数据模型与业务模型的分离

在第二代中，上下文对象的定义依然与接口方法绑定。每暴露一个服务就定义一个上下文对象，甚至不同场景有各自的构造逻辑，系统的数据模型仍隐藏在业务逻辑中。

我们逐渐意识到：多场景化的业务特性形成了一个动态的业务模型（即业务规则集），但数据模型是相对静态的。数据模型的多场景化程度远小于业务规则的多场景化程度。也就是说，同一功能模块在不同场景下的业务规则存在差异，但它们操作的是同一套数据模型。虽然不同场景对数据对象的属性设置不同（某些场景属性为 null），但数据模型本身（实体及实体间关系）并未随场景改变；属性值的设置与校验属于业务模型范畴。

明确这一点之后，为解决上下文构造复杂度高及重复编码问题，我们将数据模型分离并下沉，引入领域驱动设计（DDD）中的“聚合”概念。通俗理解，聚合是一组关联密切且关系明确的实体或值对象集合，通常支撑一个功能内聚的上层业务模块。聚合定义一个聚合根对象，聚合根是实体操作的中心，聚合中的全部实体可通过聚合根直接或间接访问。聚合根易确定，例如计划聚合的聚合根是 Campaign 实体，可以通过 Campaign 聚合根引用计划下的预算、投放时段等子实体。

将聚合落地到代码架构需要以下升级：

1. 根据业务流程设计合理的数据模型。数据模型中的实体不必与底层库表一一对应，应从业务本质出发完成实体划分和定义，并在模型中体现实体之间的关联关系。

2. 在业务流程与底层数据库之间增加聚合层。在该层中将第一步设计的数据模型定义为 Java 对象，实体之间的关系转化为类与属性的关系。例如 AdGroup 领域对象除体现 ad_group 表字段外，还定义了单元下的人群、流量包、创意列表等子实体属性。

3. 上层业务对聚合中实体的访问和修改都通过聚合根实现，聚合根需通过聚合层暴露的 Repository 接口获取。

Repository 层接口是完全面向数据模型定义的，几乎与具体业务无关，通常不会为特殊业务场景定义专用的数据查询或写入方法，而是定义通用的数据访问接口，让上层业务以声明式方法获取所需聚合根对象或集合。Repository 将数据查询和实体关系组装逻辑屏蔽在实现中，上层业务无需再次执行子实体查询和关联逻辑。

引入聚合后，上下文的构造和数据写入流程大幅简化。由于上下文中的很多数据对象被转移到聚合中，繁琐的数据查询和关联逻辑分离下沉到 Repository 的实现中，业务模型中不同服务接口可以直接复用 Repository 中沉淀的数据查询和组装逻辑，上下文构造得以精简，重复编码问题得到根本性解决，体现“代码易复用”的目标。

除了灵活复用数据查询与组装逻辑，聚合的引入实现了数据模型与业务模型的分离。聚合层几乎与业务流程无关，直接体现数据模型全貌。新同学可以通过阅读聚合层代码获取完整、准确的数据模型定义，不再需从代码中四处搜集对象关联关系，体现“逻辑易传承”的目标。

聚合还有助于简化例如 RE 降级后的补数逻辑等长期头痛的问题。本文仅探讨引入聚合的动机，关于数据模型设计、Repository 接口实现与使用的实战内容将在多体系架构的数据模型管理体系中深入讨论。个人实践经验表明，引入聚合是最让我感到改善显著的一项改进，但这种收益需要在实践中体会。熟悉 DDD 的读者可能已嗅到一些 DDD 的味道，后续文章会详细阐明设计细节和考量。

### 第四代：领域能力拆分与编排

引入聚合解决了数据查询逻辑复用的问题，但多平台、多维度、多场景带来的业务复杂度仍然存在。解决思路来源于组合复用原则（composition over inheritance）：复杂问题拆分为互不干扰的最小子问题，之后通过组合编排实现完整功能，从而将实现复杂度由乘法降为加法。

作为典型的 2B 平台，我们的业务流程冗长复杂，由多个流程节点组成。例如单元新建流程可分为基础信息设置、单元名称设置、投放周期设置、投放位置设置、定向设置、出价设置、关键词设置等节点。再叠加不同产品线（展位、快车、触点）、站外媒体（头条、腾讯、百度、快手、京X）、不同投放平台（京准通、流量货币化、京易投）以及不同站点（国内、泰国、印尼、出海）等维度，系统业务复杂度呈 O(n1* n2* ... *nm) 形式，其中 nx 为各维度的场景复杂度。组合复用原则专为解决该问题而生。

基于此思想，我们在新架构中引入了“领域能力拆分与编排”机制。

#### 领域能力的识别与拆分

在新架构中，一个完整业务流程被正交分解为多个“能力节点”。“正交分解”指拆分出的子模块互不干扰、可独立迭代。举例：早期有人将单元新建流程拆分为“出价信息校验”和“出价设置”，但这不合理，因为校验与设置并不正交，二者应合并为“出价设置”节点。

能力节点定义系统中原子模块的功能范围。一般一个能力节点包含一个能力门面和 0 到多个能力实例。能力门面不承接具体业务逻辑，而是对外暴露统一调用入口并负责请求转发；具体业务逻辑由能力实例承接。例如出价设置节点会按出价类型划分为手动出价、tCPA、MC、eCPC 等领域能力实例；在人群定向设置节点下则有京选店铺人群、乐高人群和自定义人群等能力实例。

#### 能力编排与请求路由

将系统划分为独立能力节点后，需要通过能力编排将这些节点串联成完整服务。能力编排就是把业务流程中所需的原子模块对应的能力节点串联起来，定义它们之间数据传递方式和编排规则。能力编排操作的是能力节点而不是能力实例：在处理请求时，每个能力节点负责将请求路由到正确的领域能力实例进行处理。这样设计的原因在于：业务流程相对稳定，服务流程中的节点与执行顺序很少变化，而需求迭代通常在某个能力节点上横向拓展（增删或修改能力实例）。通过能力节点的抽象与路由，我们把动态变化的部分从相对稳定的流程中分离出来，以保障核心流程稳定性不被频繁变化影响，这与我们此前将数据模型与业务模型分离的动机一致，本质上都是在隔离变化。

能力编排示例说明了：路由机制将能力门面与其下承接不同场景具体业务规则的能力实例打包，并将原子业务模块内的场景复杂度封装在模块内部。上层业务流程只需关注完整请求需要哪些能力节点，而无需关注节点下具体能力实例。请求到来时，处理流程流经相应能力节点时，通过请求上下文中的参数自动识别业务身份并路由到相应能力实例。

能力编排与路由机制在新工程中提供了框架化实现，本文不在此详述实现原理与使用方法，重点分享架构设计动机。

#### 标准的业务执行模板

在第二、三代架构中，处理请求时通常先执行全部参数校验，校验通过后将所需数据对象查询出来并校验，若合法则执行后续数据组装和处理逻辑，最后批量持久化。该模式可以充分利用批量查询接口提升性能并进行预校验。

在新架构中我们把原先连贯的业务逻辑按内聚性拆散到能力实例中，然后在领域服务中通过能力编排组装这些能力实例。若只是简单顺序执行能力实例，会导致每个能力内部又执行参数校验、依赖数据查询、逻辑处理乃至数据持久化，使代码逻辑回到“数据访问与逻辑处理交织”的第一代架构问题。此外，不同能力可能依赖相同数据对象（例如人群包绑定与预算调整都依赖 AdGroup），若框架仅简单串联执行，必然造成数据重复查询。

为解决这些问题，我们引入了标准的业务流程 Executor 模板。该模板将业务流程抽象为：参数校验、上下文初始化、上下文校验、业务逻辑处理、数据持久化、发布事件等标准步骤。无论领域能力的封装还是领域服务实现都必须继承该模板。模板一方面规范开发者的设计与实现，另一方面把逻辑串联执行权从开发者转移到能力编排框架，使框架可自动重组和执行逻辑，开发者专注业务逻辑并进行填空式开发。

框架在获得代码逻辑串联执行权后，可以在领域服务的每个标准步骤中，按照能力编排执行图组装调用各能力实例中对应的标准步骤，从而把打散到不同能力实例中的业务逻辑按标准步骤还原为连贯完整的业务流程。由于标准流程模板对每个步骤的方法参数、依赖上下文及返回对象进行了通用抽象，能力编排框架可以在能力间插入参数及上下文的映射与传递逻辑，实现不同能力以及能力与领域服务之间的数据分发与共享。尽管这些流程可采用默认自动处理规则，开发者也可通过能力编排框架提供的 DSL 修改默认串联执行、数据传递与异常处理规则。

通过领域能力拆分，我们将复杂问题正交分解为互相独立的最小问题域，使设计者能够分而治之、逐个击破，降低问题复杂度和设计成本。单个能力节点下不同业务场景的逻辑被分离到不同能力实例，避免不同业务场景互相交织，便于迅速梳理业务逻辑与定位改动点，体现“设计易拓展”的目标。

由于拆分出的各能力节点彼此正交、内聚，可在各自维度上独立迭代。例如单元维度下的出价设置与人群设置分别在出价类型与人群类型维度进行路由，避免场景交织导致圈复杂度上升，也能更灵活地在不同场景中复用能力。同时，物料新建流程中的能力往往可直接复用在物料修改流程中。对于批量物料操作，借助能力编排框架提供的循环编排与数据共享机制，我们可以在领域服务开始阶段批量完成所需数据查询，然后通过循环编排复用单请求处理能力中的纯内存校验与处理逻辑，最后在批量操作领域服务中批量完成聚合根对象集合写入，从而在保证数据准确性与性能的同时实现逻辑复用，体现“代码易复用”的目标。

领域能力的编排逻辑提供业务流程全景视图：新同学可通过阅读能力编排逻辑快速建立业务宏观认知，结合第三代的聚合机制，可快速熟悉数据模型与业务流程。通过路由机制，系统中的业务规则被打包拆分成数量有限且边界清晰的能力节点。在梳理需求对应业务规则时，可以由粗及细：先确定需求点归属的能力节点，再根据场景定位到具体能力实例，从代码中获取业务规则，体现“逻辑易传承”的目标。

基于能力拆分与编排的代码架构，最显著的收益是同一能力可在不同领域服务中直接复用。

## 04 总结

以上是我们为实现新架构所进行的尝试。这些设计是否完全正确仍需通过实际需求验证。发布这些内容并非为说服大家认同，而是希望通过架构演进推演帮助大家更好理解新架构各项功能的设计动机，从而更快上手开发；同时也希望激发大家的思考与讨论。一个好的架构通常在不断批评与改进中逐步完善。我至今怀念与永亮（我的良师益友，部门内探索中台化及领域驱动设计思想的第一人）争论到凌晨2、3 点的那些日子。
