## 正文开始

### 二叉树的致命缺陷

场景还原：商品分类表 prod_category 的 parent_id 字段建立索引。

```sql
CREATE TABLE prod_category (
  id INT PRIMARY KEY,
  parent_id INT,
  INDEX idx_parent (parent_id)  -- 二叉树实现
);
```

性能灾难：当层级超过 10 层时，查询子类目需要递归遍历。示例原始递归代码（带注释）：

```java
// 原始递归代码（林渊的注释版）
public List<Long> findChildren(Long parentId) {
  // WARNING! 每次递归都是一次磁盘寻道（8ms）
  List<Long> children = jdbc.query("SELECT id FROM prod_category WHERE parent_id=?", parentId);
  for (Long child : new ArrayList<>(children)) {
    children.addAll(findChildren(child)); // 指数级 IO 爆炸
  }
  return children;
}
```

林渊的实验数据（2004 年实验报告节选）：

- 数据量 1 万，树高 14，平均查询时间 112ms
- 数据量 10 万，树高 17，平均查询时间 136ms
- 数据量 100 万，树高 20，平均查询时间 160ms

### 二叉树的机械困境与复杂度陷阱

1970 年代，二叉搜索树（BST）的理论时间复杂度 O(logN) 掩盖了物理实现的致命缺陷。以机械硬盘为例：

- 树高灾难：100 万数据产生约 20 层高度（log₂(1,000,000)≈19.9），假设每次 IO 耗时 8ms，单次查询需 160ms。
- 局部性原理失效：随机磁盘寻道导致缓存命中率趋近于 0。
- 当所有节点都偏向一侧时，二叉树会退化为“链表”。

## B Tree

林渊的竞争对手 Jake 说：我设计了一个 B Tree。B 树通过多路平衡设计解决了磁盘 IO 问题。根据《数据库系统实现》的公式，B 树的阶数 m 与磁盘页大小的关系为：

m ≥ (PageSize - PageHeader) / (KeySize + PointerSize)

以 MySQL 默认 16KB 页为例（PageHeader 约 120 字节，KeySize 8 字节，Pointer 6 字节）：

m ≈ (16384 - 120) / (8 + 6) = 1162

在 10 万数据量下，B 树高度仅需 2 层（log₁₁₆₂(100000)≈2），查询 IO 次数从 17 次降为 3 次（根节点常驻内存），总延迟约 24ms。

首先定义一条记录为一个二元组 [key, data]：key 为记录的键值（如主键），data 为该行记录中除主键外的数据。对于不同记录，key 值互不相同。

B-Tree 中的每个节点可以包含大量关键字信息和分支。下图示意一个 3 阶的 B-Tree（图示占一个磁盘块，包含关键字与指针）：

（图示：3 阶 B-Tree）

每个节点占用一个盘块的磁盘空间，一个节点上有两个升序排序的关键字和三个指向子树的指针，指针存储子节点所在磁盘块的地址。两个关键词划分成的三个范围域对应三个指针指向的子树的数据范围。

以根节点为例，关键字为 17 和 35，P1 指针指向的数据范围为小于 17，P2 指向 17~35，P3 指向大于 35。模拟查找关键字 29 的过程：

1. 读取根节点所在磁盘块到内存（磁盘 I/O 第 1 次）。
2. 比较关键字，找到区间（17,35），得到指针 P2。
3. 读取 P2 指向的磁盘块到内存（磁盘 I/O 第 2 次）。
4. 比较并找到下一层指针 P2。
5. 读取该指针指向的磁盘块（磁盘 I/O 第 3 次）。
6. 在该块的关键字列表中找到关键字 29。

可以看到需要 3 次磁盘 I/O 操作和若干内存查找。内存中的关键字是有序表结构，可用二分查找加速。

## B+Tree

林渊反驳道：“每个 B-Tree 节点不仅包含 key 值，还有 data 值。如果 data 较大，会导致每个节点能存储的 key 数量变少，当存储量很大时 B-Tree 的深度同样会变大，增加查询时的磁盘 I/O 次数，从而影响效率。而且 B-Tree 的非叶子节点存储数据，范围查询需要跨层跳跃。”

他想到在 2025 年学到的 B+Tree 数据结构，并在 InnoDB 源码中找到相关实现片段：

```cpp
// storage/innobase/btr/btr0btr.cc
void btr_cur_search_to_nth_level(...) {
  /* 只有叶子节点存储数据 */
  if (level == 0) {
    page_cur_search_with_match(block, index, tuple, page_mode, &up_match,
                               &up_bytes, &low_match, &low_bytes, cursor);
  }
}
```

“原来 B+树通过叶子层双向链表，把离散的磁盘页变成了连续空间！”整理后，他总结 B+树通过以下创新实现质的飞跃：

1. 全数据叶子层：所有数据仅存储在叶子节点，非叶节点仅作索引目录。
2. 双向链表串联：叶子节点通过指针形成有序链表，范围扫描时间复杂度从 O(logN) 降为 O(1)（顺序扫描时每一步仅需顺序访问下一个叶子页）。

在 B+Tree 中，所有数据记录都按键值大小顺序存放在同一层的叶子节点上，非叶子节点只存储 key 信息，这可以大幅增加每个节点能存储的 key 数量，降低树的高度。通常 B+Tree 有两个头指针：一个指向根节点，另一个指向关键字最小的叶子节点，且所有叶子节点之间形成链式（可为环形或单链）结构。因此可以进行两种查找运算：一种是从根节点开始的随机查找（O(logN）），另一种是从叶子链开始的范围查找或分页查找（O(1) 的顺序访问开销）。

## 西湖论剑，单挑首席

一个月后，全球数据库峰会在西子湖畔召开。林渊抱着一台 IBM 服务器走上讲台：“给我 30 秒，让各位见识‘未来索引’！”现场演示：

- 场景：1 亿订单数据范围查询
- 传统 B 树（原版）：
  SELECT * FROM orders WHERE id BETWEEN 100000 AND 200000;
  耗时 12.8 秒
- B+ 树（林渊改进）：
  SELECT /*+ BPLUS_SCAN */ * FROM orders WHERE id BETWEEN 100000 AND 200000;
  耗时 0.3 秒

名场面台词：“诸位，这不是优化，是维度的碾压！B+树把磁盘的物理运动，变成了内存的闪电舞蹈！”当日登上《程序员》杂志封面，三月后林渊成立“深空科技”，发布“伏羲 B+ 引擎”。
