一、索引到底是什么
索引是帮助 MySQL 高效获取数据的、按某种顺序组织的数据结构。索引存储在文件里（磁盘上），在查询时可以减少磁盘 I/O，从而提高查询速度。没有索引时，数据库可能需要对每条记录逐一扫描，导致大量 I/O，速度非常慢。为了提高性能，会尽可能将常用数据放到内存缓存中，减少磁盘访问。

二、磁盘存取原理
磁盘读取数据涉及物理寻道和旋转延迟，每次从磁盘读取数据都涉及一次 I/O，I/O 是非常耗时的操作。为了减少 I/O，系统会按页（page）读取数据并缓存。操作系统的内存页通常是 4KB，而数据库（如 InnoDB）默认的页面大小通常是 16KB。索引和数据按页组织，可以一次把相邻的数据一起读入内存，从而提高读取效率。

三、MySQL 常用数据结构详解
在没有索引的情况下，如果数据量很大，线性扫描的代价非常高。为此需要合适的数据结构来降低查找次数。下面逐一介绍常见的数据结构及其在数据库索引中的适用性。

3.1 二叉树
二叉查找树（BST）每个节点保存一个键。随着数据增加，树的高度会增加，平均查找比线性扫描快很多。但在极端情况下（例如插入排序的序列），BST 可能退化为链表，导致性能退化到线性查找。因此单纯的二叉树并不适合作为磁盘索引的主要结构。

3.2 红黑树
红黑树是一种自平衡二叉树，能避免单边增长，通过旋转保持高度对数级别。它在内存中非常有效（如 C++ 的 std::map）。但是用于磁盘索引时仍有问题：树的高度虽小但节点较多，且红黑树以节点为单位，节点通常较小（一个键一个节点），在磁盘 I/O 场景下不能有效利用一次 I/O 读取大量相邻键，因此并非磁盘索引的最佳选择。

3.3 HASH
哈希索引通过哈希函数可以在常数时间内定位目标，适合等值查询（=）。如果数据都在内存中，哈希查找时间复杂度为 O(1)，速度很快。但哈希有明显缺点：
- 无法支持范围查询（例如 BETWEEN、>、<）。
- 数据在磁盘上通常不是连续存放，哈希查找可能导致随机 I/O，效率反而低。
- 哈希不能利用顺序性（如排序、前缀匹配）。
因此哈希索引不适合作为通用的磁盘索引结构（InnoDB 在内存中提供了 Hash Adaptive 索引作为辅助，但不是主索引结构）。

3.4 B-Tree
B-Tree 是多路平衡查找树，引入了“大度”（degree），即每个节点可以有多个子节点，从而降低树的高度。B-Tree 的设计目的是减少磁盘 I/O 次数：每个节点对应磁盘上的一页，节点包含多个键和子指针，一次 I/O 可以读入一个节点的所有键和指针，从而在内存中完成更多比较，减少磁盘访问次数。
但在数据库中更常用的是其变体 B+Tree，B-Tree 的内部节点和叶子节点都保存数据（或指针），在某些实现上并不如 B+Tree 高效。

3.5 B+Tree
B+Tree 是数据库索引中最常用的结构（MyISAM 和 InnoDB 都使用 B+Tree 或其变体）。其特点包括：
- 所有真实数据或数据指针都保存在叶子节点，内部节点只保存键和子指针。
- 叶子节点通过链表相连，方便范围查询和顺序遍历。
- 以页为单位组织，读取一个页能获取该页内多个键，大幅减少磁盘 I/O。
- 页面大小（page size）通常是 4KB 或 16KB，MySQL/InnoDB 默认 16KB，因此单个节点（页）可以容纳更多键，降低树的高度。

四、MyISAM 与 InnoDB 的索引实现差异
MyISAM：
- 表的文件由 .frm（表结构）、.MYD（表数据）和 .MYI（索引）等组成。
- MyISAM 的叶子节点通常存储的是数据文件的指针（record pointer），即叶子节点并不直接存放完整行数据，而是指向数据文件中的位置。
- 主键索引与普通索引是各自独立的树结构。

InnoDB：
- InnoDB 使用聚集索引（clustered index），表数据本身就是主键的 B+Tree。换言之，数据文件就是按主键组织的 B+Tree，叶子节点包含完整的行数据。
- 因为表数据按主键组织，InnoDB 表必须有主键；若没有显式主键，InnoDB 会选择一个隐含的聚簇索引（例如隐含的 ROWID）。
- 二级（非主键）索引的叶子节点不存放完整行数据，而是存放主键值。查询二级索引命中后，需要通过主键再回表到聚簇索引中取出完整行数据。这样做的原因是节省空间并保持数据一致性。

为什么推荐整型自增主键？
- 整型自增主键有利于插入操作集中在 B+Tree 的右端，减少页分裂和随机 I/O，性能更稳定。
- 主键值越短，二级索引中保存的主键占用空间越小，从而节省存储并提高缓存命中率。

五、联合索引与最左前缀原则
联合索引（复合索引）比如 (col1, col2, col3）在底层也是一棵 B+Tree。内部节点按 col1 的值组织，叶子节点按 col1、col2、col3 的顺序组合排序并保存这些键值（以及指向数据或主键的指针）。

最左前缀原则（Leftmost Prefix Rule）：
- 联合索引能被利用的前提是查询条件中包含索引的最左侧字段或最左侧字段的前缀。
- 示例：假设表 T 有索引 (A, B, C)。

以下查询是否能使用该索引：
- SELECT * FROM T WHERE B = '赵六' AND C = 123;
  无法使用 (A, B, C) 索引，因为没有提供最左列 A 的值（索引首先按 A 排序）。
- SELECT * FROM T WHERE A = '30';
  可以使用索引：先定位 A=30 的区间，然后在该区间内按需要继续筛选（如 B、C）。
- SELECT * FROM T WHERE A = '18' AND C = 345;
  可以部分使用索引：A 可以用于定位，但由于中间列 B 没有条件，C 无法利用索引进行直接查找（只能在满足 A 的结果集中进一步筛选 C）。
- SELECT * FROM T WHERE B = 'DE' AND C = 1234 AND A = '2';
  虽然条件顺序不同，但包含了 A，因此查询可以使用索引，数据库会根据索引的列顺序进行利用。

总结：
- 索引的目的是减少磁盘 I/O，提高查询性能，但不同的数据结构适用于不同的场景。
- B+Tree 是关系型数据库中最常用的磁盘索引结构，适合等值和范围查询。
- InnoDB 使用聚簇索引（主键即数据），二级索引存放主键值，查询二级索引时需要回表。
- 联合索引应遵循最左前缀原则，设计索引时要考虑查询模式以确保索引能被有效利用。
