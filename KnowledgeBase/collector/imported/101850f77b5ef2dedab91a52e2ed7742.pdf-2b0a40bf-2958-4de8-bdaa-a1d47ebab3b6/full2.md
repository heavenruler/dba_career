# Redis 脑裂问题

作者：Java中文社群  
专栏：Java面试精选

## 概述
Redis 脑裂问题是指在 Redis 哨兵模式或集群模式中，由于网络原因导致主节点（Master）与哨兵（Sentinel）或从节点（Slave）之间的通信中断。哨兵可能误以为主节点已宕机，于是在从节点中选举出一个新的主节点，从而在集群中出现两个主节点的情况，这就是所谓的“脑裂”（split-brain）问题。

## 脑裂导致数据丢失的过程
Redis 脑裂会导致数据丢失。下面是典型的产生过程（简化并按顺序说明）：

1. 网络分区导致旧 Master 与多数节点失去通信。
2. 哨兵认为旧 Master 宕机，发起故障转移，选举出新的 Master（原来的某个 Slave）。
3. 故障转移完成后，旧 Master 恢复网络连接，但它的角色尚未被调整为 Slave。
4. 最后一步，当旧 Master 被切换为 Slave 时，其执行流程如下：
   - 旧 Master（现在作为 Slave）向新的 Master 申请全量数据。
   - 新 Master 通过 bgsave 生成当前 RDB 快照，并将 RDB 发送给旧 Master。
   - 旧 Master 拿到 RDB 后，先执行 flush 清空当前数据（此时在旧 Master 恢复网络后客户端写入的数据会被清空，导致数据丢失）。
   - 然后加载 RDB，初始化数据。

可以看到，在执行到“flush 并加载 RDB”这一步时，旧 Master 恢复网络后接收到并处理的客户端写入数据会被覆盖或丢失，这就是数据丢失的根本原因。

## 如何解决脑裂问题？
解决思路是：在旧 Master 恢复网络并切换为 Slave 的期间，不应接受客户端的写入。Redis 提供了两个配置项，能尽量避免这类数据丢失：

- min-slaves-to-write：要求与主节点保持通信的从节点数量必须大于或等于该值，否则主节点将拒绝写入。
- min-slaves-max-lag：要求主节点与从节点之间的复制延迟（以秒计）必须小于该值，否则主节点将拒绝写入。

这两个配置项必须同时满足，否则主节点会拒绝写入。若在假故障期间未能满足这两个条件，主节点将被禁止写入，从而避免脑裂期间的数据不一致和丢失。
