# 真·Redis缓存优化实践——97%的优化率你见过吗？

作者：京东科技 韩国凯  
来源：京东云开发者社区（转载请注明来源）

本文通过一封 618 前的 R2M（公司内部缓存组件，可认为等同于 Redis）告警，由浅入深分析该告警的直接原因与根本原因，并根据原因提出相应的解决方法，期望为大家排查类似问题提供思路。

## 一、问题排查

告警邮件内容（节选）：
> 您好，R2M 监控报警，请您及时追踪一下！  
> 报警信息：告警 ID:6825899，应用: zr_credit_portal，责任人: zhangsan，告警类型: 内存使用率，时间: 2023-06-15 16:00:04。  
> 实例: (10.0.0.0:5011-slave)，当前: 9212MB，超过警戒值: 8748MB，实例最大内存: 10800 MB，内存使用率: 85%；  
> 实例: (10.0.0.0:5023-master)，当前: 9087MB，超过警戒值: 8748MB，实例最大内存: 10800 MB，内存使用率: 84%；  
> 实例: (10.0.0.0:5017-master)，当前: 9214MB，超过警戒值: 8748MB，实例最大内存: 10800 MB，内存使用率: 85%。

我们的缓存集群配置如下：总容量 32400 MB，三主三从，每个主节点 10800 MB。目前使用最高达到 9087 MB。

首先的思路就是统计大 key，查看哪些缓存占用了容量。因为大 key 统计是从节点进行扫描，所以不用担心会影响线上主流程。

### 1.1 代码分析

通过大 key 分析，主要有两类大 key：一类是 xxx_data，一类是 xxx_interfacecode_01，按此规律在代码中找到相关位置。代码示例（已修正）：

```java
String dataKey = task.getTaskNo() + "_data";
cacheClusterClient.setex(dataKey.getBytes(), EXPIRATION, DataUtil.objectToByte(paramList));

String key = task.getTaskNo() + "_" + item.getInterfaceCode() + "_" + partCount;
cacheClusterClient.setex(key.getBytes(), EXPIRATION, DataUtil.objectToByte(dataList));
```

找到代码位置后，分析其业务流程。

### 1.2 告警原因

综合分析，此次占用率过高的原因可分为直接原因与根本原因。

#### 1.2.1 直接原因
运营后台发现有用户在此前三天大量创建跑批任务，导致缓存中样本与结果数量增加，从而使缓存使用率过高。

#### 1.2.2 根本原因
代码中缓存主要有两块数据：样本与结果。

- 样本在缓存中被短暂存储后又取出，这一操作毫无意义，仅占用缓存容量，应避免。
- 结果采用分批分片存储，此举有意义，是为了防止多任务并行时数据在 JVM 中占用大量内存，导致 Full GC。因此分片本身合理。
- 跑批结束后，中间数据已无用，但业务流程没有主动删除，而是等待超时自动删除。这样会导致数据在缓存中停留较长时间。

虽然 TTL 被设置为 1 天，但监控显示缓存三天内持续上升，并没有如预期那样下降很多。怀疑是数据虽然 TTL 变为 0，但并未立即被物理删除，和 Redis 的渐进式删除机制有关。

Redis（R2M）文档指出：如果带有生存时间的键非常多的话，那么在键的生存时间变为 0 到键真正被删除这中间，可能会有一段比较显著的时间间隔。也就是说大量已过期但未被访问的 key，可能因渐进删除速率慢而长时间占用内存。

至此，直接原因的“根本原因”已找到：大量 key 到期后并未及时被物理删除。

## 二、问题解决

上文分析了告警的排查过程，以下分为如何解决直接原因与根本原因。

### 2.1 直接原因

#### 2.1.1 原因分析
在 618 前夕，不适合对生产环境做大改动，短期内需要尽快阻止用户继续创建跑批以避免内存继续被占用。但同时观察到 TTL 已设置为一天，但内存并未下降，说明存在过期未被及时删除的问题（与 Redis 渐进删除特性相关）。

Redis 删除过期键的两种方式：
- 惰性删除：当一个键被访问时，系统会检查其是否已过期，如果已过期则删除。
- 定期（渐进）删除：后台会在一定频率下扫描并删除过期键，以处理那些不会被访问到的键。

当大量 key 都设有 TTL 且这些 key 不再被访问时，惰性删除不起作用，系统就依赖后台渐进删除的速率，如果该速率被降低，会导致大量过期 key 长时间存在。

#### 2.1.2 解决
我们联系了 R2M 运维，确认有参数可以调整渐进式物理删除的频率。发现集群此前因为项目调整被设置为 10（比默认或期望值低很多），降低了删除速率。618 前未直接修改参数，节后提交工单将该参数从 10 提高到 80。

调整后观察到 R2M 的内存使用下降速率明显加快。在没有大量新数据加入的情况下，缓存使用率回落，验证了猜想：大量过期 key 因删除速率低未及时释放内存。

至此，告警的直接原因已解决：大量 key 到期但未及时删除导致内存占用。

### 2.2 根本原因

虽然提高渐进删除速率解决了过期 key 未及时释放的问题，但若用户在一天内仍有大量跑批任务（短时间内大量写入）时，依然可能导致内存短期内被大量占用，影响线上业务。

要从根本上解决问题，需要优化跑批流程：

- 样本数据无需存入缓存：样本通过参数传递给下一步流程，避免占用缓存。
- 结果分片数据虽然有意义，但 Redis 内存较为宝贵，而 OSS 等对象存储的成本较低且持久。由于业务为离线任务，对时效性和查询速率要求不高，可将跑批结果分片存储到 OSS。
- 在跑批流程结束后主动删除 OSS 中的结果分片，避免无用数据长期占用存储；同时在 OSS 端设置自动删除策略（例如 7 天）以防异常未清理导致长期存在。

综上，重新设计流程后（改为使用 OSS 存储结果分片、主动清理与设置自动过期），即便用户在短时间内大量创建跑批任务，也不会再因缓存被占满而触发内存告警或影响线上业务。

## 三、总结

通过一封告警邮件，由浅入深分析系统缓存问题并进行流程优化，本文给出一些心得供参考：

### 3.1 不同中间件应该负责不同的事
要根据数据特性选择合适的中间件。Redis 适合少量、高速访问的数据；大量、不要求极高速度的离线数据适合 OSS、数据库或数据仓库（Mysql/Hive/ES/ClickHouse 等）。选择合适的存储能够避免把不适合放到 Redis 的数据放入 Redis，从而减少风险与成本。

### 3.2 学习技术细节是有用的
理论与实践结合很重要。事发前作者刚学完 Redis 的相关特性，能快速联想到渐进式删除机制并定位问题，说明学习实现细节在实际排查中非常有价值。此外，将理论用于实践能够增强学习兴趣与效果。
