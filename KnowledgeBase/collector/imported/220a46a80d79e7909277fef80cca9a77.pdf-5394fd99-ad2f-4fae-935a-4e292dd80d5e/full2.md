# 用蜜蜂(eBPF)来追踪海豚(MySQL)，性能追的上吗

作者：聚好看 DBdoctor  
日期：2024-06-07

DBdoctor 基于 eBPF 实现了数据库可观测，能在不改代码、不改启动参数、且不关注重启进程的前提下对数据库内核进行追踪，拿到更细粒度的内核数据用于性能分析。有人会问：用 eBPF 观测 MySQL 会不会是个“玩具”？能否用于生产？每一条 SQL 的执行过程都被探测，对数据库性能的消耗到底有多大？本文通过严谨的压测对比，用数据来回答这些问题。若想快速看结论，可直接查看最后的总结。

## 测试目标与设计

目标：评估 Agent（基于 eBPF 的探针 + 用户态处理）对 MySQL 业务性能和自身资源消耗的影响，尤其在高 QPS / 极端负载下的表现。

我们设计了以下测试场景与评估维度：

- 评估 Agent 对业务性能的影响
  - 场景 A：基准压测（读写、只读、只写模式），使用 sysbench，不同并发下测量 TPS/QPS、延迟。
  - 场景 B：TPC-C（典型 OLTP 交易场景），评估对事务处理能力的影响。
  - 场景 C：单条 SQL 请求消耗测量（探针触发开销测量）。
  - 每个场景对比两种状态：停用 Agent（基线）与启用 Agent（默认配置），并在不同 TPS 下持续增压到业务极限，评估影响是否随压力变化。
- 评估 Agent 自身的处理性能
  - 场景 A（只采主机性能指标）与场景 B（开启全功能采集并增加 QPS），记录 Agent 进程的 CPU/MEM 占用随 QPS 的变化。

所有测试中 DBdoctor Agent 均使用默认配置，无任何调优。

## 测试环境

- 数据库服务器：ECS（独享 CPU）
- CPU：Intel Xeon Platinum 8269CY @ 2.50GHz
- 主机规格：16c16g（机器内存 32GB DDR4）
- MySQL 版本：5.7.37
- 操作系统：CentOS Linux 8
- 内核版本：4.18.0-348.7.1.el8_5.x86_64
- Agent 容器资源在测试中限制为 1c1g 或 2c1g（用于对比）

每个压测点运行 30 分钟，取 5 次均值，分别记录 TPS/QPS、延迟，以及 Agent 与 MySQL 的 CPU/MEM 消耗。

## 测试结果

### 场景 A — 基准压测（读写 / 只读 / 只写）

测试以 MySQL 4c8g（Agent 限 1c1g/2c1g）为例，分别在并发用户数 1/2/4/8/16/32/64 下进行压测。

a) 读写模式（读写混合）
- 对照组 vs Agent 组对比结果：
  - QPS 下降约 1.05%
  - 平均延迟提高约 2%
  - 每 1W QPS MySQL 的 CPU 占用增加约 0.2c
  - 内存使用无明显变化
- 说明：主要开销来自高并发下大量的 perf 事件提交（内核探测相关的提交开销）。

b) 只读模式
- 观察：
  - 无 Agent 时只读峰值 QPS 可达 ~45k；当 CPU 饱和时 QPS 可能下降到 ~35k。
  - 启用 Agent 会消耗额外 CPU（相当于 eBPF 执行占用），示例中约占用 0.9c（4c 情况下），导致在 CPU 打满时的 QPS 峰值下降；若为该实例额外增加 ~0.9c CPU，吞吐量可恢复到无 Agent 的水平。
  - 内存使用无变化。
- 说明：在只读高并发场景下，Agent 的主要影响体现在占用 MySQL 可用的 CPU 核心数量上。

c) 只写模式
- 结果：
  - QPS 无明显下降
  - 延迟无明显变化
  - 每 5k QPS 的写负载下，MySQL CPU 占用增加约 0.1c

总体结论（场景 A）：在读写与写密集场景下，Agent 对业务影响很小（QPS/延迟变动在几乎可忽略范围）。在极端只读且 CPU 接近饱和时，Agent 会占用一定 CPU（可通过增加少量 CPU 资源弥补）。

### 场景 B — TPC-C（OLTP 交易场景）

对比结果：
- QPS 下降约 1.28%
- 平均延迟提高约 2%
- 每 1W QPS MySQL 的 CPU 占用增加约 0.2c

说明：场景 B 与场景 A 的结论基本一致，说明 Agent 的主要消耗与 QPS 成正相关，约每 1W QPS 占用 ~0.2c CPU。

### 场景 C — 单条 SQL 请求消耗（探针开销）

通过在探针内打印并记录时间戳统计（uprobe/bpf_trace_printk），得到如下典型值：

- SQL 请求（entry）原始延迟：119,222 ns（≈0.119 ms）
  - eBPF 程序消耗约 3,000 ns（≈0.003 ms）
  - 合计约 0.122 ms
- SQL 返回（return）原始延迟：21,158 ns（≈0.021 ms）
  - eBPF 程序消耗约 4,500 ns（≈0.0045 ms）
  - 合计约 0.025 ms

综合两端探针的额外开销，使单条 SQL 的平均延迟增加约 0.147 ms（147 μs）。

说明：单条 SQL 的延迟增加在微秒级别，总体对业务影响取决于 QPS；当 QPS 很大时，累积的 eBPF 执行消耗会体现为明显的 CPU 占用。

## Agent 进程自身资源消耗

测试过程中记录 Agent 在不同场景下的资源占用，关键观测点如下：

- 只采集主机性能数据（不采集 MySQL 详情）时：
  - Agent CPU 占用约 0.01c
  - Agent 内存占用约 0.028 GB
- 增加一个被管理的实例（该实例无压力）时：
  - Agent CPU 占用增加约 0.04c
  - 内存增加约 0.14 GB
- 当采集的实例每增加 1W QPS：
  - Agent 的 CPU 消耗增加约 0.22c（内存无明显增长）
- 示例：在采集状态下（MySQL QPS ≈ 20k），Agent 使用约 0.85c CPU，内存约 0.24 GB

说明：Agent 的 CPU 消耗与被采集实例的 QPS 呈正相关，内存占用相对固定且较小。

## 总结

- Agent 的主要消耗来自 eBPF 内核态探测（触发时在函数进入/返回处执行 eBPF 程序）与用户态的数据组装/网络投递。总体开销主要体现为 CPU。
- 量化结论：
  - 每 1W QPS 大约消耗 0.2c 的 CPU（Agent + MySQL 因探针导致的额外占用合计）。
  - 对业务的整体性能损耗通常控制在约 1%~2%（延迟和吞吐波动在可接受范围内）。
  - 对 MySQL 内存占用没有明显影响；Agent 本身内存占用小（数百 MB 级别）。
- 在极端只读高并发场景，Agent 会占用 MySQL 可用的 CPU 核心（例如示例中约 0.9c），可通过为实例分配额外少量 CPU 来恢复吞吐。
- 单条 SQL 的探针开销在微秒级（示例中约增加 0.147 ms），但累计效果随 QPS 放大而显著。
- 结论：在默认配置下，基于 eBPF 的 DBdoctor Agent 在大多数业务场景中对性能影响很小，可用于生产环境。其优势是能够在不改代码、不改启动参数的前提下获取内核级和更细粒度的数据库指标，便于精确诊断与优化。
