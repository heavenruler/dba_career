# TiDB 资源管控的原理与实践

TiDB Club — 2024-10-31 18:03 北京

在数据库运维管理中，资源管控技术扮演着至关重要的角色。TiDB 7.0 版本引入的资源管控功能，不仅保障了在负载波动时的服务质量，还实现了多租户环境下的资源隔离，从而降低了数据库的运行成本。本文涵盖了 TiDB 资源管控的设计思路、配置策略以及实际应用场景，旨在帮助读者深入理解 TiDB 资源管控的内在机制，并掌握其在提升数据库性能和优化成本效益方面的应用。

## 新特性解析

### 背景
资源管控旨在解决共享集群中业务负载突变影响其他业务、混合负载下资源竞争、高可用性业务系统中后台任务影响服务质量，以及灰度上线时小流量引发性能问题等挑战。TiDB 在 6.6 引入资源管控技术，并在 7.0 版本中增强，通过资源组划分逻辑单元，限制资源使用，并允许在集群资源空闲时超额使用，从而优化资源利用。

### 核心优势
- 双层资源管控：TiDB 实现了包含流量控制（TiDB 层）和调度控制（TiKV 层）的双层资源管控机制，以确保操作在资源限额内执行，并在负载变化时优先处理高优先级任务。  
- I/O 资源限制：基于存储计算分离架构，TiDB 对关键的 I/O 资源进行限制，解决数据库系统的资源瓶颈问题。  
- 资源抽象与统一单位：TiDB 将资源抽象成统一单位（RU，Request Unit）进行分配和限制，统一衡量 CPU、IOPS 和 IO 带宽，简化管理并保持灵活性，以自适应地完成最佳调度。  
- 资源利用率提升：在资源充足时，TiDB 允许业务超过资源组限制（BURSTABLE 模式），以提高资源的整体利用率。  
- 用量统计与反馈：TiDB 提供精确的用量统计反馈，帮助用户通过监控面板优化配置并进行成本分摊。  
- 灵活的资源绑定：支持在不同级别（用户级、会话级、语句级）绑定资源，以适应多样化的资源控制场景。

## 管理方式

资源管控引入了“资源组（Resource Group）”的概念，通过设置“用户”和“资源组”的对应关系，把会话与用户组绑定，利用“用量（RU）”对资源限额进行定义。

- 资源组（Resource Group）：TiDB 中用于资源管理的逻辑单元，每个会话都归属于一个资源组，共享该组的资源限额。TiDB 允许将数据库用户映射到不同的资源组，未明确指定资源组的用户则默认关联至系统预设的 default 资源组。  
- 资源限额：通过资源组（RU）配置用量，统一衡量 CPU、IOPS 和 IO 带宽，支持 BURSTABLE 模式超用空闲资源。用户可通过监控面板优化资源配置，并通过实际负载测试确定系统用量上限。

配置步骤（简要）：
- 启用资源管控
- 估算集群容量
- 根据实际业务目标规划和创建资源组
- 将用户（例如 usr1）的资源组设置为 rg1

## 场景演示

假定有三个租户分别运行三种不同的业务形态，每类业务有不同的管控目标。面对多租户资源管理，若采用物理隔离需要为每租户分配独立资源，导致资源浪费；若共用集群则难以避免相互影响。TiDB 的资源管控允许创建不同资源组，实现资源的有效隔离与合理分配。

示例策略：
- 为租户 oltp 分配较高的用量，租户 batch 与租户 hr 分配较低的用量，这样在系统资源紧张时可以保证 oltp 的服务质量。  
- 将 tenant oltp 和 batch 设置为 burstable，当 oltp 出现超预期负载时仍可能保证质量；当集群有空闲资源时，tenant batch 可以利用空闲资源加速其工作。

创建资源组的基本流程：
- 为租户创建对应资源组并设置配额。  
- 将业务用户映射到相应资源组。  
通过资源管控技术模拟不同租户负载后，可以及时响应负载变化，保持 oltp 租户低响应时间并限制 hr 租户吞吐量，证明 TiDB 在一套系统中能同时满足多个租户需求并提升资源效率。

## 资源管控技术协助企业兼顾质量与成本

资源管控技术解决了负载波动导致的服务质量问题，防止应用资源过载，适用于新业务灰度测试，并支持数据库多租户隔离，帮助企业降低运行成本。成本节约主要来自以下方面：
- 节省预留资源：通过合并多个业务系统到一个 TiDB 集群，减少预留空间，利用 TiDB 的透明扩展性应对负载波动和业务增长。  
- 缓解峰谷波动：整合不同负载峰谷的系统，降低整体负载波动，提高资源利用率。  
- 快速扩展与收缩：共享资源并快速调整计算资源，无需重复部署，节省人力和成本。  
- 提升运维效率：集成多个业务系统，减少管理实体和系统数量，降低运维复杂度和成本。

## 案例一：中泰证券 x TiDB（TiDB 7.1 多租户应用）

### 项目背景
在国产化系统推广背景下，中泰证券在改造中采用 TiDB 及国产软硬件，以增强自主可控能力。中泰科技研发部使用两套 TiDB V7.1 集群整合业务系统，分别服务外部和内部客户。由于这两套集群共用多套业务系统，迫切需要资源管控技术以确保每个业务系统拥有独立的资源池。

### TiDB 多租户介绍
TiDB 6.6 引入资源管控（RC），7.0 优化后通过资源组限制资源使用并引入 burst 属性以提高资源利用率。该特性可解决以下痛点：
- 业务系统间的相互影响与干扰；  
- 数据分析或批量作业对交易系统的影响；  
- 运维操作（如数据备份、统计信息收集）对服务质量的消耗。

### 具体应用和实施
- 资源评估：在 Dashboard 的 Resource Manager 中使用 Estimate Capacity 根据标准测试类型进行资源评估。  
- 应用绑定 RU：梳理数据库中的业务用户，将用户与业务系统对应，便于后续将不同资源组与不同用户绑定。  
- 观察 RU 使用情况：完成绑定后，TiDB 可以实时统计各业务消耗的资源。生产运行一段时间后，需要观察实际消耗并进行调整。

### RU 使用收益
- 可观测性：结合 Dashboard，可以清晰观察每个业务系统消耗的资源及集群整体资源是否充足。  
- 可配置性：在资源紧张时可在线调整 RU 大小，控制资源缩减对业务的影响，实现有效管理。

## 案例二：多点 DMALL x TiDB（从 100+ 套 MySQL 到一套 TiDB）

### 背景
多点 DMALL 推进零售 SaaS 私有化项目，目标提高系统效率并满足可扩展性、多租户管理和高可靠性需求。系统改造过程中通过整合应用和数据库降低复杂性与成本，计划将多个 MySQL 集群合并为一个 TiDB 集群，利用资源管控特性优化合库流程并提升成本效率。

### 迁移实施

#### MySQL 合库到 TiDB 的前提条件
在私有化项目中计划合并 QPS 较低的 MySQL 集群至 TiDB，选择原则排除了高 QPS 业务、使用 DDH 的集群，以及具有特殊数据库结构或未接入配置中心的集群。此次调整涉及 104 个 MySQL 集群，其中 75 个合并至 TiDB，29 个下线，释放了 212G 内存和 1.1T 磁盘空间。

#### 升级与扩容
- 合库目标版本：TiDB 升级至 7.5。升级前在测试环境中充分验证并做好备份兜底。通过 tiup 升级，读写延迟下降约 40–50%，性能提升明显。  
- 计算节点扩容：新增两台 TiDB 计算节点，提高性能并减少高并发写入延迟，通过智能资源管理确保数据同步和合库操作顺畅，支持业务增长。

#### 同步过程中的参数配置与资源隔离
TiDB 资源管控提供两层能力：TiDB 层的流控和 TiKV 层的优先级调度。可以在用户、会话、语句级别设置资源组，将高压应用绑定到某资源组后，TiDB 层根据配额对读写请求做流控，TiKV 层根据配额映射的优先级对请求做调度。通过流控和调度两层控制，实现应用资源隔离并满足服务质量（QoS）要求。资源隔离通过 RU（对 CPU、IO 等资源的统一抽象计量单位）实现，而非直接指定 IO 或 CPU 的具体数值。

#### 全链路压测
在 85 个数据库合并的场景中，仅对 4 个大库开启资源管控。集群预估可用 RU 为 2.7 万个，实际使用约 2 万个，其中 70% 分配给 4 个大库。RU 耗尽会导致 SQL 报错，需谨慎设置。在 7.5 GA 版本中，超出 RU 会在等待预估时间后报错（时间可调）。Runaway Queries 可以避免报错，但相关问题需要在 8.x 版本进一步优化。

### 整体收益
- 简化系统架构与提高效率：将 100+ 套 MySQL 实例合并至 1 套 TiDB，简化架构、集中运维并提升开发效率。  
- 降低硬件成本：TiDB 的高性能与资源使用效率减少硬件需求，计算存储分离架构提高资源利用率，存储成本显著下降。  
- 提升运维效率：运维成本大幅降低（例如案例中提及下降约 90%），统一监控与管理简化部署与维护。  
- 增强数据安全性：TiDB 的多副本与自动故障转移提高数据可靠性，资源隔离与动态扩缩容减少业务中断风险。
