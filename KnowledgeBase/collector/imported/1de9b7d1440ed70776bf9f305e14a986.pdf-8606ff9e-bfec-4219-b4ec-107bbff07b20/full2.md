# TiDB 底层存储结构：LSM 树原理介绍

作者：京东物流 刘家存

本文介绍分布式关系型数据库 TiDB 所采用的底层存储结构 LSM（Log-Structured Merge）树的原理，包含提出背景、算法思路、组成部分、Compact 策略、插入/修改/删除与查找方法，并与 B+ 树做比较，总结 LSM 树的特点及适用场景（如写操作吞吐量高、读操作吞吐量要求较高的场景）。

## 1 LSM 树介绍

LSM 树（Log-Structured Merge-Tree）由 Patrick O’Neil 等人在论文 The Log-Structured Merge Tree 中提出。LSM 树的核心特点是利用顺序写来提高写性能，代价是读性能会有所下降（读放大）、写入量增加（写放大）以及磁盘空间占用增加（空间放大）。

LSM 树常用于 NoSQL 数据库，如 HBase、RocksDB、LevelDB 等。TiDB 的 KV 存储引擎 TiKV 底层使用 RocksDB，因此 TiKV 的底层存储即基于 LSM 树实现。

## 2 LSM 树算法大概思路

LSM 树由两个或多个树状结构组成。下面以双层 LSM 树为例，说明基本思路。

### 2.1 数据结构

双层 LSM 树包含一个较小的驻留内存层 C0（MemTable 或 C0 树），以及一个驻留在磁盘上的较大层 C1。虽然 C1 在磁盘上，但 C1 中经常访问的节点会被缓存在内存中，可视为部分内存驻留。

### 2.2 写入

写入时，首先将数据写入顺序日志文件 WAL（用于崩溃恢复），然后把该条记录的索引项插入内存中的 C0 树。通过异步任务，内存中的数据会及时迁移到磁盘上的 C1 层。

### 2.3 读取

查找索引项时，先在 C0 中查找；若未找到，再到 C1 中查找。若发生崩溃恢复，还需读取 WAL 中未被持久化到磁盘的索引项以恢复数据。

### 2.4 Compact 过程

MemTable（C0）中的插入操作没有直接的 I/O 成本，但内存容量有限。当 MemTable 达到阈值时，需要将数据迁移到磁盘层（C1）。为此，LSM 树采用滚动合并（Compact）过程，将 C0 中的一部分或全部连续记录合并并写入到 C1。

当前常用的 Compact 策略主要有两种：size-tiered 策略和 leveled 策略，下面详细介绍这两种策略。

### 2.5 崩溃恢复

在将 C0 的项迁移到磁盘前存在延迟。为保证崩溃后数据不丢失，每次新的记录写入前，先将用于恢复该插入操作的日志写入 WAL，然后再写入 MemTable。

## 3 LSM 树的组成

LSM 树主要由以下三个部分组成：MemTable、Immutable MemTable、SSTable（Sorted String Table）。

不同组成部分对应 LSM 树的不同层次，层级间的数据转移通过 Compact 完成。下面介绍这些组件的实现方式。

### 3.1 MemTable

MemTable 是内存中的数据结构，用于保存最近更新的数据，并按 Key 有序组织。LSM 树并不强制具体的有序数据结构，可以使用红黑树、跳表等保证内存中 key 的有序性。

### 3.2 Immutable MemTable

为了在将内存数据持久化到磁盘时不阻塞新的写入操作，当 MemTable 达到一定大小后，会变为 Immutable MemTable 并加入队列。后台任务会从队列头部取出 Immutable MemTable 并将其持久化为磁盘上的 SSTable。

### 3.3 SSTable（Sorted String Table）

SSTable 是存放在磁盘上的有序键值对集合。典型的文件结构是将数据划分为数据块（类似数据库页），为数据块建立索引，索引项放在文件末尾，并使用布隆过滤器来优化查找性能。

## 4 LSM 树的 Compact 策略

当某一层数据量达到预设阈值时，需要通过 Compact 策略将其转换到下一层。设计 Compact 策略时，常考虑以下问题：

- 某一层是用单个文件还是多个文件实现？
- 如果用多个文件，同一层 SSTable 的 key 范围是有序（不重叠）还是重合（有重叠）？有序方便读，重合方便写。
- 每层 SSTable 的大小以及不同层之间文件大小是否相等？
- 每层 SSTable 的数量是多少？同一层 key 范围重合时，文件数量越多读效率越低。

不同选择会影响以下三类放大效应：

- 读放大：读取数据时实际读取的数据量大于目标数据量（可能需查多层或多个 SSTable）。
- 写放大：写入数据时实际写入的数据量大于目标数据量（Compact 可能重复写数据）。
- 空间放大：数据在磁盘上的实际占用大于原始数据大小（同一 key 在多个层或多个 SSTable 中重复）。

下面介绍两种基本策略：size-tiered（大小分层）和 leveled（层级化），它们在上述三种放大效应之间做不同权衡。

### 4.1 size-tiered 策略

#### 4.1.1 算法

size-tiered 策略使得每层 SSTable 的大小相近。当某一层的 SSTable 数量达到阈值 N 时，触发 Compact，合并这些 SSTable 并将合并后的结果写入到更大的 SSTable，放到下一层的队尾。因此，同一层不同 SSTable 的 key 范围会重叠，查找时需从后向前扫描同一层的 SSTable，最坏情况可能扫描该层所有 SSTable，从而增大读放大。

#### 4.1.2 总结

size-tiered 的特点包括：

- 每层 SSTable 的数量相近。
- 底层单个 SSTable 可能变得非常大。
- 同一层或不同层之间会有重复 key，空间放大严重；只有在该层执行 compact 时才能消除冗余。
- 读操作常需同时读取同一层的多个 SSTable，读放大严重。

### 4.2 leveled 策略

#### 4.2.1 算法

leveled 策略限制 SSTable 文件的大小，并使每一层中不同 SSTable 的 key 范围不重叠（后一个文件的最小 key 大于前一个文件的最大 key）。当某一层的总数据量达到阈值 N 时，触发 Compact。通常会选择一个 SSTable 与下一层中所有与其 key 范围重叠的 SSTable 进行合并。由于下一层保持全局有序，这种合并可能需要将当前 SSTable 与下一层中多个（甚至全部）SSTable 一并处理，导致写放大增加。

#### 4.2.2 总结

leveled 的特点包括：

- 不会产生非常大的单个 SSTable 文件。
- 每一层不同 SSTable 的 key 范围不重叠，相对于 size-tiered，读放大更小。
- Compact 时需要与下一层 SSTable 一起合并，写放大较严重。

## 5 插入、修改、删除

从名字 Log-Structured Merge-Tree 可见，LSM 树倾向于顺序追加而非在磁盘上就地修改。插入、修改、删除操作都在 L0（内存层）记录相应的条目并带有时间戳。查找时只需读取最新的记录，旧的历史记录会在后续的 Compact 过程中被覆盖或删除，因此无需即时在所有层级上进行修改。

## 6 查找

查找从 L0 层向下遍历，遇到某个 key 的最新记录即可返回，之前的记录无需继续查找。对于 size-tiered 策略，同一层的 SSTable 需从后向前遍历直到找到匹配项。查找过程中还会结合其他优化手段，如缓存（block cache）、布隆过滤器等，以减少磁盘 I/O。

## 7 LSM 树与 B+ 树的比较

- 写入（插入/修改/删除）：不考虑写日志等操作，B+ 树在插入/修改/删除时通常需先定位数据位置，可能涉及多次磁盘 I/O；LSM 树通过内存写入和顺序磁盘写入来减少单次磁盘 I/O，因此单次插入耗时短，写入吞吐量高于 B+ 树。但 LSM 的 Compact 会对数据进行多次写入，总写量大于 B+ 树；可通过将 Compact 安排在业务低峰期减轻影响。
- 读取：LSM 树可能需要遍历多层或多个 SSTable，查找效率低于 B+ 树。但由于 LSM 在写入时节省了磁盘随机写入资源，某些场景下可以通过缓存与布隆过滤器等手段弥补读效率差距。

## 8 总结

LSM 树的主要特点包括顺序写入、基于 Compact 的后台合并，以及读、写和空间放大效应。LSM 树适用于写操作吞吐量要求高、可以容忍一定读放大的场景，目前广泛应用于 NoSQL 系统以及作为分布式 KV 存储引擎的基础（例如 TiKV 使用 RocksDB）。
