# 技术 011 - 《My Philosophy on Alerting》——监控报警的哲学

译者：姚钢强

《My Philosophy on Alerting》是我认为关于监控和报警最好的论文，我斗胆尝试翻译成中文。因为水平有限，翻译肯定有纰漏。欢迎大家到原文处进行评论，一起改进翻译质量。

译者注：做任何事儿的时候，都要尽可能尝试总结一定的方法，从而逐渐形成知识体系。不要让知识是一个个孤零零的点。

译文：

审核和编写报警规则时，需要考虑以下这些原则：

- 报警的触达（电话、短信等）应当是紧急的、重要的、可行动的、真实的。
- 规则应当表示你的服务处于过程中或即将发生的问题。
- 为了保持报警项的精确、有效，宁可过度移除报警噪音。因为过度监控比监控不足更难解决。
- 你应该总是能够将问题分为以下几类：基本功能的可用性问题；延迟；正确性（数据的完整性、新鲜性和持久性）；以及特定功能问题。
- 规则描述症状是更好的方法，可以更轻松、更全面、更可靠地捕获更多的问题。
- 在基于症状的页面或仪表板中包含基于原因的信息，但要避免直接针对原因发出警报。
- 报警越向上层服务走，在一个报警规则中可以抓住的明显问题就越多。但不要走得太远，否则你无法充分区分究竟发生了什么。
- 如果你希望值班时报警系统保持安静，那么需要有一套系统和标准化的流程能够自动处理那些需要被尽快处理但不至于让你半夜三点起床上线的事件。

导论

在为各种不同的服务（包括大型和小型、快速发展的产品以及核心基础架构的多个部分）服务了七年之后，我已经形成了一套监控和报警的理念。它反映了我对监控和报警的基本看法。

每次收到实时报警（传呼机响起），我都应该能做出紧急反应。但是我每天只能这样做几次，否则会感到疲劳。每个报警项都应该是可行动的，“报警又来了”这种信息不能指导行动。每个报警项都应该需要人的智慧来处理，而不是机器人或脚本式的自动回应。

当我审核或者编写新的报警规则时，下面这些是我经常思考的问题：

- 它是否检测到了原本未被检测到的状况，这些状况是紧急的、可操作的、正在发生的或即将被用户看到的？请注意，零冗余的情况（本来是多活三个节点，现在只剩一个节点存活）也算迫在眉睫，如你的服务中“几乎已满且越来越满”的部分（如存储耗尽）。
- 是否发生过无视这条报警的情况，并且知道它是良性报警？什么时候、为什么？我能改进规则来避免这种情况吗？
- 它是否识别出肯定会或将会伤害用户的情况？是否有可检测的情况不会对用户造成伤害，应该将其过滤掉？例如具有测试流量的服务器集群之类的事情。
- 我可以针对这个警报采取什么行动？该行动是紧急的，还是可以等到我醒来后、周末或者下一季度结束后再做？
- 是否有其他人需要同时被通知到？他们会不会解决问题？或者我要替别人解决问题？我们能把这些事情联系起来吗？我的新添加的报警能不能等他们尝试去解决？

下面的这些想法当然有些理想化；在一个不断发展、不断变化的服务中，没有一个报警系统是完全干净的。但也有一些技巧可以让你更接近这个目标。

为用户进行监控

我称之为“基于症状的监测”，而不是“基于原因的监测”。你的用户关心你的 MySQL 是否宕机吗？不，他们关心的是他们的查询是否失败。也许你已经很依赖 MySQL 的监控规则，但你的用户甚至不知道你的 MySQL 服务器的存在！用户关心的是功能是否失效，关心数据推送是否失败，关心是否拿到最新的结果。

通常情况下用户只关心很少的一部分事情：

- 基本的可用性和正确性：没有 "Oops!"，没有 500，没有挂起的请求、半加载的页面；没有丢失的 JavaScript、CSS、图像、视频。任何以某种方式破坏核心服务的事情都应该被认为是不可用的。
- 延迟：系统响应要足够快。
- 完整性/新鲜度/持久性：用户数据应该是安全的，应该能在需要时返回，搜索索引应该是最新的。即使暂时不可用，用户也应该完全相信它会恢复。
- 功能：用户关心服务的所有重要功能是否正常工作——你应该监控对你的服务很重要的任何方面，即使它不是核心功能/可用性（例如，计算器和股票行情显示在搜索结果中）。

数据库服务器不可用和用户数据不可用之间有一个微妙但重要的区别。前者是一个可能的原因，后者是一个症状。你不可能总是干净利落地区分这些事情，特别是当你没有办法模拟用户视角时。但如果你能做到模拟，你就应该去做。

基于原因的报警是糟糕的（有时是必要的）

你可能会说，“我知道数据库服务器无法访问会导致用户数据不可用。” 这是有价值的思路。警惕数据不可用的问题。在出现以下症状时应发出警报：500、Oops！但是一些白盒指标表明并不是所有的服务都直接从数据库获取数据然后返回客户端。基于原因报警为什么不够好？

- 无论如何你都要抓住症状。症状可能由网络断开、CPU 征用或其他无数你还没想到的问题引起。所以你必须抓住症状，而不是原因。
- 同时关注症状和原因会产生冗余的报警；这些警报需要单独调整和管理，并导致症状和原因报警之间重复或复杂的依赖关系树。
- 所谓不可避免的结果并不总是不可避免的：也许数据库服务器不可用了，因为你正在开启新实例或关闭旧实例；或者你添加了请求的快速 failover，因此单个服务器的可用性不再那么重要。当然，可以用越来越复杂的规则来捕捉所有这些细节，但这样只会导致更多的虚假报警、更多的混乱、更多的调整，而没有收获，反而减少了你处理重要警报的时间。

但有时基于原因的报警是必要的。对于“几乎”用完配额的情况（如内存、磁盘即将用满），通常没有明显症状，所以你需要这些报警来知道系统即将出现问题。但应尽量少用这些，一定不要为你可以捕捉到的症状写基于原因的报警规则。

从出口（或更远处）发出警报

在 C/S 系统中，最好的报警来自客户端。客户端可以看到重试的结果、客户端与服务器之间的网络延迟，并且可以比服务端更好地知晓面向用户的延迟和错误。

在许多情况下，客户端（例如浏览器或应用服务器）聚合来自许多后端的响应，例如缓存服务、数据库、账户管理/授权服务、查询分片等。当你在做基础设施变更时，如果使用客户端指标做监控，这些监控规则会更加健壮，不用频繁更改。

客户端往往能比后端呈现出更简单的整体视图。例如，如果一个请求打到数百个查询服务器上，那么从每个查询服务器的视角来看都太有限，无法成为有用的报警来源。对于很多服务来说，这意味着要对最前端的负载均衡器所看到的延迟、错误等情况进行报警。这样如果 server 宕机了，只有在它们真正影响用户时你才会看到报警。但从负载均衡器看到的问题覆盖面更广：如果多台 server 宕机、出现大量 500 或 10% 的连接断开，负载均衡器知道，但单个服务器可能不知道。

不过请注意，使用特别外层的监控可能会引入你无法控制、也不负责的代理。如果你能可靠地捕捉到用户到底看到了什么（例如通过浏览器端工具），那就太好了！但请记住，这类信息充满噪音（ISP、浏览器、客户端负载和性能）；因此，最外层的监控不应该是唯一方法。如果外部监控不能总是与你保持畅通，它可能是有损的。即便如此，这仍然是有用的信息，但可能并不是你唯一想要依赖的。

原因仍然有用

基于原因的规则仍然有用，特别是它们可以帮助你快速定位生产系统中的已知缺陷。如果你在自动将症状与原因联系起来的过程中获得了很多价值，也许是因为有一些原因是你无法控制或无法消除的。我提倡这样做：

- 当你写下（或发现）一个代表原因的规则时，检查相关症状的规则是否也在报警系统中。如果没有，就将症状规则添加进系统。
- 在报警信息里写一个简短的摘要，列出所有基于原因的规则。一个人快速浏览一下，就可以确定他们刚刚收到的症状是否有已知原因。例如：

  TooMany500StatusCodes  
  Served 10.7% 5xx results in the last 3 minutes!  
  Also firing:  
  JanitorProcessNotKeepingUp  
  UserDatabaseShardDown  
  FreshnessIndexBehind

  在这种情况下，很明显最有可能的 500 来源是数据库问题。相反，如果引发的症状是磁盘已满，或者结果页面返回为空或数据陈旧，就可能是其他原因。

删除有噪声、持续的、低价值的基于原因的报警规则。使用上述方法，那些不协调、嘈杂的规则所带来的精神负担会从传呼机的哔哔声（以及调查、跟进）变成一行要浏览的文本。由于你无论如何都需要清晰的调试仪表板（用于非报警开始的问题），这是公开基于原因规则的好地方。

也就是说，如果你的调试仪表板使你能够从症状足够快地定位到原因，那么你根本不需要花太多时间在基于原因的规则上。

工单、报告和电子邮件

通常有一些报警需要尽快注意，但不需要立即处理。我称之为“次关键报警”。工单跟踪系统会很有用。只要能将同一警报的多次触发正确地放入单个故障单/bug 中，让报警系统在工单系统建立一个 bug 类型的工单，这样用户跟踪效果就会很好。

如果没有后续负责分类和关闭 bug 工单的责任，此系统将失败。如果报警系统打开的 bug 工单在数周内不被处理，那么这显然不能作为在次关键报警变得严重之前处理它们的方式。如果你的团队只是超负荷工作或没有分配足够人员来跟进，此系统也会失败；你需要诚实地知道这需要花费多少时间，否则会越来越落后。

每日（或更频繁）的报表也是可行的。一种可行的方法是编写长期存在的次关键规则（例如“数据库容量已超过 90%”或“过去一天处理了 1000 个非常慢的请求”），然后定期发出报表，显示所有当前触发的规则。但是同样，如果没有问责制，这意味着电子邮件警报会更加成为垃圾邮件，因此请确保指定值班人员（或其他人）每天或每次交班时处理这些事项。

每个报警都应通过工作流系统进行跟踪。不要只将它们转存到电子邮件列表或 IRC 频道。通常这很快就会变成专门的垃圾邮件列表或渠道，从而被忽略。除了需要短期（通常几天，最多几周）审核新报警规会不会频繁触发外，这么做几乎总是一个坏主意。也容易突然间成千上万的应用服务器每分钟触发一些旧的、调整不当的规则，导致邮箱被打爆；这种情况十分糟糕。

根本的一点是要建立一个系统和标准化流程，这仍然需要有响应的问责制，但不会高成本地叫醒某人、打断他们的晚餐或阻止其与孩子的拥抱。

操作手册

操作手册是报警系统的重要组成部分；对于捕捉到的基于症状的每个报警，最好有一个条目可以进一步解释警报的含义以及如何处理警报。

通常，如果你的操作手册有很长的详细流程图，就有可能花较多时间去记录可能出现的问题，而花很少的时间去修复它，除非根本原因完全不在你的控制范围内，或者根本上不需要人工干预（例如致电供应商）。我见过的最好的操作手册会对警报的含义做一些说明，以及有关警报应该注意的要点（例如“我们从 VendorX 的小部件中发现了一系列断电现象；如果你发现了这一点，请添加到错误 12345，我们将跟踪处理”）。大多数类似说明应该是短期的、非永久性的，因此 Wiki 或类似的产品是很好的工具。

跟踪和问责

追踪所有收到的报警。如果收到了报警，而人们只是说“我看了，没有问题”，这是一个强烈的信号：你需要删除报警规则、降级，或以其他方式收集数据。准确率低于 50% 的报警是不可接受的；即使是 10% 的假阳性也值得考虑是否对规则进行修改。

拥有一个适当的系统（例如每周对所有页面进行检查、每季度进行一次统计）可以帮助掌握正在发生的事情，并弄清将报警从一个人转移到另一个人时出现的问题。

你太天真了！

尽管我认为上述使用报警的原则非常好，但在以下情况中可以打破这些原则：

- 在基于症状的报警噪音基础上，有明确导致症状的原因。例如，如果你的服务有 99.99% 的可用性，但有一个导致 0.001% 请求失败的事件，则不能将其作为症状报警（因为它位于基础症状报警噪音中），但可以捕获导致症状的原因。也许值得在栈中传递这些信息，但也许最简单的做法就是基于原因发出报警。
- 你无法监控出口的数据，因为数据精度丢失。例如，可以容忍某些 handlers/endpoints/backends/URLs 响应慢或可用性低（如信用卡验证与浏览商品相比），在负载均衡上难以将这些特殊接口区分出来。需要从链路进行追踪，从能发现数据区别的地方发出报警。
- 症状要到很晚才会出现，比如你的配额用完了。你确实需要提前报警，有时这意味着要找到一个基于原因的报警（比如使用率 > 80%，按最近 1 小时的增长速度将在 <4 小时内用完）。但如果能做到这一点，你也应该能找到不那么紧急的类似原因（例如配额 > 90%，按最近 1 天的增长速度将在 <4 天内用完），可以通过工单、邮件提醒或每日问题报告的方式处理，而不是等到报警时已经面临最紧急的状况。
- 设置的报警规则比它们要检测的问题更复杂。有时候报警确实会这样。我们的目标应趋向于简单、健壮、自我保护的系统（例如：为什么你没注意到自己用完了配额？为什么这些数据不能转移到其他地方？）。从长远来看，系统应趋向于简化。但为了保持报警的精确性，也不要过于执着于局部最优，导致规则相对复杂。
