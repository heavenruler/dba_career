# 不同数据库的存算分离有何不同

作者：胖头鱼的鱼缸（尹海文）  
Oracle ACE Pro: Database，PostgreSQL ACE Partner，10 年数据库行业经验。拥有 OCM 11g/12c/19c、MySQL 8.0 OCP、Exadata、CDP 等认证。

数据库的存算分离架构是一个经久不衰的架构，其中最为出名的当属 Oracle RAC 集群。但是在不同数据库中，存算分离的实现存在差异。总体上，不同数据库或不同场景对“存算分离”概念的侧重点也不同：

- 对于分布式数据库，更侧重将计算功能与存储引擎分离——二者都利用各自节点的本机资源。
- 对于集中式数据库或从硬件角度看，常把数据存放在共享存储中，计算放在本机，数据库通过将共享存储映射为本机资源来使用。

下面根据不同数据库和场景的特点做简要解析。

## 1 基于共享存储

在 Oracle RAC 集群中，GRID 组件的 ASM（Automatic Storage Management，自动存储管理）将来自共享存储映射的磁盘以不同的冗余模式组织为磁盘组，并在磁盘组内为数据库各节点提供统一的存储能力。ASM 除了提供冗余（即数据副本数），还会对磁盘组内的磁盘进行条带化以提升性能，从而最大化利用共享存储的吞吐与并发能力。这时 Oracle RAC 的性能上限往往依赖于共享存储的性能。

国产数据库中类似的架构还有金仓数据库的 KES RAC、达梦数据库的 DSC、崖山数据库的 YAC、以及 PolarDB 的存算分离架构等。

基于共享存储的存算分离架构有诸多优点：

- 可以选择集中式或分布式共享存储，根据需求灵活选型。
- 节点对数据的访问不一定通过节点间网络交互，可使用专用且低延迟的存储网络实现更稳定快速的数据访问。
- 节点异常后，整个集群的重配置时间较短；故障判断可以依赖存储链路，节点恢复不需要占用大量网络带宽进行数据同步；集群节点的增减对系统影响可做到近乎无感知。

缺点与风险包括：

- 数据库的安全性与可用性高度依赖共享存储。一旦共享存储发生故障（存储宕机、链路闪断、静默错误等），可能对数据库造成灾难性影响。
- 需要选择可靠的共享存储并投入相应成本，特别是在企业级场景下，推荐使用性能与可靠性都较好的存储设备。

## 2 基于存储引擎

很多数据库有“存储引擎”的概念，不同的存储引擎具有不同特性。以 MySQL 为例，常见有 MyISAM、InnoDB 等主流引擎，还有相对小众但用于分布式场景的 NDB Cluster。NDB Cluster 本身就是一种存算分离架构：mysqld 提供数据库访问和计算能力，ndb_mgm 提供集群管理及元数据，ndbd 提供分布式存储能力。

类似的国产/开源架构还有 TiDB：TiDB Server 提供计算能力，PD Server 存储元数据，TiKV/TiFlash（Storage）提供分布式存储能力。

与存算一体的分布式架构相比，这类基于存储引擎的存算分离架构在设计上不需要业务方关心数据在各节点上的具体分布（即透明的数据分片），直接使用服务器本机资源，节点之间物理上没有共享存储。在网络健康且延迟可控的情况下，理论上可以实现大规模的横向扩展。但这也带来一些挑战：

- 集群的健壮性与数据安全高度依赖数据库本身的高可用特性（副本复制、容错机制等）。
- 节点间交互严重依赖网络，而网络具有不确定性（基于 CAP 原则可能需要在一致性、可用性、分区容忍性之间权衡）。
- 任何节点异常都需要通过网络进行故障判断与恢复；恢复后也需要占用网络带宽进行数据同步。

## 3 混合架构

从某个版本开始（例如 OceanBase 从 4.3.4），同时支持无共享（Shared-Nothing, SN）和共享存储（Shared-Storage, SS）模式部署。OceanBase 的 SS 模式是基于通用对象存储实现的共享存储存算分离架构，利用云原生基础设施，可以在一定程度上降低使用成本。

在这种混合架构中，每个租户在共享对象存储上保存一份数据与日志，并在节点本地存储上缓存热数据和日志。每个主副本负责将一份全量基线数据上传到对象存储，副本间共享对象存储上的基线数据；所有副本自动识别数据热度，仅在本地缓存热点数据。租户的每个副本独立转储，转储数据不在副本间共享。

与纯共享存储模式相比，OceanBase 的 SS 模式限定使用对象存储，本机仍存储日志和部分数据，架构更复杂。多数对象存储的性能通常不如本地高速缓存，因此数据库的性能依赖本机内存和热数据缓存。但这种架构为不同场景提供了更多选择，在云环境下具有成本与运维上的优势。

## 4 优点（存算分离带来的好处）

相较于单纯依赖本机硬件资源的存算一体数据库架构（无论是集中式还是分布式），存算分离架构解决了以下问题：

- 存储扩展更便捷：单机 CPU 和内存性能提升明显，但可承载的高性能磁盘容量有限。通过各种存算分离架构可以更方便地扩展存储容量与性能。
- 降低数据拆分负担：在分布式架构下，存算分离可以减少甚至消除业务层面对数据拆分（分片）的关注与复杂性。
- 计算与存储解耦：计算节点故障不影响数据访问，数据存储依托自身高可用机制可以确保数据安全性。

使用专用存储设备时，还能带来进一步优势：

- 更稳定高效的存储链路（例如专用存储网络、端到端优化）。
- 磁盘高可用性可由存储端完全托管。
- 更稳定和高效的 I/O 性能表现。
- 更全面的磁盘性能与隐患监控，能够实时发现并处理性能衰退与故障。
- 对高性能磁盘（尤其 NVMe SSD），专用存储已较好地解决了在大规模环境下的数据保护与性能管理问题。

专用存储可以服务于多套数据库集群，且在大量节点的数据库集群场景中，从全局角度看对磁盘的利用更节省，并简化本机磁盘管理。虽然成本较高，但更适合企业级应用场景。

## 5 展望

此前也讨论过一种“逻辑上存算一体但物理上存算分离”的异形分布式架构：每个节点利用本机 CPU 和内存，但不使用本机磁盘，而是使用共享存储的映射磁盘。在这种情况下，如果存储与数据库能够联动，通过存储端直接对分布式数据库进行全局一致性快照或备份，可以极大提升分布式数据库的备份与恢复效率。

无论是专用存储（集中或分布式），还是数据库自带的基于存储引擎的分布式存储集群，通常都配备了大量未被充分利用的 CPU 与内存。在传统使用方式下，这部分计算资源大多被闲置。如果能够充分利用这些资源来持续优化存储结构、优化数据分布、自动冷热分层、在存储端提前筛选或预处理数据等，将显著提升数据库整体性能并简化大量优化与维护工作。结合 RDMA 等高性能低延迟网络，可以进一步提升存储链路带宽并降低延迟，同时扩展存储在计算中能承担的任务。

## 总结

本文总结了主流数据库和不同场景下的存算分离架构，比较了基于共享存储、基于存储引擎以及混合架构的特点与优缺点，并对未来的发展方向进行了展望。存算分离为解决存储扩展、简化数据拆分与提高可用性提供了多种路径，具体选型应根据业务场景、性能与成本权衡而定。
