# 探索 Redis 与 MySQL 的双写问题

作者：Java随想录

## 简介

在实际应用 Redis 和 MySQL 的过程中，如何保证 Redis 和 MySQL 双写时的数据一致性，成为开发者面临的重要挑战。Redis 提供极快的读写速度，适合作为内存缓存；MySQL 是关系型数据库，支持事务处理并提供较强的数据一致性。两者结合可以提升系统性能，但也带来了双写不一致的问题。本文将深入探讨这些问题并给出常见的解决思路和方案。

## 双写一致性问题

双写一致性问题主要是指当我们同时向 Redis 和 MySQL 写数据时，由于网络延迟、服务器故障等原因，可能导致数据在两个系统之间产生不一致。例如，你可能已经更新了 MySQL 中的数据，但 Redis 中的数据还未来得及更新；或者反过来。这样的结果就可能导致用户读到旧的或不正确的数据。

举例：在购物网站场景中，用户 A 购买一件库存仅剩 1 件的商品，系统接收到请求后先将 MySQL 中的库存减少 1，但因网络或系统故障，Redis 中的库存没有减少。此时用户 B 看到缓存仍有 1 件也发起了购买请求，如果系统又先更新了 MySQL，就会导致超卖。

严格意义上任何非原子操作都无法保证强一致性（除非采用阻塞读写等强一致性手段），因此缓存架构通常追求最终一致性。缓存是通过牺牲强一致性来换取性能，这与 CAP 理论一致，缓存系统适用于对强一致性要求不高的场景（属于 CAP 的 AP 侧）。

## 缓存读写策略

常见的缓存读写策略用于处理“先更新数据库还是先更新缓存”的问题。下面介绍三种常见策略：旁路缓存（Cache-Aside）、读写穿透/写穿透（Read/Write Through）和异步写入（Write Behind）。各有优缺点，应根据具体场景选择。

### Cache-Aside Pattern（旁路缓存模式）

旁路缓存模式要求服务端同时维护 DB 和 Cache，并以 DB 为准。

- 读：先从缓存读取，命中则直接返回；若未命中，则从数据库加载、写入缓存后返回。
- 写：更新时先更新数据库，然后再删除缓存（即先写 DB，后删缓存）。

这种模式是最常用的缓存使用模式，客户端与缓存、数据库之间的交互清晰，适用于多数读多写少的场景。

### Read/Write Through Pattern（读/写穿透模式）

读/写穿透模式将 Cache 视为主要的数据入口，Cache 服务负责在未命中时从 DB 加载数据并写回 DB，从而对应用透明化。

- 读：从 Cache 读取，未命中则由 Cache 服务从 DB 加载并写入 Cache，然后返回。
- 写：写操作首先写入 Cache，Cache 服务再将数据写入 DB（同步更新 Cache 和 DB）。

由于常用的分布式缓存如 Redis 并不内建将数据写入 DB 的功能，该模式在实际应用中较少直接使用，更多用于有专门缓存服务的架构中。

### Write Behind Pattern（异步缓存写入/延迟写入模式）

Write Behind 与 Read/Write Through 相似，均由 Cache 服务负责 Cache 与 DB 的读写，但不同点在于：

- Read/Write Through 是同步更新 Cache 与 DB；
- Write Behind 只更新 Cache，不立即更新 DB，而是异步批量地将缓存变更写入 DB。

这种方式能显著提高写入吞吐量，适用于对最终一致性要求不高且写入频繁的场景（如浏览量、点赞数统计）。但它对数据一致性带来更大挑战：若在异步写入到 DB 前 Cache 服务挂掉，可能导致数据丢失或不一致。

## 旁路缓存模式解析

Cache-Aside（旁路缓存）是最常用的模式，但在实践中会有一些疑问和陷阱，下面一一说明。

### 为什么写操作是删除缓存，而不是更新缓存？

如果在写操作时直接更新缓存，可能出现并发覆盖导致脏数据的问题。例如：

- 线程 A 发起写操作并先更新 DB；
- 线程 B 也发起写操作并更新 DB；
- 因网络等原因，线程 B 比线程 A 先更新了缓存，然后线程 A 更新缓存；
- 缓存最终保存的是线程 A 的旧数据，而 DB 是线程 B 的新数据，造成不一致。

因此通常采用“先更新 DB，再删除缓存”的策略来避免这种脏数据问题。当然，也可以选择更新缓存，但需要加锁或分布式锁来保证并发安全。

### 在写数据过程中，为什么要先更新 DB 再删除缓存？

如果先删除缓存再更新 DB，可能出现下面的问题：

- 请求 1（写操作）先删除了缓存；
- 紧接着请求 2（读操作）来读，发现缓存被删，于是去读 DB；
- 若请求 1 尚未完成 DB 更新，请求 2 会读到旧数据并把旧数据写入缓存，造成数据不一致。

因此通常推荐先更新 DB，再删除缓存（或采用延时双删等策略以兼顾情况）。

### 先更新 DB、后删除缓存会不会仍然不安全？

理论上仍存在概率性不一致的情况，典型场景如下：

- 缓存刚好失效；
- 请求 A（读）查询 DB，得到旧值；
- 请求 B（写）将新值写入 DB；
- 请求 B 删除缓存；
- 请求 A 将旧值写入缓存。

若发生上述情况，会出现脏数据。但这种情况发生的前提是写操作比读操作在时间上更快（写先完成并删除缓存），在绝大多数场景下数据库读速度通常比写速度快（因此该情形相对少见）。不过在高并发和复杂网络延迟环境下，仍需谨慎处理。

### 其他导致不一致的原因

如果删除缓存过程中失败，也会造成不一致。可以使用数据库 binlog 订阅工具（如 Canal）来监听 DB 的变更，然后由独立程序去删除缓存，作为补偿手段。

## Cache-Aside 的缺陷和缓解办法

缺陷与应对方法：

- 缺陷 1：首次请求数据一定不在 Cache（冷启动问题）。
  - 解决：提前将热点数据预热到 Cache 中。
- 缺陷 2：写操作频繁会导致缓存被频繁删除，影响缓存命中率。
  - 解决：对于要求强一致性的场景，在更新 DB 的同时也更新 Cache，但需要加锁或使用分布式锁保证更新时的线程安全。
  - 对于短暂允许不一致的场景，可以更新 DB 后同时更新 Cache 并设置较短的过期时间，降低不一致带来的影响。

## 延时双删（Delayed Double Delete）

延时双删策略是一种常用的工程实践，基本步骤如下：

1. 先删除缓存；
2. 更新数据库；
3. 延时一段时间后再次删除缓存。

理念是：如果其他线程在步骤 1 与步骤 2 之间读取到旧数据并写回了缓存，步骤 3 可以再次清理这些旧数据，从而尽可能维持 DB 与 Cache 的一致性。

下面是一个 Java 实现示例（使用 Jedis）：

```java
import redis.clients.jedis.Jedis;

public class RedisDoubleDelStrategy {
    private Jedis jedis;
    private static final long DELAY_MILLIS = 1000L; // 设置为你需要的延时时间

    public RedisDoubleDelStrategy(String host, int port) {
        this.jedis = new Jedis(host, port);
    }

    public void updateDBAndCache(String key, String value) {
        // Step 1: 删除缓存
        jedis.del(key);

        // Step 2: 更新数据库，此处以打印输出代替真实 DB 操作
        System.out.println("Update DB with: " + value);

        // 延迟任务来完成第二次删除
        new Thread(() -> {
            try {
                Thread.sleep(DELAY_MILLIS);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            // Step 3: 延时后再次删除缓存
            jedis.del(key);
        }).start();
    }
}
```

注意：延时双删仍不能完全保证一致性。在大量并发情况下，仍可能出现不一致的问题。需要结合系统的特性和一致性需求做权衡，例如合理选择延时长度、结合消息队列或 binlog 订阅做补偿。
