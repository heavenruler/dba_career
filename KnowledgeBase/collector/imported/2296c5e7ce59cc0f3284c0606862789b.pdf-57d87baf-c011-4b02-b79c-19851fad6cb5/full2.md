# 广告系统的平台架构与交互流程

发布日期：2020-01-17

商业化是互联网产品常用的流量变现手段，商业化平台（广告平台）的核心是围绕客户（商家、广告主等）、用户（网民、产品使用者）和平台三者展开的商业活动。业界较大的广告平台包括百度凤巢、腾讯广点通、阿里妈妈、今日头条巨量引擎等。

这些广告平台面向的业务领域大体相同，都属于计算广告学范畴。无论是展示广告、效果广告、信息流广告，还是 CPT、CPM、CPC 模式；无论是竞价（RTB）还是合约广告，系统架构上都有很多共性。本文尝试抽象一种通用的广告系统架构，从工程角度阐述其模块构成，并从模块交互流转角度介绍主要业务流程。

下文提到的每一个模块都可以展开为一个庞大的工程或算法问题，甚至需要几十到上百人的团队来搭建与运维。本文主旨在抽象和简化通用的架构组成。

## 总体架构（四个大模块/端）

一个广告系统内部由四个大模块（端）组成：

1. 业务端（平台端，to B）  
   负责将营销参与者融入商业体系，管理“生产资料”，是广告平台的后勤部。例如广告主设置投放条件、提交创意；运营分析优化广告主投放设置；与代理商进行激励返点、财务计费结算等。

2. 检索端（to C）  
   面向网民，将流量商业化。要求在极短时间内把最合适的广告出给最合适的网民，是一个在线检索系统，用于实时召回、排序和返回广告结果。

3. 数据端  
   收集展现、点击、转化等数据，进行计算、转换和加工，形成数仓和报表供业务端与 BI 使用。同时整合各方数据，提供统一的数据平台（如 DMP、标签系统等）。

4. 策略端（算法端）  
   基于机器学习技术构建模型，从机制、定向、触发角度与检索端交互，为在线广告提供智能决策能力，更好平衡用户、平台和广告主的诉求与利益，包括但不限于 CTR、CVR 预估、风控、反作弊等。

下面展开各模块的流程说明。

## 流程阐述（主要交互流程）

1. 注册/登录  
   广告主登录系统进行鉴权、角色分配和账户管理。同时为产品、运营、销售等营销参与者提供登录和代理功能。

2. 代理开户与订单合同管理  
   代理商可代广告主开户，客户统一维护在用户中心；对于大客户（KA）通常投放 CPT、GD 等合约广告，需要订单管理、询量询价、合同确认等。

3. 资质审核  
   风控平台审核广告主资质，进行打标和风险控制。

4. 用户充值  
   广告主通过财务平台充值，只有账户余额 balance > 0 时，才具备广告投放资格。

5. 设置广告  
   通过客户端、API 或投放平台系统进行广告投放设置，包括预算、排期、推广计划、单元、关键词、定向、创意等。广告主通过此步骤表达投放诉求。现代系统常通过 DMP 提供“圈人”操作实现人群定向，投放平台一般是业务系统，常用服务化方式构建。

6. 存储投放设置  
   投放平台作为复杂状态机，其状态数据保存在广告库中，通常使用结构化 OLTP 存储引擎（如分库分表的 MySQL/RDS，或分布式数据库 PolarDB、OceanBase、TiDB 等），保存全局广告主的投放诉求。

7. 制作创意、建站与产品库管理  
   创意是传达推广意图的载体，通常有专门的创意制作工具或公司。对于中小广告主（SME），平台可提供建站能力；对于大型广告主（KA），可通过产品库方式进行创意制作（例如百度闪投），以节约成本并实现千人千面投放。越来越多平台采用创意组件化，根据素材动态拼接组件，提高个性化投放效果和点击率。

8. 存储物料  
   物料（图片、视频等非结构化数据）需专门的存储引擎（如 OSS）并同步到 CDN，用于客户端在线展示。

9. 审核广告物料  
   风控审核平台对增量物料进行机审和人审，保证符合平台规范，避免风险物料展示，并对创意进行分类打标。

10. 广告传输  
   广告库需要将数据下发到检索端的广告索引中才能在线生效。可以全量导出（使用 Spark/MaxCompute 等分布式计算引擎生成基准文件），也可以实时增量下发（通过 binlog、canal、fountain、hiriver、DTS 等），下发到消息队列实现多端解耦与广播订阅。

11. 构建广告索引（离线/实时）  
   检索端订阅广告传输流以及计划撞线、广告主状态的增量更新，并将信息保存或更新到索引库。广告索引规模通常很大，通过索引服务暴露查询接口（类 SQL），根据查询条件查找符合条件的广告。与传统搜索引擎不同，这里的倒排索引以 Ad 为单位（而非网页），查询主键可以是关键词或多个定向维度。

12. 请求广告 / 展现广告  
   用户在客户端访问产品，产品发起广告请求。访问端也可能是 ADX（符合 OpenRTB 的 bid request），广告平台可以接入其他 ADX 进行竞价。

13. 发起检索  
   请求通过 UI 服务器（7 层负载或封装服务）向内网检索服务发起请求。

14. 查询特征（特征提取）  
   在 session 中提取特征：通过某些 id 或 id-mapping 唯一定位访问者，获取 query、访问上下文、用户信息（IP、IDFA、安卓 ID、OAID 等）、用户标签（兴趣、社会属性）、历史行为等。这些特征与用户数据供后续环节使用，通常存储在分布式 KV 存储中。

15. 召回备选广告  
   输入请求特征，输出符合条件的 N 条广告或创意。召回通常访问广告索引（倒排索引，feature -> unit/creative），可包含多个通道（关键词触发、兴趣定向触发等）。召回广告经过业务过滤并只包含基础信息，然后构造返回队列并截断到阈值内，属于粗排海选。搜索广告的关键词匹配也可在此环节发生（短语匹配、宽泛匹配等）。

16. 预估 CTR/CVR（精排前的预估）  
   对粗排后的广告进行 CTR/CVR 等预估以便精排。输入 <query, List<Ad>>，输出 <query, List<pCXR>>。预估往往是并行且计算密集型的操作，依赖于 modeling 模块或服务。一次查询涉及数百次的特征提取和查表，表通常是大的 hashmap（策略端负责更新模型文件）。模型文件规模可能很大（数百 GB），需要全量加载到内存以保证低延时访问。

17. 过滤 / 排序 / 计费并返回结果  
   进行匀速消费（pacing）与超预算控制、对媒体和用户的频次控制等过滤操作。对精选的广告进行排序（通常按 eCPM 排序），同时查正排（如标题、描述、URL），拼接用于展示的数据并拼接加密后的点击串，落展现日志，并通过 GSP 机制进行计费。

18. 样式渲染  
   检索引擎返回的创意通常为 raw 的结构化信息，需要结合前端模板进行渲染和组装后返回给前端。

19. 展现日志  
   对于搜索广告，可以近似认为该条广告已展现并将展现日志 push 到消息队列或日志系统；对于推荐广告，还涉及填充与真实曝光率问题，实际用户看到的广告远小于返回的数量。

20. 点击广告  
   用户对感兴趣的广告进行点击。

21. 跳转到落地页  
   点击后触发跳转到落地页、拨打电话等操作。

22. 转化日志  
   如发生真正的转化（如购买、注册等），转化事件会被记录并 push 到消息队列或日志系统。

23. 反作弊过滤  
   对展现、点击和转化数据进行反作弊处理，保护广告主和平台利益，过滤无效展现、无效点击（如站长刷量、恶意点击）以及各种不符合统计学规律的异常数据。

24. 扣费（计费）  
   通过计费服务进行扣费，计费服务需满足高性能、高吞吐、高可用和可扩展性，实时处理展现与点击事件并在指定推广层级进行扣费。计费通常涉及流式计算（例如使用 Flink），需要保证不丢不重、端到端的 exactly-once 语义。扣费结果最终反馈到财务中心，扣除广告主余额。

25. 计划撞线 / 广告下线  
   对于预算撞线或广告主余额不足的情况，需要及时下线广告以避免超预算损失。此类信息需实时反馈到广告索引中；出于收入敏感性考虑，可与广告库分离单独下发，通过隔离保证下发实时性。

26. 报表数据计算  
   离线/实时数据处理：将收集的展/点/消数据通过批量或实时计算引擎抽取、转换、拼接与过滤，输出到数仓（warehouse）并供广告主报表平台使用。数仓会面向广告领域建模，包含各种事实表与维度表供业务、算法与 BI 查询使用。面对广告主的高并发海量查询，常采用 MOLAP 选型（如 PALO、Kylin、Druid 等）。

27. 查看报表  
   广告主通过业务平台查询报表（例如最近 7 天的推广计划展/点/消数据）。

28. 查询数据（OLAP 引擎）  
   报表和 BI 查询通过 OLAP 引擎（Presto、Impala 等）或 MOLAP 系统进行。

29. 代理与客户管理（CRM）  
   CRM 面向销售、代理商和客服，典型功能包括线索管理、合同管理、开户、客户管理、业绩管理、返点等，通过运营手段为广告平台赋能。

30. 分析与优化产品  
   利用数据仓库和实时化技术形成基础事实与维度表，通常构建于 HDFS，上使用 ORC、Parquet、CarbonData 等格式，使用 Hive on MR、Spark 做计算，ROLAP 类查询使用 Presto/Impala 等。开发面向数据的服务供 BI、Dashboard 与算法工程师使用，用于系统监控、实验对比、消费趋势分析、决策支持与预警。

31. 策略与数据闭环  
   算法数据驱动业务流程，包括策略端通过机器学习建模后的模型数据供 modeling 使用；通过实时计算进行反作弊与实时反馈到特征库；DMP 引入多方数据帮助广告主更好锁定目标人群，做用户画像和标签供在线检索理解 session 使用。

## 注

1. 本文主要阐述工程架构，对于策略端的算法可分为机制、触发（搜索广告）、定向（推荐广告）与模型等方向，限于篇幅暂不展开。
2. 样式平台是一套统一业务、检索与算法使用的标准化服务：业务端通过标准样式供广告主编辑创意，检索端渲染会使用样式，算法可以基于样式做优选。
3. 算法驱动业务流程，主要包括策略端建模后的模型数据供 modeling 使用；实时计算用于反作弊与实时反馈服务到特征库；DMP 通过多方数据帮助广告主更精确地锁定目标人群，构建用户画像与标签，用于在线检索的 session 理解。

总结：本文尝试抽象一种通用的广告系统架构，从工程角度阐述模块构成，并从模块交互流转角度介绍主要业务流程。希望通过流程的描述，读者能对广告系统的内部工作方式建立宏观认识，为进一步深入学习提供帮助。
