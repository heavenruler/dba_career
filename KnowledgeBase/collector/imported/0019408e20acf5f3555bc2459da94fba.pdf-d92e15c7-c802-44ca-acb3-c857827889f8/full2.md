# DBA不仅仅是管理数据库——也要管理中间件

作者：薛晓刚  
日期：2024-02-05

看到这个标题可能觉得我在乱说，其实很多中间件作为数据库的上下游，它们的稳定取决于数据库的稳定。这里我主要指的是 SQL。

很早之前我就说，开发的“主题语言”其实更多的是 SQL（并不是说 Java 不重要），只是 Java 中并没有太多的业务逻辑。真正的业务逻辑主要还是在 SQL 中体现。说白了，就是如何写数据、如何读数据。

我现在工作不太做数据库安装、备份等，更多的是关注开发中的问题。用工具打开开发代码后，很多项目超过一半是 SQL，甚至这个比例能达到 70–80%。所以这更加证明了我之前的观点。

每当说到这里，非数据库出身的开发会质疑我夸大数据库的作用。因为我也见过有人把数据全部拿走，然后在程序中做各种运算，这样看起来与数据库关系不大。但在一次次数据库问题甚至故障后，越来越多的开发人员意识到我说的是对的。我在一些顾问和问诊过程中发现，其实 SQL 的影响不仅仅限于数据库。多次中间件（这里指 Tomcat 的 JVM、消息队列等）的问题（OOM 或频繁 Full GC）也是由于 SQL 导致的。因为一次取的数据太多，数据库把几十万甚至上百万的明细数据送到相关环节，这些环节一时处理不了，就发生了 OOM 或频繁 Full GC。

问题可能是一次性取了很多数据，也可能是数据本身对象很大。总之都是数据占据空间太大，导致了中间件层出现问题。

上周一个开发主动找我，他们的 JVM 中大约 80% 的问题都是 Full GC，而这些最终都发现是由 SQL 导致的。他们因此来咨询解决方案。这说明越来越多的开发人员开始觉醒了，这是件好事。

我想起从前在一个单位，曾对一个开发说他写法不对，会有问题。大家可能能猜到当时谁理我？但后来有一天，他来向我请教，问的还是比较深刻的问题。之后遇到问题也会来问我。我很好奇怎么会有这样的改变。后来才知道他去一些比较好的公司面试，技术面的问题就是我平时说的。因为不会，他错过了机会。回来后发愤学习，后来果然进了更好的企业并换过公司，收入也增加不少。

我第一家公司不大，但当时管研发的都是 70 后，招聘看的是计算机知识、算法、数据库，最后看编程。那时招聘的开发人员素质很高，可能几个人就做一个大项目。现在的招聘更多只看编程（用什么框架）、技术栈和参与项目。我也遇到过问开发“这个数据从哪里来？”答：“不知道，我调接口。”再问“这个数据去哪里？”答：“不知道，我调接口。”你知道为什么要用 Redis、为什么要用 ES、以及怎么用吗？答：“不知道，我调接口。”

对开发易用是好事，但开发也应该知其然和知其所以然。当然并不要求每个开发都精通源码，但应该能知道问题在哪里。例如开发人员应能判断是数据库的问题还是中间件的问题，然后知道是什么原因导致的就行。剩下的交给专业数据库人员来解决，给出解决方案即可。管好数据库和中间件，是从 DBA（administrator）向 DBA（architect）转变的阶段。
