# 基于代价的慢查询优化建议

2022-04-21 作者：粟含

1 背景
--------

慢查询是指数据库中查询时间超过指定阈值（美团设置为 100ms）的 SQL。它是数据库的性能杀手，也是业务优化数据库访问的重要切入点。随着美团业务的高速增长，日均慢查询量已经过亿条，此前因慢查询导致的故障约占数据库故障总数的 10% 以上，而且高级别的故障呈增长趋势。因此，对慢查询的优化变得刻不容缓。

最直接有效的方法是为查询选用高效索引。基于经验规则的索引推荐在简单 SQL 上常见有效，但对稍复杂的 SQL（如涉及多个可选索引、范围条件或多表 Join、子查询等）经验规则往往无法给出准确答案。基于代价的推荐更通用：它使用与数据库优化器相同的方式量化评估所有可能性，选出执行 SQL 耗费代价最小的索引。

2 基于代价的优化器介绍
--------------------

### 2.1 SQL 执行与优化器

一条 SQL 在 MySQL 服务器中的执行流程主要包含：SQL 解析、基于语法树的准备、优化器逻辑处理、代价准备、基于代价模型的优化、额外优化及执行计划运行等部分。

（图：SQL 执行与优化器）

### 2.2 代价模型介绍

优化器可在多种方案中选择（是否用索引、选择哪个索引、是否使用范围扫描、多表 Join 的连接顺序、子查询的执行方式等）。选出耗时最短方案需要定义量化指标——代价（Cost），对各方案计算代价并取最小值。

代价模型把操作分为 Server 层和 Engine（存储引擎）层两类：Server 层主要是 CPU 代价，Engine 层主要是 IO 代价。例如 MySQL 从磁盘读取一个数据页的代价 io_block_read_cost 为 1，计算符合条件的行代价 row_evaluate_cost 为 0.2。其他常见参数（MySQL 5.7 默认且可配置）包括：

- memory_temptable_create_cost (default 2.0) 内存临时表创建代价  
- memory_temptable_row_cost (default 0.2) 内存临时表每行代价  
- key_compare_cost (default 0.1) 键比较代价（如排序）  
- disk_temptable_create_cost (default 40.0) 磁盘临时表创建代价  
- disk_temptable_row_cost (default 1.0) 磁盘临时表每行代价

### 2.3 基于代价的索引选择

以如下 SQL 为例，观察 MySQL 优化器如何根据代价模型选择索引：

select * from sync_test1 where name like 'Bobby%' and dt > '2021-07-06'

建表并加入四个候选索引：

```sql
CREATE TABLE `sync_test1` (
  `id` int(11) NOT NULL AUTO_INCREMENT,
  `cid` int(11) NOT NULL,
  `phone` int(11) NOT NULL,
  `name` varchar(10) NOT NULL,
  `address` varchar(255) DEFAULT NULL,
  `dt` datetime DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `IX_name` (`name`),
  KEY `IX_dt` (`dt`),
  KEY `IX_dt_name` (`dt`,`name`),
  KEY `IX_name_dt` (`name`,`dt`)
) ENGINE=InnoDB;
```

通过 EXPLAIN 可以看到 MySQL 最终选择了 IX_name 索引：

```text
mysql> explain select * from sync_test1 where name like 'Bobby%' and dt > '2021-07-06';
+----+-------------+------------+------------+-------+-------------------------------------+---------+---------+------+------+----------+------------------------------------+
| id | select_type | table      | partitions | type  | possible_keys                       | key     | key_len | ref  | rows | filtered | Extra                              |
+----+-------------+------------+------------+-------+-------------------------------------+---------+---------+------+------+----------+------------------------------------+
| 1  | SIMPLE      | sync_test1  | NULL       | range | IX_name,IX_dt,IX_dt_name,IX_name_dt | IX_name | 12      | NULL | 572  | 36.83    | Using index condition; Using where |
+----+-------------+------------+------------+-------+-------------------------------------+---------+---------+------+------+----------+------------------------------------+
```

打开 MySQL 优化器 Trace，可以看到没有选择其他三个索引是因为在它们上的 range scan 的代价都 >= IX_name（这里摘录了部分 TRACE 输出）：

下面按代价模型推演代价计算过程（采用 MySQL 默认参数，block_size 默认 16KB，可通过 information_schema.tables 查询 data_length）：

- 全表扫描代价：
  io_cost + cpu_cost =（数据页数 * io_block_read_cost） +（数据行数 * row_evaluate_cost + 1.1）
  =（data_length / block_size + 1） +（rows * 0.2 + 1.1）
  = (9977856 / 16384 + 1) + (105084 * 0.2 + 1.1) = 21627.9

- 二级索引 IX_name 的代价：
  io_cost + cpu_cost =（预估范围行数 * io_block_read_cost + 1） +（数据行数 * row_evaluate_cost + 0.01）
  = (572 * 1 + 1) + (572 * 0.2 + 0.01) = 687.41

- 二级索引 IX_dt 的代价：
  = (38698 * 1 + 1) + (38698 * 0.2 + 0.01) = 46438.61

- 二级索引 IX_dt_name 的代价：
  = (38292 * 1 + 1) + (38292 * 0.2 + 0.01) = 45951.41

- 二级索引 IX_name_dt 的代价：
  = (572 * 1 + 1) + (572 * 0.2 + 0.01) = 687.41

补充说明：

- 计算结果在小数上有偏差，因为 MySQL 使用 %g 打印浮点数，小数会以最短方式输出。  
- 除了像 “+1.1 +1” 这种调节值外，Cost 计算还会出现 +0.01，用以避免 index scan 和 range scan 在 Cost 上产生微小竞争。  
- Cost 计算基于 MySQL 默认参数，若 Cost Model 参数、optimizer_switch 或数据分布变化都会影响最终 Cost。

### 2.4 基于代价的索引推荐思路

若想借助 MySQL 优化器为慢查询计算出最佳索引，理论上需要在业务表上真实添加所有候选索引，但对线上业务直接添加索引的时间与空间成本太高，不可接受。优化器选最佳索引用到的数据主要是索引元数据和统计数据，因此思路是向优化器提供候选索引的这些数据，而非在业务表上真实添加索引。

经过调研发现这是可行的：索引信息一部分在 Server 层的 frm 文件（如索引定义），另一部分在 Engine 层或可通过 Engine 接口获取（如索引列的不同值个数、索引占用页面大小等）。因此设计：构建一个支持“虚假索引”的存储引擎，在其上建立包含候选索引的空表，再采集样本数据计算统计信息提供给优化器，让优化器选出最优索引。调用关系如下：

3 索引推荐实现
------------

系统拆分为两个部分：支持虚假索引的 Fakeindex 存储引擎（C++ 实现），以及对外提供服务的 Go-Server。整体架构如下：

（图：架构图）

Fakeindex 是一个轻量级存储引擎，负责将索引相关接口透传到 Go-Server。由于需采用 C++ 实现并与 Go-Server 跨语言调用，采用 cgo 与 Go 原生轻量 RPC 技术，避免引入重量级 RPC 框架和第三方依赖。函数调用链路为：MySQL 优化器调用 Fakeindex 的 C++ 函数 → 参数转换成 C 语言 → 通过 cgo 调用到 Go 方法 → 使用 Go RPC 客户端向服务端发起调用。

（图：调用链路）

下面重点阐述 Go-Server 的核心逻辑，主要流程如下。

### 3.1 前置校验

根据经验规则排除一些不适合通过添加索引来优化的场景，例如系统库的 SQL、非 select/update/delete 的语句等。

### 3.2 提取关键列名

提取 SQL 中可作为索引候选的列名。除了 WHERE 中出现的列，MySQL 对排序、聚合、表连接、聚合函数（如 max）也能利用索引提高效率。通过解析 SQL 语法树，在 where、join、order by、group by、聚合函数节点中提取列名作为候选。

注意：有些 SQL 需结合表结构才能准确提取列归属，例如：

- select * from tb1, tb2 where a = 1：列 a 属于 tb1 还是 tb2 取决于哪个表包含该列。  
- select * from tb1 natural join tb2 where tb1.a = 1：自然连接可能使用相同列名连接，但 SQL 中并未直接暴露出这些可用于索引的列。

### 3.3 生成候选索引

对提取的关键列做全排列以包含所有索引组合（例如列 A、B、C 的所有组合包含 A、B、C、AB、AC、BA、BC、CA、CB、ABC、ACB、BAC、BCA、CAB、CBA）。需要排除不合适的索引，例如：

- 已存在的索引（例如已有 AB，则可以排除 AB、A，因为 MySQL 支持前缀索引）。  
- 超过最大索引长度（3072 字节）的索引。  
- 暂不支持的索引类型（如带地理数据类型列的空间索引）。

### 3.4 数据采集

从业务数据库采集数据，分为元数据、统计数据、样本数据三类：

- 元数据：表定义（列定义、索引定义），通过 show create table 获取。  
- 统计数据：表行数、表数据大小、索引大小等，通过 information_schema.tables 获取；已有索引的 cardinality（不同值个数）可通过 mysql.innodb_index_stats 查询。  
- 样本数据：候选索引为“假索引”，采集的统计数据不包含假索引的数据，因此需要从原表采样来计算假索引的统计数据。

（图：数据采集）

样本数据的采样算法应尽量反映原表的数据分布。直接使用随机采样（如 select * from table where rand() < rate）会对线上库造成大量 I/O，严重时可能引发故障。我们采用基于块的采样方式，参考 MySQL 8.0 的直方图采样算法：例如对一张 100 万行的表，需采集 10 万行样本，根据主键的最小值和最大值将表均分成 100 个区间，每个区间取一块 1000 行数据。示例采集 SQL：

```sql
select A,B,C,id from table where id >= 1000 and id <= 10000 limit 1000;
select A,B,C,id from table where id >= 10000 and id <= 20000 limit 1000;
...
```

将采集到的数据插入采样表后用于后续统计。

### 3.5 统计数据计算

两个核心统计数据示例：

- records_in_range：优化器在处理范围查询时用来估算走某索引可过滤出的行数，从而决定是否使用该索引。对假索引，我们按如下估算：样本中满足条件的范围行数 *（原表行数 / 样本表行数），即在样本数据中查找满足条件的行数，然后按采样比例放大估算原表中满足条件的范围行数。

- cardinality（索引区分度）：若直接按“样本不同值个数 *（原表行数 / 样本表行数）”放大，可能对像状态码等低基数字段造成高估（状态码可能最多 100 个不同值）。我们引入斜率和两趟计算规避该问题，流程如下：
  1. 第一趟：对样本数据的一半统计列 A 的不同值个数 R1（区间 [min_id, min_id + (max_id - min_id) / 2]）。  
  2. 第二趟：对所有样本统计不同值个数 R2（区间 [min_id, max_id]），计算斜率 R2/R1。  
  3. 判断斜率：若斜率 < 1.1，则视为固定值（示例为 100）；否则按采样比例放大（示例为 10,000）。

（图：统计数据计算）

### 3.6 候选索引代价评估

流程如下：

1. 在推荐引擎的 mysqld 上建包含候选索引的表：将候选索引塞入原表定义，并将存储引擎改为 Fakeindex。  
2. 在推荐引擎 mysqld 上执行 explain format=json SQL，获取优化器选择的索引及其代价信息。

注意：MySQL 表最多可建 64 个二级索引，但候选索引数随候选列数增长呈指数级（全排列），很快超过限制。统计美团线上索引列数分布后发现，95% 以上的索引列数 <= 3。基于经验，3 列索引能满足绝大多数场景，剩余场景通过拆表等手段处理。

即使限制为最多 3 列索引，当候选列数为 5 时，组合数量为 P(5,1)+P(5,2)+P(5,3) = 5 + 20 + 60 = 85，已远超 64。我们采用归并思路：将候选索引拆分到多个表中，分批次计算。先让 MySQL 在每个批次中选出最佳索引（可并行化），再在这些批次最佳索引之间选出最终最佳索引。该方案最多支持 4096 个候选索引，结合最大 3 列的限制，可支持计算出最多 17 个候选列的最佳索引。

（图：候选索引代价评估示意）

4 推荐质量保证
---------------

我们在美团数据库最近一周的线下慢查询数据上做了初步测试（共 246G，约 3 万个 SQL 模板用例）。结果显示系统基本覆盖了大部分慢查询，但仍存在无效推荐，主要原因包括：

- 推荐的 Cost 严重依赖样本数据质量，当表数据分布不均或数据倾斜时统计数据会误差，导致错误推荐。  
- 索引推荐系统自身缺陷导致错误推荐。  
- MySQL 优化器本身的局限或缺陷导致错误推荐。

5 慢查询治理运营
----------------

我们从时间维度将慢查询接入索引推荐并进行推广治理，分三类：过去（历史慢查询）、现在（新增慢查询）、未来（潜在慢查询）。

（图：慢查询治理运营）

### 5.1 过去 — 历史慢查询

历史遗留的慢查询数量大且推动力不足，可分两类接入：

- 核心数据库：这类慢查询通常会被周期性关注（如慢查询周报、月报），可提前生成优化建议并集中治理。  
- 普通数据库：将优化建议接入数据库平台的慢查询模块，让研发自助选择治理项。

### 5.2 现在 — 新增慢查询

当前产生、数量较少且为治理重点的慢查询，可实时发现并接入：

- 影响程度一般的慢查询：通过实时分析慢查询日志，对比历史慢查询识别新增慢查询并生成优化建议，为用户创建数据库风险项跟进治理。  
- 影响较大的慢查询：通常会触发数据库告警（如导致 Load 高），可通过故障诊断系统定位具体慢查询并生成优化建议，及时推送到故障处理群，缩短处理时间。

### 5.3 未来 — 潜在慢查询

尚未成为慢查询但可能演变成慢查询的 SQL（例如随着数据增长变慢的查询）：

- 未上线的准慢查询：在发布前的集成测试流水线中拦截被测试用例覆盖到的 SQL（Java 项目可通过 agentmain 代理方式），结合经验与 EXPLAIN 识别出潜在慢查询并生成优化建议，在需求管理系统上创建缺陷任务，解决后方可发布。  
- 已上线的准慢查询：当前执行较快但随数据增长会变慢（如全表扫描）。可通过开启 log_queries_not_using_indexes 记录到慢日志并生成优化建议，为用户创建风险项跟进治理。

7 未来规划
----------

美团日均产生近亿级别的慢查询数据，要实现诊断分析需要提升系统的大规模并发处理能力。目前系统针对单条 SQL 的优化，尚未考虑维护新索引带来的代价（如额外磁盘空间、写操作变慢），也未考虑 MySQL 选错索引可能会对其他 SQL 引发性能回退。业务或 DBA 更关心数据库或集群层面的优化。

业界（如阿里云 DAS）从全局角度综合考虑各因素，输出需要创建/改写/删除的索引，旨在实现数据库性能最大化提升并最小化磁盘空间消耗。未来我们也会不断改进，向基于 Workload 的全局优化方向发展。
