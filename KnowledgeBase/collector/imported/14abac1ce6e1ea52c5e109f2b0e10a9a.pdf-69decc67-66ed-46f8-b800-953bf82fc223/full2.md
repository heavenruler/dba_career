# MySQL 查询优化的三种处理阶段：Index Key、Index Filter 和 Table Filter

作者：二师兄（师兄奇谈）  
日期：2025-09-28，北京

在 MySQL 中，索引主要用于优化查询。查询优化涉及到三种处理阶段：Index Key、Index Filter 和 Table Filter。它们描述的是数据库在使用索引时，查询条件的匹配和过滤发生的过程，这个分类通常在讨论索引下推（Index Condition Pushdown, ICP）时出现。关于索引下推可参考相关文章《什么是MySQL的索引下推技术？》。

以下是对这三种索引处理阶段的详细说明：

## 1. Index Key（索引键）
定义：索引键是指索引的关键列，即索引中存储的字段值。索引键用来帮助定位数据行，查询通常首先使用索引键来筛选满足最基础条件的记录。

使用场景：当查询条件和索引的定义一致时，例如：
```sql
SELECT * FROM employees WHERE employee_id = 123;
```
如果存在 employee_id 的索引，数据库可以通过索引键快速找到符合条件的记录。

特性：
- 这是索引的基本功能，通过索引键直接定位满足条件的记录。
- 索引键检索的是精确匹配或范围扫描的记录。

查询执行流程：
- 通过索引键快速定位对应的记录集合。

## 2. Index Filter（索引过滤）
定义：索引过滤是在索引存储的数据上进行进一步过滤，用于实现更复杂的查询条件，而无需先通过索引定位所有数据然后回表。索引过滤在存储引擎层完成，是索引下推优化的关键部分。

使用场景：查询条件涉及多个字段，但不是全部字段都能通过索引键直接定位。例如：
```sql
SELECT * FROM employees WHERE employee_id > 100 AND salary < 50000;
```
假设有复合索引 (employee_id, salary)。数据库可以先基于 employee_id 定位部分范围的记录，然后在存储层通过 salary 进一步过滤索引中的记录，而不是直接将所有候选记录返回到 Server 层再过滤。

特性：
- 索引过滤是对索引本身存储的数据进行字段值筛选，而不是直接访问表行。
- 在支持索引下推的情况下，这部分过滤在存储引擎层完成，可以提高查询效率。

查询执行流程：
- 基于索引键定位候选记录。
- 在存储层进一步筛选索引中的记录，减少上层（Server 层）需要处理的数据量。

## 3. Table Filter（表过滤）
定义：表过滤是指数据库在回表（访问数据行）之后，对返回的表中数据进行过滤。通常用于查询条件涉及非索引列，或索引无法完成的过滤场景。

使用场景：查询条件涉及非索引字段，例如：
```sql
SELECT * FROM employees WHERE employee_id > 100 AND department = 'Engineering';
```
假设只有索引 (employee_id)。数据库通过索引范围查询 employee_id > 100 获取候选记录后，需要回表读取 department 列，并在 Server 层进行 department = 'Engineering' 的过滤。

特性：
- 表过滤发生在 Server 层，需要通过索引定位记录后回表查询原始记录再进行过滤。
- 如果查询条件中非索引列较多，或候选数据量较大，表过滤会带来较大的性能开销。

查询执行流程：
- 基于索引键定位候选记录。
- 回表查询原始数据。
- 在 Server 层对数据进行过滤，符合条件的记录才会返回给用户。

## 三类过滤的物理过程（概述）
- Index Key：初始阶段，通过索引键快速定位候选记录。
- Index Filter：在存储引擎层对候选记录的索引条目进行进一步过滤，减少需要回表的记录数（索引下推的实现）。
- Table Filter：如果查询涉及非索引列或更复杂的过滤条件，需要回表查询并在服务器层最终过滤。

## 索引下推（ICP）的重要点
- 在 MySQL 5.6 之前，一旦记录通过索引键查找到，复杂条件的过滤通常都在 Server 层完成（包括本应在索引层过滤的情况）。
- 从 MySQL 5.6 开始引入索引下推（ICP），将部分过滤逻辑（Index Filter）下推到存储引擎层，在回表查询之前完成过滤，显著减少回表次数并降低 Server 层负担。

## 示例
假设有一个包含索引 (employee_id, salary) 的表，查询如下：
```sql
SELECT * FROM employees
WHERE employee_id > 100 AND salary < 50000 AND department = 'Engineering';
```
- Index Key：索引通过 employee_id > 100 进行范围扫描，获取候选记录。
- Index Filter（索引下推实现）：在存储层通过 salary < 50000 过滤出满足条件的记录，减少回表次数。
- Table Filter：回表查询后，对 department = 'Engineering' 的条件进行过滤，最终返回结果。

## 小结
索引下推利用 Index Filter 在存储层完成过滤的能力，减少回表次数和 Server 层处理数据的压力，从而优化查询性能。通过合理的索引设计（例如覆盖索引），可以进一步减少回表，提高查询效率。
