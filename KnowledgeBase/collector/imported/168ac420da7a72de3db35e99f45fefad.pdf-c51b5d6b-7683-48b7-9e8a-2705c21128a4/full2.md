# 数据倾斜是分布式数据库应用中的两难问题

作者：白鳝

分布式数据库最初在互联网企业被大规模采用，最初要解决的也是大规模并发写入的性能问题。因此早期的互联网分布式数据库大多基于 HASH 分区，目的是让数据尽可能均衡。

把这套技术应用到通用业务场景中，完全的均衡就十分困难了。像 OceanBase（OB）这样的数据库采用小粒度的分片方式，尽可能让数据副本较为均匀地部署到多个 observer 上。而对于采用 CN/DN 这种存算分离架构的分布式数据库来说，更多要依赖于应用设计时分区键的设计。TiDB 的存算分离架构是另一种流派，数据分布可以做到更加均衡，但 TiDB 在理论上的稳定性和延时表现可能无法支撑对延时要求较高的场景，因此近期 TiDB 也可能推出存算一体、类似 OB 架构的产品。

无论哪种架构，在数据均衡分布上都无法完全依靠数据库本身，必须在应用设计上做大量工作。这也是将一个 Oracle 数据库的应用迁移到分布式数据库上难以实现平滑迁移的主要原因。

均匀分布数据会带来另一个问题：分布式事务的延时。如果采用完全均衡的数据分布，就不可避免会出现大量的分布式事务，而分布式事务会劣化应用性能。这样就像一个跷跷板：理论上可以做到完全平衡，但在实际应用中，总有一头会翘起。因此在一些关键业务系统中，分布式数据库厂商都会建议用户对应用做单元化改造，尽可能减少 XA 事务。

如果数据分布不均衡，最直接的问题是 CN 在生成执行计划时会变得复杂和不稳定，出现错误执行计划的概率会比集中式数据库大得多。如果并发执行计划不合理，就会因为长尾问题导致延时增加；同时，不同 DN 节点上同一个并行任务的相同算子生成的执行计划不同，也会导致执行延时不可控。

哪怕数据分布均衡，数据访问也不一定均衡。某些节点上的数据可能更“热”，该节点就会成为热点节点。如果某个热点节点的 CPU、内存、IO、网络等任一资源出现问题，该节点就会成为病灶，严重时会危及整个集群的健康。出现上述问题后，该节点上的子任务执行延时会变大，从而影响整个集群的性能。极端情况下，某个节点的会话数达到上限，该节点将处于不稳定状态，表现为时好时坏，甚至彻底不可用，这时设计者认为“坚不可摧”的分布式数据库可能就会陷入险境。

在传统的 OLTP 应用场景中，分布式数据库是门槛较高的产品，使用时应用需要做充分的适配工作，运维时也应密切关注各个服务之间的均衡性问题。如果出现严重偏差，一定要及时进行分析，最好在故障爆发前解决隐患。一旦在生产环境中爆发故障，恐怕不是十分钟二十分钟能够解决的。对于十分关键、高负载的分布式数据库应用来说，增加容量冗余可能是确保稳定运行的必不可少手段：用更大的硬件配置来降低节点负载不均衡引发的资源不足风险。
