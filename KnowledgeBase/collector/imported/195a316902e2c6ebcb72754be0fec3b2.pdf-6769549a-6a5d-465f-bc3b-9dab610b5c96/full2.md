# 图灵奖数据库大师 Stonebraker 师徒对数据库近20年发展与展望的论文整理

整理：叶正盛（NineData 创始人 & CEO、资深数据库专家）  
来源：对 Michael Stonebraker（MIT）与 Andrew Pavlo（CMU）2024 年联名 SIGMOD 综述论文的中文翻译与整理（节选与重构）

简要说明：本文为对 Stonebraker 与 Pavlo 关于过去二十年数据库发展与未来展望论文的中文整理与翻译。保留了原论文的主要观点与结构，修复了段落与换行，保留部分参考标注。该整理供学习与讨论使用，若需查阅原文请检索 SIGMOD 2024 或作者页面。

摘要
二十年前，Stonebraker 等人撰写的论文“What Goes Around and Comes Around”评论了自 1960 年代以来的数据建模运动，结论是具有可扩展类型系统的关系模型（RM，Relational Model）与 SQL 仍占主导，并不断吸纳替代方案的优秀思想。我们重新审视这一问题，认为自 2005 年以来这一演变仍在继续。尽管有人试图重复替换 SQL 或 RM，但 RM 仍是主导，SQL 已扩展以捕获来自其他模型的有益思想。我们预计 SQL 与关系型数据库管理系统（RDBMS）将继续演化；同时实现层面也发生了戏剧性的变化，主要由硬件发展驱动。

## 1 引言
2005 年，作者之一（Stonebraker）参与撰写的章节回顾了自 1960 年代以来的主要数据建模运动，包括层次结构（如 IMS）、网格结构（如 CODASYL）、关系型、实体-关系、面向对象、半结构化（如 XML）等。结论是：具有可扩展类型系统的关系模型（即面向对象-关系型）已经统治了竞争者，市场上没有其他成功的系统。即使许多非关系型 DBMS 仍存在，供应商通常将它们视为遗留维护目标，而不是用于构建新应用的首选。

自 2005 年以来，DBMS 扩展到几乎所有类型的数据，并催生了“大数据”时代，以及将机器学习（ML）与 DBMS 集成的趋势。本文分析过去二十年数据模型与查询语言的研究与开发，并讨论实现架构与未来关键考量。

## 2 数据模型和查询语言
我们将数据库中的数据模型与查询语言的研究与开发分为八类：MapReduce 系统、键值存储、文档数据库、列族/宽列、文本搜索引擎、数组数据库、向量数据库、图数据库。总体观点是：大多数偏离 SQL/RM 的系统未能主导 DBMS 格局，很多系统最终演变为支持 SQL 或被 RDBMS 吞并；SQL 吸收了许多优秀思想以保持相关性。

### 2.1 MapReduce 系统
Google 在 2003 年构建 MapReduce（MR）框架以处理周期性抓取数据。MR 将 Map 视为 UDF，Reduce 近似为 GROUP BY 操作，整体上可视为类似如下的单一查询：
```sql
SELECT map() FROM crawl_table GROUP BY reduce();
```
MR 并不规定特定数据模型或查询语言，Map/Reduce 函数自行解析数据格式。Yahoo! 在 2005 年开发了 Hadoop（基于 HDFS），并出现了大量生态系统。2009 年研究表明传统数据仓库 DBMS 在性能上优于 Hadoop，引发社区争论。随后两件事使这场争论变得无关紧要：一是 Hadoop 技术与服务市场在 2010 年代衰退；二是 Google 将抓取处理从 MR 转移到 BigTable，并在 2014 年逐步淘汰 MR。许多厂商在 Hadoop 之上构建了 SQL 接口（如 Hive），同时 Spark、Flink 等系统作为 MR 的改进应运而生并增加了 SQL 支持。

讨论：MR/Hadoop 的缺陷导致其衰落，留下 HDFS 遗产与一系列衍生产品。分布式 RDBMS 在云时代蓬勃发展，而 MR 的一些思想（共享磁盘、开放文件格式）促成了数据湖等新范式。

### 2.2 键/值存储（Key/Value Stores）
KV 模型最简单，表示 (key, value) 二元关系，value 通常是未标记的字节数组（blob），DBMS 本身并不理解其内容，应用负责解析。许多互联网公司在 2000 年代为特定场景（如缓存）构建了分布式 KV 存储，如 Memcached、Redis、Dynamo。第二类是嵌入式存储引擎，如 BerkeleyDB、LevelDB、RocksDB。

讨论：KV 存储提供快速“开箱即用”的存储方式，但当应用需要多字段或复杂查询时会很危险（无二级索引、没有联接等）。为此一些 KV 产品演进为记录/文档存储（例如支持 JSON）。近年来的趋势是将嵌入式 KV 存储作为全功能 DBMS 的底层引擎（例如 MySQL 的可插拔存储引擎、MongoDB 采用 WiredTiger、Meta 在 MySQL 中使用 RocksDB），以加速新 DBMS 的开发。

### 2.3 文档数据库（Document）
文档模型将数据库视为文档集合，文档包含字段/值对的层次结构（常见为 JSON）。示例：
```json
{
  "name": "First Last",
  "orders": [
    { "id": 123, "items": [...] },
    { "id": 456, "items": [...] }
  ]
}
```
文档 DBMS 在 2000 年代兴起以应对 OLTP RDBMS 的扩展性限制，并借助 NoSQL 口号推广。其主张包括：逐条记录接口更快、现代应用不总是需要 ACID。MongoDB 是最受欢迎的代表。

讨论：文档 DBMS 本质上与早期的面向对象 DBMS 和 XML DBMS 相似，主张消除对象关系不匹配、通过非规范化减少查询次数。但问题仍包括数据重复、预连接并不总是优于联接、缺乏数据独立性。到 2010 年代末，几乎所有 NoSQL DBMS 都加入了 SQL 接口并逐步支持 ACID，SQL 也在标准中加入了 JSON 数据类型/操作（SQL:2016）。因此文档系统与 RDBMS 的差距逐渐缩小，未来两者难以区分。

### 2.4 列族数据库（Column-Family）
列族（宽列）模型是对文档模型的简化，支持一级嵌套并允许可选属性与数组值。例：
User1000 → { "name": "Alice", "accounts": [123,456], "email": "xxx@xxx.edu" }

BigTable（Google，2004）是第一个列族模型实现，随后有 Cassandra、HBase 等。列族通常缺乏联接与二级索引。

讨论：文档模型的许多评论同样适用列族。Google 在 BigTable 上构建了 Spanner 等系统，但后来 Spanner 被重写以去除 BigTable 的残余。部分 NoSQL 系统逐步提供 SQL 前端（如 Cassandra 的 CQL、HBase + Phoenix），列族模型作为一个较为小众的选择存在。

### 2.5 文本搜索引擎
文本搜索以倒排索引为核心，通过将文档标记化为“bag of words”来支持全文检索。现代系统包括 Elasticsearch、Solr（均基于 Lucene）。这些系统在文本存储与索引方面强大，但事务能力有限。主流 RDBMS（Oracle、SQL Server、MySQL、PostgreSQL）也均支持全文索引，差别在于集成方式与便利性。

讨论：文本是无结构的，DBMS 的任务是从中提取结构（元数据、索引）以避免线性扫描。专用搜索引擎与 RDBMS 在很多场景可替代，视事务、索引与集成需求而定。

### 2.6 数组数据库（Array Databases）
数组是很多科学与工程数据的自然表示（向量、矩阵、张量）。早期数组 DBMS 可追溯到 1980s，代表有 Rasdaman、kdb+、SciDB、TileDB；HDF5、NetCDF 是常见数组文件格式。数组 DBMS 面临的挑战包括不规则网格映射、如何将多维数组高效存储在线性介质上以及支持高效的维度遍历。

讨论：数组 DBMS 属于细分市场，在基因组学、卫星图像、科学计算等垂直领域有价值。SQL 对数组支持逐步增强（SQL:2023 引入 SQL/MDA），但列式 RDBMS 在 OLAP 场景中占优，数组 DBMS 在专业领域仍有位置。

### 2.7 向量数据库（Vector）
向量模型可视为数组模型的一维特化，近年来因 AI 工具（embedding）而备受关注。向量数据库用于存储由模型生成的 embedding，并支持相似性搜索（近似最近邻，ANN）。示例数据形式：
(title, date, author, [embedding_vector])

向量数据库通过 ANN 索引加速相似性检索；常见系统有 Pinecone、Milvus、Weaviate，搜索引擎与传统 DBMS 也逐步加入向量支持（Elasticsearch、Solr、Vespa、Oracle、SingleStore、ClickHouse 等）。

讨论：向量索引作为特性被迅速扩散到 RDBMS 和 NoSQL 系统。预计向量 DBMS 会像文档 DBMS 一样逐步演化，加入事务、SQL 等功能；同时 RDBMS 将原生支持向量索引，使专用向量数据库的必要性下降。

### 2.8 图数据库（Graph Databases）
图数据库在学术与工业界受到关注，常见表示为属性图或 RDF（三元组存储）。属性图为主流，图 DBMS 有两类用例：OLTP（如 Neo4j，利于长链遍历）和图分析（最短路径、连通分量等）。许多图系统提供图专用查询语言或并行计算框架（TigerGraph、JanusGraph、Giraph、Turi/GraphLab 等）。

讨论：图可以在关系模型中用节点/边表来表示，因此 RDBMS 一直是选项之一，但原生 SQL 在图遍历上的表达力不足。许多 RDBMS 已添加图扩展（MsSQL、Oracle）或通过层支持图 API（Amazon Neptune、Apache AGE）。SQL:2023 引入属性图查询（SQL/PGQ），进一步缩小了原生图 DBMS 与 RDBMS 之间的功能差距。实证研究显示，在很多情况下 RDBMS 模拟图并配合优化（如因子分解执行、最坏情况优化连接）可优于专用图 DBMS，因此专门图数据库市场可能不会很大。

### 2.9 小结
总体可得结论：
- MapReduce 系统：已成为历史或遗留技术。
- 键值存储：许多演化为 RM 系统或用于特定场景，可被现代高性能 RDBMS 匹配或超越。
- 文档数据库：逐步与 RDBMS 融合，差异缩小。
- 列族系统：仍偏小众（若无 Google 生态，其存在性受限）。
- 文本搜索引擎：在多个存储架构中用于文本字段，若 RDBMS 搜索支持更好则可能不必单独存在。
- 数组数据库：在科学应用保有市场，通用 RDBMS 难以完全替代。
- 向量数据库：单一用途数据库，RDBMS 很快会原生支持相关索引与搜索。
- 图数据库：OLTP 图应用多被 RDBMS 服务，图分析在内存中使用专门数据结构更优，专门图 DBMS 市场有限。

此外，很多所谓“新”模型实际上是对既有模型的重新命名或小幅变体。社区将继续探索新想法，但不太可能被关系模型完全取代。

## 3 系统架构
过去二十年 DBMS 架构出现了若干重要新思想，这些思想反映了应用与硬件特性的变化：列式系统、云数据库、数据湖/湖仓、NewSQL、硬件加速器、区块链数据库等。下面逐项讨论。

### 3.1 列式系统
数据仓库（OLAP）通常具备：历史性、数据量大（TB~PB）、查询只访问少数属性且多为即席查询。列式存储相比行存储有多重优势：更高的压缩率、向量化执行更适合列式而非 Volcano 风格每行操作、以及更小的每记录存储开销。过去二十年，数据仓库供应商纷纷转向列式存储，新厂商（如 Redshift、BigQuery、Snowflake）也以列式为核心。列式存储已彻底改变 OLAP DBMS 架构并主导数据仓库市场。

### 3.2 云数据库
云平台兴起改变了 DBMS 的实现与销售模式。对象存储（如 S3）使得计算与存储分离成为可行方案，带来弹性伸缩、不同硬件选择、资源复用与查询下推等优势。Snowflake 借助 serverless 与多租户执行方案推动了云原生 DBMS 的实践。云改变了许多对于 DBMS 架构的传统认知，shared-disk（或共享存储）在云中重新流行，而 shared-nothing 的优先级下降。云环境也让供应商更容易进行持续监控、无缝推送更新与收集运营数据。商业上，开源 DBMS 也面临被云厂商货币化的风险（如 Amazon 与 MongoDB、Elasticsearch 的冲突）。

### 3.3 数据湖 / Lakehouses
在云对象存储上直接存储文件，使得数据湖成为替代单一专用数据仓库的方案。应用程序将文件写入对象存储后，借助 lakehouse 引擎（如 Databricks、Dremio、Presto/Trino）进行统一查询，支持 SQL 与非 SQL 工作负载。开源列式文件格式（Parquet、ORC）和内存交换格式（Apache Arrow）促进了系统之间的数据共享。中间件（如 Delta Lake、Iceberg、Hudi）为数据湖提供元数据管理、版本控制与事务支持，缓解治理问题。数据湖带来新的优化挑战（如统计信息缺乏），促使自适应查询处理的兴起。

### 3.4 NewSQL 系统
为满足分布式 OLTP 的事务性需求，NewSQL 在 2010 年代初出现，目标是兼顾 NoSQL 的可扩展性与传统 RDBMS 的 SQL 与 ACID。NewSQL 可分为内存型（如 H-Store/VoltDB、SingleStore、Hekaton、HyPer）与面向磁盘的分布式系统（如 NuoDB、Clustrix）。但 NewSQL 的大规模爆发并未出现，部分原因是既有 DBMS 已够用且迁移成本高；此外 NewSQL 常只支持 SQL 子集或在多节点事务上受限。随后又出现了若干支持分布式事务的现代系统（TiDB、CockroachDB、PlanetScale、YugabyteDB），而主流 NoSQL 系统也逐步加入事务支持。

### 3.5 硬件加速器
过去尝试为 DBMS 定制硬件（数据库机器）多次失败。最近二十年趋势转为利用通用加速器（FPGA、GPU）来加速查询，例如 Netezza（FPGA）、若干 GPU 加速 DBMS（Kinetica、Sqream、HeavyDB 等）。但硬件集成的工程成本、云 CPU 资源的低价与规模经济让专用加速器难以被广泛采用。大云厂商具有整合定制硬件的能力（如 Amazon 的 Redshift AQUA、Google 在 BigQuery 的内存 shuffle 上的定制），因此加速器成功的可能性更多依赖云提供商。预期未来仍会有许多尝试，但商业化难度大。

### 3.6 区块链数据库
区块链作为去中心化的不可变日志，适用于点对点、无法信任任何参与者的场景。除加密货币外，区块链在传统商业数据库场景的适用性有限，因为它在性能上付出巨大代价且对大多数企业应用而言不必要。一些尝试（如 Fluree、BigChainDB、ResilientDB）试图把区块链概念应用于 DBMS，但通常基于错误的前提出发。私有链或类区块链产品（如 Amazon QLDB）提供可验证与不可变更新特性，但并非去中心化。总体来看，区块链数据库目前处于低效且应用有限的趋势。

### 3.7 小结
系统架构方面的要点：
- 列式系统彻底改造了 OLAP DBMS 架构；
- 云已颠覆可扩展 DBMS 的传统做法，云原生成为主流；
- 基于对象存储的数据湖/湖仓、开源格式将成为未来 OLAP 原型；
- NewSQL 引入新思想但影响尚未达到列式/云的程度，促成分布式事务性 SQL DBMS 的出现；
- 硬件加速器主要可能由云供应商推动，独立厂商面临较大挑战；
- 区块链数据库在大多数场景下为低效的解决方案。

## 4 结语
对过去二十年数据库演进的几点启示与建议：

- 不要低估良好营销对糟糕产品的推动力。数据库市场竞争激烈，优秀的营销能为不成熟产品争取时间（历史上 Oracle、MySQL、MongoDB 等都有类似例子）。
- 对大型非专业数据库供应商的产品要保持警惕。许多成功系统最初是为特定公司内部构建，再以开源形式发布（例如 Meta 的 Hive、Presto、Cassandra、RocksDB；LinkedIn 的 Kafka、Pinot）。这类系统在开源初期常不够成熟，需谨慎采用。
- 重视“开箱即用”的体验。许多 NoSQL 的卖点是更好的初始体验；现代 DBMS 应简化本地与云文件的原位处理（例如 DuckDB 的流行部分归因于此）。
- 提供自动化功能（物理设计、参数调优、模式设计、查询调优）——“自动驾驶”DBMS 是重要需求。
- 开发者需要直接查询数据库。虽然 ORM/抽象层方便原型开发，但在性能关键场景下显式 SQL 更可控，因此支持 SQL 的 RDBMS 更为稳妥。
- AI/ML 将对 DBMS 产生重大影响。自然语言（NL）到 SQL 的转换因 LLM 进步而复兴，但 NL 并不能完全替代 SQL，尤其在 OLTP 或需要可解释性的企业决策场景中。LLM 在 SQL 生成、查询构建时可作为辅助工具，但其输出可解释性与训练数据需求限制了企业的快速采用。另一方面，AI/ML 也可用于优化 DBMS（如 ML 驱动的查询优化器、配置调优、访问方法），这是有前景的研究与工程方向。

## 5 结论
我们预测未来几十年数据库发展仍将循环往复：新一轮开发者会宣称 SQL 与 RM 不足以应对新需求，新的数据模型与语言将被提出与探索。但这些新模型更多会为 SQL 吸收有价值的思想而非完全取代关系模型。为加速下一代 DBMS 的发展，社区应促进开源可重用组件（文件格式、优化器框架、执行引擎等）的发展，以减少重复造轮子、提高互操作性。已有努力包括 Parquet/ORC、Calcite、Orca、DataFusion、Velox 等。数据库社区应努力形成类 POSIX 的内部标准以促进互操作。

最后的寄语：希望能在未来二十年后再次回顾数据库领域的发展，站在前人的肩膀上继续前行。
