# 面试官：MySQL 空值字段应该保存 NULL 还是默认值？

作者：君哥聊技术（朱晋君），后端架构师

大家好，我是君哥。使用 MySQL 数据库时，对于一个可以为空的字段，如果没有值，应该保存 NULL 还是给一个默认值呢？多数时候我们不太注意，有时候不赋值直接保存 NULL，有时候赋值一个业务指定的默认值。今天来聊一聊这个话题。

## 1. 行数据存储

MySQL 保存一行数据时，不仅保存数据本身，还会保存与数据相关的额外信息。InnoDB 存储引擎支持四种行格式：REDUNDANT、COMPACT、DYNAMIC、COMPRESSED。MySQL 5.7 之后默认使用 DYNAMIC 行格式。DYNAMIC 和 COMPRESSED 是 COMPACT 的改进版，基本结构与 COMPACT 类似，我们这里看一下 COMPACT 这种格式。如下图：

我们创建一张表：

```
CREATE TABLE `t_user` (
  `id` bigint ( 20 ) NOT NULL AUTO_INCREMENT,
  `name` varchar ( 16 ) DEFAULT NULL ,
  `email` varchar ( 32 ) DEFAULT NULL ,
  `address` varchar ( 255 ) DEFAULT NULL ,
  PRIMARY KEY ( `id` )
) ENGINE = InnoDB DEFAULT CHARSET =latin1;
```

插入 2 行数据，数据行保存格式如下图：

- 变长字段宽度列表保存变长字段非空值长度。从上图可以看到，变长字段宽度列表存放的列宽度顺序和数据表中的列顺序相反，也就是说变长字段宽度列表是逆序存放列宽度。
- 如果表中所有列都是 NOT NULL 并且具有固定长度，则没有变长字段宽度列表这部分。
- NULL 值列表也是逆序保存，当该值是 NULL 时，用二进制 1 表示，否则保存二进制 0。如果表中所有列都是 NOT NULL，就没有 NULL 值列表这个部分。
- 记录头信息用 5 个字节保存，主要记录数据的一些信息，比如：
  - delete-flag：记录是否删除（MySQL 中删除一条数据并不会马上从磁盘上删除，而是打上删除标记，空闲时间再进行异步清理）。
  - record_type：记录类型，比如普通记录、非叶子节点记录。
  - next_record：指向下一条记录的地址指针。
  - n_owned：记录该组数据的条数。
- 隐藏列：
  - DB_TRX_ID：修改（插入、更新或删除）这一条数据的事务 id；
  - DB_ROLL_PTR：回滚指针，指向修改前的历史版本，用于回滚操作；
  - DB_ROW_ID：当表中不定义主键时用作主键来自动生成聚簇索引。

## 2. NULL 处理

根据上面的存储结构和实际使用，如果把字段定义为 NOT NULL，有下面好处：

- 节省存储空间：NULL 值虽然不占用数据存储空间，但需要额外 1~2 个字节来保存 NULL 值列表（null bitmap）。
- 减少应用端 NullPointerException 的可能性。
- 减少统计问题：比如 `count(字段)` 不会统计 NULL 值。
- 对索引有好处：索引通常不保存 NULL 值，定义成 NULL 可能使索引效率下降。
- 比较操作更直观：字段为 NULL 时，很多比较会返回 NULL（不可直接用 `=`、`!=`、`>`、`<`），避免歧义。
- 范围/集合操作：字段为 NULL 时，使用 `IN`、`NOT IN` 等语句需要注意 NULL 带来的三值逻辑影响。

当然，设置为 NULL 也有好处：

- 语义清晰：NULL 表示“无值”或“未知”，语义上更准确。
- 灵活性：使用 `IS NULL` 更容易筛选缺失值。
- 兼容性：在某些 JOIN 或规则中，NULL 的行为有助于保持数据一致性和完整性。

在实际项目中，我们经常在值为 NULL 的情况下给一个默认值，比如 "-"、""、"N/A" 等，这一定程度上避免了空指针，但往往带来额外问题，例如上下游系统因为默认值不一致导致业务处理受影响。

在表设计时，不必过度考虑定义为 NULL 或默认值在存储空间上的微小影响。更重要的是制定系统整体设计规范，保证各子系统在设计上的一致性，这样才能让处理逻辑更加健壮。
