# 【稳定性】浅谈团队如何做好系统稳定性
京东云开发者 · 2024年04月09日 · 2616 次阅读

背景  
稳定性建设需要一系列具体的建设活动推进和落地，这些建设活动涉及人员、机制和文化，全方位的建设活动才能更好地落实建设模式。

一、稳定性保障机制  
稳定性涉及团队所有不同水平技术人员、所有系统、研发所有环节、线上时时刻刻，单个技术人员无法完全保障，需要建立团队流程和机制来可持续保障。

人为因素的根源一方面是专业能力不足、经验不足，另一方面很多都是无心之失，所以需要通过流程和规范来保住“底线”，减少人为因素导致的故障。大家严格遵守各种规范（CodeReview 规范、发布 xbp 流程、上线后 double-check 机制），通过流程和 double-check 机制确保每个人发布不会太差，解决人的因素。永远要记住团队的力量是无穷的，要学会借力。

关键还是要靠大家，稳定性工作需以规范先行，落地一套稳定性的机制体系，用机制的严格执行来约束大家。否则无法展开。该机制包括：

- 方案评审机制：在完成系统建设或改造方案初稿后，需通过由业务、技术、测试、运维组成的团队进行方案评审，才能进一步实施。
- 架构设计规范：包含概要设计、模块详细设计、API、Domain、数据缓存、容错设计、风险设计等。
- 代码编写规范：覆盖代码基础、日志、配置、多线程、数据库、异常使用等多层面，提升代码质量。
- 代码评审规范：包括 changelist 描述、兼容性、性能、复杂性、团队评审文化等。
- 代码提测规范：包含单元测试、代码编译构建、系统运行稳定性等要求。
- 代码测试规范：进入稳定性测试阶段，要严格审查系统是否达到测试准入条件，满足测试实施的所有必要条件，否则不开展稳定性测试；测试结束后严格检查所有测试准出条件是否满足（如无进行中的缺陷），否则不予通过。
- 预发与引流压测规范：黄金链路必须进行 R2 引流验证。
- 发布上线规范：支持灰度、可验证、可回滚等能力。
- 验收规范：业务、产品验收规范。
- 变更规范：提供变更级别、角色职责、活动阶段以及输入输出的详细规定。
- 运维操作规范：针对公司日志标准，提供统一的日志排查命令及规范。
- 报警响应机制：针对运维相关的监控告警制定告警处理流程与告警升级机制。
- 值班及责任判定机制：设置值班制度，每天有技术人员负责值班，值班周期内的问题由值班人员治理；不能及时完成的，添加到 BUG 定期跟踪并统计。生产事件发生后，由专家团队详细分析原因和解决办法，并对问题进行问责，明确责任团队、责任人及责任承担比例，避免稳定性治理中出现“囚徒困境”。
- 故障管理机制：规范故障响应流程、故障升级机制、故障复盘机制，明确技术人员在应对突发故障时的操作流程与职责边界，提升沟通效率，推动故障闭环，提升故障处理效率。

二、开发与 SRE 的区别  
SRE（Site Reliability Engineering，站点可靠性/稳定性工程师）既不同于传统的软件开发工程师（Dev），也不同于传统的运维工程师（Ops），更像是二者的结合。SRE 模型是对 DevOps 的实践和拓展，是一种对系统稳定性、高可用、团队持续迭代和持续建设的体系化解决方案。

举例说明两者对“问题”的不同处理方式：

- 开发：了解业务 -> 定位问题 -> 排查问题 -> 解决问题
- SRE：了解业务归属 -> 快速定位问题范围 -> 协调相关人投入排查 -> 评估影响面 -> 决策恢复手段

开发人员面对问题，会倾向于探究根因并研究解决方案；而 SRE 首先评估影响，快速定位并止损恢复。目标和侧重点的不同，造成了 SRE 思考问题的特殊性。因此，成为一名 SRE，需要从态度和方式上进行转变，切换到“团队稳定性负责人”的角度思考问题。

三、对 SRE 的几点要求

1. 责任心、细心、耐心。  
   - 负责任是第一要素，主动承担，对报警、工单、线上问题、风险主动响应，不怕吃苦。一个不负责任、边界感过强的人难以做好稳定性工作。  
   - 及时、快速的响应是关键。SRE 必须能够第一时间冲上去，不要先问是否自己的责任，而要关注事件的影响、是否存在坑、是否有需要优化的风险。  
   - 主动走到最前面：主动想优化办法、主动出头解决问题、主动挖掘系统风险薄弱点。

2. 不能只做当下，要看到未来的风险，善于总结。

3. 把机制建立好并切实落地。作为 SRE，要以机制保障“尽量不出问题”这一基线，关键在于团队协作和机制执行。

四、稳定性建设方向

1. 地基要打牢  
稳定性建设重在预防。根据多年经验，至少 70% 的线上故障可以通过预防工作消除。因此需要投入精力进行根基建设，把开发、测试和上线三大流程做到透彻，包括 Design Review、Code Review、提测流程、上线流程、引流验证、性能测试等。

2. 工作在日常  
稳定性工作不是一蹴而就，而是通过日常积累逐步完成，需要团队人人参与、持续完善监控告警，检查每一个告警是否已配置并及时消灭线上小隐患。可参考每周的稳定性会议，工作要点包括：

- 梳理：主动梳理团队的业务时序、核心链路流程、流量地图及依赖风险，通过该过程明确链路风险、流量水位与时序冗余。
- 技术债务治理：主动组织技术债务风险治理，将梳理出的风险以专项形式治理，防患于未然。但治理过程中需注意避免导致线上问题，应加强引流验证比对。
- 演练：把风险化成“攻击”，在无故障时制造一些可控故障点，通过演练提高大家的响应能力与对风险点的认知。
- 报警：除了主动响应外，还要经常做报警保险和机制调整，保证报警的准确度与团队对报警的敏感度，同时避免疏忽任何一个点，因为疏忽可能导致问题。

3. 预案是关键  
认识到预案的重要性，投入精力制定和更新预案，才能更好应对突发情况，保障项目顺利进行。通过每周的稳定性会议深入挖掘每个接口的隐患与不足，例如业务指标是否齐全、是否能真实反映接口特性等。

4. 大促特殊场景  
系统在大促期间与日常稳定性的区别主要有两点：

- 技术层面：高并发流量。大促流量峰值是日常的几十到几百倍，需要具备更高的并发处理能力，做好流量评估与容量规划。
- 业务层面：场景多样化。大促会增加许多日常不常用的场景（如预售、特殊时效控制、停运降级功能等）。应整理这些功能点，在大促前 1 个月模拟大促，进行相关功能配置并演练全流程，避免在大促期间发现功能不可用或不兼容的问题。

5. 执行是王道  
复盘只是手段，关键是对本系统进行自查并立即整改。别人暴露的问题很可能也存在于自己系统，应优先解决，而不是知晓却不行动。

五、前置：扁鹊三兄弟的启示  
SRE 不应在事后被动应对问题，而要走在前面，看到未来风险：

- 做“扁鹊大哥”（事前控制）：擅长事前发现与防范，具有敏锐洞察力和战略眼光，在系统健康时发现问题。
- 做“扁鹊二哥”（事中控制）：擅长快速、果断处置，在系统有隐患时及时发现并控制。
- 做“扁鹊小弟”（事后控制）：擅长应急处置，在系统发生问题时快速解决。

古籍《鶡冠子·卷下·世贤第十六》记载：魏文王曾问扁鹊哪位兄弟最好，扁鹊回答长兄最好，中兄次之，他自己最差；但他最出名是因为他总在病情严重时进行显著的治疗操作，所以名声更大。借此类比，最好、最有价值的稳定性工作往往是那些在事前和事中就已解决与防范的问题，而不是只靠事后救火获得名声。
