# MySQL 版本核心差异分析（以 8.0.28 起的演进为主）

作者：iuaustin3（AustinDatabases）

本文整理了从 MySQL 8.0.28 起的一些核心改进与差异点（作者为社区观察与汇总，供参考）。同时列出作者在数据库领域的系列文章分类索引，便于阅读与学习。

## 核心演进摘要（要点）

- Instant DDL（ALGORITHM=INSTANT）
  - 从 8.0.28 起，MySQL 在某些 DDL 操作上加强了 ALGORITHM=INSTANT 的支持。ALGORITHM=INSTANT 的操作不涉及数据文件或索引的大规模重写或拷贝，通常只修改数据字典和内存中的表定义（metadata），因此能够在毫秒级完成且对表的并发读写影响极小。
  - 具体到列操作：过去修改列名或删除列常常需要重建表；自 8.0.x 的逐步演进后，rename column、drop column 等在更多场景下可以通过 ALGORITHM=INSTANT 执行，避免表重建。对 add column 指定 BEFORE/AFTER 导致需要重建表的问题也在后续版本中得到改善，但仍需注意具体限制和兼容性。

- Redo log / redo buffer 的可配置性
  - 近年版本引入了对 redo log buffer 或 redo 相关容量配置的改进（原默认值和调整方式存在历史约束，部分参数在早期需要重启才能生效，后续版本逐步增强了可配置性和行为透明度）。在做大事务或高并发写入的场景时，关注 redo log、innodb_log_file_size、innodb_log_files_in_group 等配置仍然很重要。

- binlog 格式的收敛
  - 在部分较新版本中，binlog 格式的选择发生了收敛趋势，社区/官方逐步倾向于仅使用 ROW（基于行的二进制日志）而弱化或移除 STATEMENT / MIXED 等老旧格式。实际环境中请以当前安装版本的官方文档为准，并评估复制与恢复策略的影响。

- 认证与插件的调整
  - 部分认证方法或插件在演进中被移除或调整（例如有些较老的认证方式被标注为弃用或在云/产品化部署中被替代为更现代的集成方式）。云厂商与托管服务会倾向于将认证与访问控制外部化或做差异化处理，迁移与升级时需注意插件兼容性。

- 安全、OpenSSL 与兼容性调整
  - 各版本中持续包含安全修复、性能改进以及 OpenSSL 等第三方组件的升级。某些版本引入了不兼容变更（例如索引格式或元数据层面的调整），可能需要重建索引或做额外兼容性处理；升级前务必阅读相应版本的变更日志与兼容性说明。

## 版本要点速览（摘要，按原文整理）

- 8.0.28 以后：对 RENAME COLUMN 和 DROP COLUMN 在更多情形下支持 ALGORITHM=INSTANT，Instant DDL 优化使此类操作只修改数据字典并快速生效。
- 8.0.29、8.0.30 等：继续完善 Instant DDL 的支持；引入或改进 redo log / redo buffer 相关可配置能力（提升对大事务场景的弹性）。
- 8.0.34：关于 binlog format 的调整（强调 ROW 格式）；对认证与插件有若干调整（部分旧的认证方式被弱化/移除）。
- 8.0.35 / 8.0.36 / 8.0.39 / 8.0.40 等：持续推出安全与性能修复，OpenSSL 等依赖升级；个别版本被回退或修复（例如某些 InnoDB 重建失败的问题在后续版本被修复）。
- 8.0.41 及之后：在某些发布中出现需要重建索引的兼容性变更（升级前请务必查阅具体 release notes），并继续跟进 OpenSSL 的小版本升级。
- 8.4.x（文中提及为 LTS）：在 LTS 盘点中列出若干稳定性、复制与日志相关的改进。注意：不同发行与厂商（云服务）对版本号、LTS 策略可能有所不同，请以实际产品文档为准。

## 结语与建议

- 升级前必读：任何主要版本升级前，应仔细阅读该版本的 release notes 与兼容性章节，关注 InnoDB、索引、binlog、认证插件以及第三方依赖（如 OpenSSL）的变更。有些改动可能需要重建索引或调整配置。
- 测试环境先行：在生产环境升级前，务必在测试环境充分验证业务场景、备份恢复、复制拓扑与性能行为。
- 关注云厂商差异：云托管或数据库即服务（DBaaS）在版本、配置与插件支持上与原生 MySQL 会有所差异，迁移/升级时需同步厂商文档与限制。
