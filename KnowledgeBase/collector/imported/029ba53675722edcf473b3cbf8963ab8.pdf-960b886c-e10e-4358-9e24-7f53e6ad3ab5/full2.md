# 转行DBA，给你分享数据库运维的N条建议（随时更新）

作者：多明戈教你玩狼人杀  
日期：2023-06-28

这个月一直在构思一篇和技术关联不太多的文章，导致临近月底才发现自己完成的文章是0。最近在给朋友传授一些转行 DBA 的经验，顺手把这些要点整理出来。这些是同行们总结的经验，也有一些是我自己的切身体会，希望对你有所帮助。

## 总则
- 备份。永远不要让生产库处于没有备份或无法恢复的状态。在资源允许的范围内，确保你的备份集的时间戳是连续的。只有这样在最坏情况下你才有机会重来。
- 冗余。备份只是最后一道防线，重要的生产库还需要冗余，包括实例级别的冗余和数据级别的冗余。
- 定期演练。无论是备份恢复还是灾备演练，都要定期做，少则一个季度，多则一年。真正需要用到这些手段时，演练能暴露很多未预见的问题。
- 防患于未然。再及时、再专业的故障处理，也不如从设计、配置上避免问题。早发现、早处理，而不要等问题堆成大雪球。

## 用户管理
- 密码复杂度。超级管理员密码绝不可敷衍，确保长度与大小写、数字、特殊字符的组合。生产环境密码最好以 12 位起步。除非有正规流程，不要将管理员密码交给他人，包括直属领导。
- 最小权限分配。为每个非管理账号只分配完成其业务所需的最小权限，避免冗余权限，因为你无法预判这个账号会被如何使用。
- 权责分离。不要用个人用户进行管理操作，也不要给个人用户管理员权限。应使用专门的管理用户，并配合跳板机、临时凭证等鉴权方式。
- 及时下线。对不再使用的数据库账号要及时锁定并排期删除，不要长期处于开放状态。
- 业务分离。不同业务系统绝对不要使用同一账号，即便属于同一部门或同一批人在使用。

## 监控管理
- 无监控不上线。所有上线到生产的数据库必须有监控，确保随时掌握数据库运行状态，便于排查、追踪并界定责任。
- 动态调整。监控指标不是一成不变的，需要在长期运维中不断调整，以适应运维团队配置和业务诉求。
- 指标覆盖。除了基础的资源使用（CPU、内存、存储、IO、连接数等），还要关注慢查询、锁等待、复制延迟、备份状态等与可用性和一致性相关的指标。

## 访问安全
- 环境隔离。生产库和测试库必须严格隔离，不仅分离账号和服务器，甚至要做到网络防火墙、权限等层面的隔离。
- 跨库访问控制。尽量避免数据库层面的跨库访问（如 dblink）；优先使用 API 或中间层来降低安全隐患。
- 审计。对重要业务系统要有足够的审计手段，能够追踪谁在什么时候对数据库做了什么操作。

## 成本管理
- 有限分配。对于重要度低且业务量小的系统，如果没有专门预算，资源分配以“够用”为原则（算力、内存、存储）。
- 利用冗余资源。备库和灾备中心的冗余资源可以被合理利用，例如将只读 standby 用于报表、把边缘服务部署到灾备中心的空闲服务器上。
- 冷热数据分离。将热数据放在高性能存储，冷数据放在成本更低的存储上。随着 SSD 价格下降，这一策略需要随时间评估调整。
- 做好利旧。生产淘汰的设备在符合公司流程和安全标准的前提下可以继续用于测试，既节约成本也便于交接。

## 流程管理
- 无工单不变更。所有针对生产环境的配置变更、账号变更、启停等应走工单流程。即便事出紧急，也要在事后尽早补齐流程与记录。
- 标准化部署。新环境的部署要形成标准化方案并落成文档，无论是手动还是自动化部署，都要可复用、可交接，降低后续风险和成本。
- 事故报告。每次生产事故都要写事故报告，交代来龙去脉，附上相关日志，并尽量详尽地记录处理办法与根因分析，以便复盘与改进。
