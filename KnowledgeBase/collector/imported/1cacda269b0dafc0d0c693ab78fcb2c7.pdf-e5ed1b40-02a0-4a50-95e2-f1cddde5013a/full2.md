# 淘宝质量保障之主动预警能力建设

营销交易技术团队 | 大淘宝技术 | 2025-04-09

本文围绕淘宝特价版的主动预警建设展开，详细阐述了在业务质量保障中主动预警的重要性及其具体实施策略。文章首先分析了预警范围，包括活动/资源位配置过期、权益类问题、开发常用配置平台、实验人群过期以及舆情类问题等五大类。接着介绍了预警流程的设计，借鉴集团技术风险平台的风险预警机制，并结合自身业务实践，搭建了预警数据采集能力与规则制定体系。核心部分探讨了针对不同场景（如平台类配置、权益类、配置类、实验及舆情）的具体解决方案，解决了多数据源对接、动态识别新增预警等难点。最后展望了后续计划，强调主动预警能力建设的持续优化与场景拓展，以进一步提升业务稳定性与用户体验。

## 背景

在业务的质量保障过程中，主动预警是一项重要能力，可以帮助我们提前发现问题，尤其是权益过期、库存耗尽、资源位过期等问题。等到监控发现再恢复时，再快也快不过提前预警。因此，我们探索在业务各个方面实现主动预警监控的能力。

## 主动预警策略

### 预警范围

预警的第一步是弄清楚在项目质量保障过程中需要主动预警的项有哪些，这是一个探索发现的过程。最初遇到最多的问题是活动配置过期、红包库存不足，但随着深入了解业务，发现更多需要预警的点。我们将配置划分为以下几类：

- 活动/资源位配置过期：不同业务域可能使用不同的配置后台，有淘宝特价版自建的运营后台，也有集团的各种活动配置、资源位配置平台，这些地方都会存在配置过期的问题。
- 权益类：包括集团统一权益、自建运营平台权益、其他资格券权益等，主要问题是库存不足、权益过期。
- 开发常用的配置平台：一些开发会将时间配置在这些平台上，这里的配置过期也会导致业务异常。
- 实验类：人群或实验配置过期也会导致业务异常。
- 舆情类：包括淘宝特价版和手淘的舆情监控。

### 预警流程

在确定预警范围后，需要确定预警及接手处理的流程。我们借鉴了集团技术风险平台的风险预警机制，并结合自身业务实践，最终确认了以下流程要点：

- 预警告警、群内通知、预警升级、预警接手处理与结单等流程，复用了集团风险平台成熟的能力。
- 核心工作是将需要预警的内容接入风险平台：搭建预警数据采集能力、对接各个平台数据、制定适合业务的预警规则。
- 根据优先级先覆盖资金相关、前台资源位等高优先项，逐步实现整套主动预警体系。

## 核心解决方案

在预警能力建设中，遇到的主要问题包括：

- 平台较多，需要对接多个平台，接入方式需兼容多种数据源。
- 不同预警场景需要不同的预警规则。
- 预警发出后如何接手与跟踪处理数据。
- 如何动态识别与维护新增的预警项。

针对这些问题，制定了通用解决思路并落地多种场景的方案。

### 解决方案概述

在建设各项预警能力中，不同的预警类型有不同特点，因此建设了多种能力以解决不同问题。

#### 平台类配置预警方案

平台类配置预警面临的平台多、数据来源差异大等问题。核心做法：

- 支持多种数据源获取方式（接口、离线表、消息等）。
- 定义通用数据模型，对采集到的数据进行清洗与统一。
- 在平台上自定义预警规则，根据规则发送告警，并利用风险平台的预警接手机制跟进问题。

#### 权益类预警方案

权益类预警是关注重点且复杂的场景，涉及问题包括如何覆盖线上所有权益、如何减少噪声并提高响应效率等。

- 覆盖线上权益：借鉴资金监控的思路，从资金监控的离线表（按业务维度的每日发放红包表）中提取权益 code，即可覆盖线上生效中的全部权益。
- 库存预警改进：在获取权益库存后，利用历史采集数据预测当前库存还能够支撑的时间，从而判断是否需要预警，并在发出工单后由运营接手处理。
- 试验与规则：实验了多种预警规则以提高处理率，包括过期预警与库存预警。

过期预警规则示例：
- 工作日 24 小时内过期预警
- 周五 3 天内过期预警
- 自定义日期筛选过期（如节假日、大促）

库存预警规则探索：
- 低于固定数值后告警
- 总量低于 10% 后告警
- 根据流速告警

流速计算公式示例：
```
库存剩余可用时间 = 今日 x 点的剩余库存 / ( (今日 x 点的剩余库存 - 昨日 x 点剩余库存) / 24 )
如果 库存剩余可用时间 < 24，则触发告警
```

实践发现，24 小时级别的流速预警在实际使用中接手率最高、误报率最低。但若权益在短时间内消耗极快，仍可能无法预警到，因此后续计划建设分钟级的极速消耗预警方案。

#### 配置预警方案

有些开发在配置平台中配置了时间字段，因此时间过期也需预警。核心做法是对指定应用的配置平台进行全量扫描，查找配置中的关键字（如 time、date 等），如果 Key 或 Value 中包含此类字段则解析对应的值并检查是否达到过期告警条件。

#### 实验预警

实验平台通常自带告警能力，但因人员流动较大，不少实验 owner 离职或转岗后，告警无法通知到对应负责人。我们通过扫码本业务相关的实验离线数据表，识别过期或无人接手的实验，并通知到指定的接手人。

#### 舆情预警方案

舆情预警主要通过算法聚类与场景打标实现，利用体验引擎的现有能力，将舆情告警推送到淘宝特价版的舆情预警群内以便快速响应。

## 后续计划

主动预警能力建设是一个持续探索的过程。后续工作重点包括：

- 继续发现更多预警场景，扩展覆盖范围。
- 优化预警准确性，降低误报与漏报。
- 提升预警与处理闭环的效率（例如分钟级极速消耗预警）。
- 加强多数据源对接能力与规则的自动化维护，支持动态识别与新增预警管理。
