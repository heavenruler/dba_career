最近⼀次⾯试中，⾯试官让我分析  MySQL InnoDB 的四种事务隔离级别，以及相关的  MVCC
（多版本并发控制）和  ReadView 机制。还要求顺带讲讲不同隔离级别解决了哪些问题，以及为
什么从  Repeatable Read 切换到  Read Committed 可以降低死锁概率。以下是我的复盘和总
结。
⼀、 MySQL InnoDB 的四种事务隔离级别
MySQL InnoDB ⽀持  SQL 标准定义的四种事务隔离级别，从低到⾼分别是：
1. Read Uncommitted （读未提交）
事务可以读取其他事务尚未提交的数据（即 “ 脏数据 ” ）。
这是隔离级别最低的⼀种，⼏乎不提供隔离性，但在实际⽣产中很少使⽤。
2. Read Committed （读已提交）
事务只能读取其他事务已经提交的数据，解决了 “ 脏读 ” （ Dirty Read ）问题。
但仍存在 “ 不可重复读 ” （ Non-repeatable Read ）问题，即同⼀事务内多次读取同⼀数
据时，可能因为其他事务的提交⽽返回不同的结果。
3. Repeatable Read （可重复读）
保证同⼀事务内多次读取同⼀数据的结果⼀致，解决了不可重复读问题。

4. Serializable （串⾏化）
最⾼隔离级别，通过强制事务串⾏执⾏，避免所有并发问题（如脏读、不可重复读、
幻读）。
但性能开销较⼤，适⽤于对数据⼀致性要求极⾼的场景。
⼆、 MVCC 和  ReadView 的作⽤
1. MVCC （多版本并发控制）
MVCC 是  InnoDB 实现事务隔离的核⼼机制。它通过为每⾏记录维护多个版本（基于事务  ID 和
回滚指针），允许多个事务并发访问数据，⽽⽆需加锁阻塞。 MVCC 的关键点包括：
事务  ID （ trx_id ）：每开启⼀个事务， InnoDB 会分配⼀个唯⼀的事务  ID 。
回滚段（ Undo Log ）：记录数据的旧版本，⽤于回滚和多版本读取。
版本链：通过回滚指针（ roll_pointer ）串联起数据的多个历史版本。
MVCC 的好处是读写分离，读操作⽆需等待写操作完成，从⽽提升并发性能。
2. ReadView
ReadView 是  MVCC 的实现基础，⽤于决定事务能看到哪些数据版本。 ReadView 包含以下关键
信息：
活跃事务列表：记录当前未提交的事务  ID 集合。
up_limit_id：活跃事务列表中最⼩的事务  ID ，表示 “ 最早未提交的事务 ” 。
low_limit_id：当前系统最⼤事务  ID + 1 ，表示 “ 未来事务的起点 ” 。
创建时间（ trx_id ）： ReadView ⽣成时的事务  ID 。
数据可⻅性规则（以下是对规则的详细解释）：
当⼀个事务通过  ReadView 判断某⾏数据的版本（由  trx_id  表示）是否可⻅时，会按以下规
则进⾏：

如果数据的  trx_id  < up_limit_id ：

up_limit_id = 100 ，⽽数据  trx_id = 90 ，表示这个数据在事务  100 开始前已提交，
对当前事务可⻅。
如果数据的  trx_id  在活跃事务列表中：
说明这个数据版本是由⼀个尚未提交的事务创建的，因此不可⻅。例如，活跃事务列表中有
事务  105 ，⽽数据  trx_id = 105 ，表示数据还未提交，不能被当前事务看到。
如果数据的  trx_id  >= low_limit_id ：
说明这个数据版本是在  ReadView 创建之后才由新事务⽣成的（即 “ 未来的数据 ” ），因此不
可⻅。例如，low_limit_id = 110 ，⽽数据  trx_id = 115 ，表示数据是在当前事务视图
⽣成后才提交的，对当前事务不可⻅。
此外，如果  trx_id  >= up_limit_id  但不在活跃事务列表中，且  < low_limit_id ，则表示
数据是在  ReadView 创建前提交的，也可⻅。
在  Read Committed 中，每次  SELECT 都会⽣成⼀个新的  ReadView ，因此能看到最新的已提
交数据。⽽在  Repeatable Read 中，整个事务只⽣成⼀次  ReadView ，保证了数据的⼀致性。
三、事务隔离级别解决的问题
1. Read Uncommitted
问题：脏读（读取未提交数据，可能被回滚）。
未解决：不可重复读、幻读。
2. Read Committed
解决：脏读。
未解决：不可重复读（同⼀事务内数据可能变化）、幻读。
3. Repeatable Read
解决：脏读、不可重复读。
部分解决：幻读（通过  MVCC 和间隙锁在⼤多数情况下避免，但特定场景仍可能发
⽣）。

四、为什么从  Repeatable Read 切换到  Read Committed 能降低死锁概率？
1. Repeatable Read 的锁机制
在  Repeatable Read 下， InnoDB 使⽤  MVCC 结合间隙锁（ Gap Lock ）和下⼀键锁（ Next-Key
Lock ）来防⽌幻读。例如：
当执⾏  SELECT ... FOR UPDATE  或更新操作时，会锁住索引范围（包括间隙）。
如果多个事务同时尝试操作相邻的数据范围，可能导致锁冲突，进⽽引发死锁。
例如，事务  A 更新  id=5  的记录并锁住  (3, 5] ，事务  B 更新  id=4  并试图锁住  (1, 4] ，
两个事务互相等待对⽅的锁释放，形成死锁。
2. Read Committed 的锁机制
在  Read Committed 下：
不使⽤间隙锁，仅对当前记录加⾏锁。
每次读取都会⽣成新的  ReadView ，数据视图更 “ 实时 ” ，但牺牲了重复读的保证。
由于锁的范围更⼩（仅限单⾏，不涉及间隙），锁冲突和死锁的概率显著降低。
3. 为什么死锁概率降低？
锁粒度减⼩： Repeatable Read 的间隙锁范围较⼤，容易与其他事务的锁重叠；⽽  Read
Committed 只锁单⾏，冲突概率低。
锁持有时间缩短： Read Committed 不需要维持整个事务的视图⼀致性，锁释放更快，减少
了锁竞争窗⼝。
4. 权衡
从  Repeatable Read 切换到  Read Committed 虽然降低了死锁概率，但会导致不可重复读问
题。如果业务对⼀致性要求不⾼（如实时统计场景），这种切换是合理的优化选择。
