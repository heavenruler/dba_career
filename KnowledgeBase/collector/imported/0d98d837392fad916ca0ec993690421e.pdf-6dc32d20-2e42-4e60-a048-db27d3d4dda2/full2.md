# 学习 MySQL 必须了解的几个 Undo 概念

作者：操盛春（爱可生技术专家）  
基于 MySQL 8.0.32 源码，存储引擎为 InnoDB。原创内容未经授权不得随意使用，转载请注明来源。

这篇文章基于 MySQL 源码，介绍了 InnoDB 中与 Undo 相关的几个概念。Undo 日志存放在磁盘文件中，其逻辑结构从上到下分为四层：Undo 表空间 → 回滚段 → Undo 段 → Undo 页。文中还阐述了不同情况下 Undo 日志的管理、分配及受哪些系统变量控制，并在最后给出一个关于 Undo 段数量限制的思考题。

关联问题：Undo 页如何分配？回滚段如何选择？Undo 表空间如何截断？

## 1. 引子

事务执行过程中，如果改变了表中数据（插入、更新、删除），会产生 Undo 日志。Undo 日志用于事务回滚，并在 MVCC 中为读取历史版本提供支持。不同操作产生的 Undo 日志用途和清除时机不同，因此 InnoDB 在逻辑上对 Undo 的存放和管理做了分层设计。

接下来从下往上介绍这四层逻辑结构。

## 2. Undo 页

Undo 页是用于存放 Undo 日志的直接容器。Undo 页与存放表数据的页大小相同，默认为 16K。一个事务产生的 Undo 日志可能需要一个或多个 Undo 页来存放，归属于同一个 Undo 段的多个 Undo 页会形成链表。

- 插入操作产生的 Undo 日志：只用于事务回滚，不用于读取记录的历史版本；事务即将提交完成时这些 Undo 可以被直接清除。
- 更新、删除操作产生的 Undo 日志：既用于事务回滚，也用于读取记录的历史版本，必须等到任何事务都不再需要这些历史版本时才可以被清除。

基于清除时机的不同，InnoDB 使用不同类型的 Undo 段来管理插入产生的 Undo 与更新/删除产生的 Undo：
- Insert Undo 段：管理插入产生的 Undo 页。
- Update Undo 段：管理更新与删除产生的 Undo 页。

尽管结构相同，代码中会区分这两类 Undo 段以便在清理策略上有所区分。

## 3. Undo 段

当一个事务产生大量 Undo 日志时，会有多个 Undo 页。为管理这些页，InnoDB 使用一种称为“段”的逻辑结构，管理 Undo 页的段称为 Undo 段。即使事务只产生一个 Undo 页，也会使用 Undo 段来管理，因为在事务提交前无法确定最终需要多少页面。

Undo 段负责管理属于它的 Undo 页链表和元信息，并与回滚段协作完成分配与回收。

## 4. 回滚段

InnoDB 支持大量并发事务执行。为了管理众多的 Undo 段，InnoDB 又引入了回滚段（rollback segment）这一层。回滚段用于管理一定数量的 Undo 段，从而便于分配与调度。

回滚段的结构要点：
- 每个回滚段有一个段首页（segment header），其中包含 1024 个小格子（slots）。
- 每个格子占 4 字节，用于保存一个 Undo 段的段首页页号（page number）。因此，一个回滚段最多可以管理 1024 个 Undo 段。
- 如果某个格子暂时没有对应的 Undo 段（即空闲），该格子存放的值为 4 字节能表示的最大整数，代码中用 `FIL_NULL`（值 4294967295）表示空闲。

回滚段的分配与事务类型有关。根据表中数据的生命周期，用户表分为两类：
- 用户普通表：MySQL 重启之后，表结构和表中数据都存在。对普通表的数据更改必须保证 Undo 日志不丢失（同时产生对应的 Redo），以便重启后回滚未提交事务。
- 用户临时表：MySQL 重启之后，表结构存在但表中数据被丢弃。对临时表的数据更改不需要保证 Undo 持久化，因此不会产生对应的 Redo。

读写事务分配回滚段的三种情况：
1. 事务只改变普通表的数据：分配一个回滚段（普通表类型）。
2. 事务只改变临时表的数据：分配一个回滚段（临时表类型）。
3. 事务既改变普通表又改变临时表的数据：分配两个回滚段（一个普通表回滚段 + 一个临时表回滚段）。

## 5. Undo 表空间

磁盘上用于存放 Undo 日志的文件逻辑上称为 Undo 表空间（undo tablespace）。MySQL 支持多个 Undo 表空间以便在高并发下管理大量回滚段与 Undo 段，避免单一表空间文件无限增长导致无法截断的问题。

重要系统变量与行为：
- `innodb_undo_tablespaces`：控制 Undo 表空间的数量，默认值为 2（最小值，表示至少有 2 个 Undo 表空间），最大值为 127。
- `innodb_rollback_segments`：控制每个 Undo 表空间中回滚段的数量，默认与最大均为 128，最小值为 1。
- `innodb_undo_log_truncate` 与 `innodb_max_undo_log_size`：用于控制 InnoDB 是否自动截断 Undo 表空间并恢复到初始大小，但自动截断的前提是该 Undo 表空间中所有回滚段均处于未使用状态。

如果系统只有单个 Undo 表空间且读写事务不断，就无法满足“全部回滚段未被使用”的条件，从而无法自动截断 Undo 表空间。多个 Undo 表空间能提高截断成功的概率并更好地分散并发负载。

## 6. 总结

- 四层关系：Undo 表空间管理回滚段 → 回滚段管理 Undo 段 → Undo 段管理 Undo 页 → Undo 页管理 Undo 日志。
- InnoDB 支持 2 ~ 127 个 Undo 表空间（`innodb_undo_tablespaces`），每个 Undo 表空间支持 `innodb_rollback_segments` 个回滚段（默认 128），因此总共支持 256 ~ 16256 个回滚段。
- 每个回滚段管理 1024 个 Undo 段，因此总共支持 262,144 ~ 16,646,144 个 Undo 段。
