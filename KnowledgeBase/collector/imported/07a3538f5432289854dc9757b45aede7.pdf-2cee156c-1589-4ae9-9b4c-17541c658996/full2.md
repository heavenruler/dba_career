# 如何精确监控DB响应延时

## 1、背景

日常工作中最常被问的是：“现在应用有点慢，帮忙看看 DB 是不是有问题？”第一反应是查看监控告警。

线上监控指标很多：OS 层包括 CPU、内存、IO、网络；DB 层主要有 TPS/QPS、活跃连接数、锁等待（还有更多细分指标）。即便把所有监控信息放到同一页面挨个查看也要耗费时间。更关键的是，即便上述指标都正常，也不等同于 DB 无恙——这些指标只能算是 DB 健康的必要条件，而不是充分条件。在某些场景下，即便这些指标都很平稳，开发仍可能质疑你，这时如何快速自证清白？

从应用程序角度观察，DB 响应速度 = 网络延时 + 处理延时，其中处理延时指请求抵达 DB 服务器开始到服务器将响应结果发出结束的时间。DB 服务器任何一个环节出现问题，都会增大处理延时。因此，我们只需要监控每个 DB 请求在“进入 DB 服务器”到“DB 响应结束”这段时间的耗时，就能计算每个请求的处理延时，从而判定 DB 服务器是否健康。

tcprstat 是专门为统计处理延时而生的工具。

## 2、原理

通过四元组 (src_ip, src_port, dst_ip, dst_port) 可以唯一标识一个 TCP 连接。对于每个连接，tcprstat 计算其最后一个入包和第一个出包的时间差，以此得出一次请求的处理延时，然后将所有连接的请求处理延时聚合统计并输出。

有个前提：服务端 I/O 模型必须是同步阻塞模式，即当前响应完成后才能接受下一个 request，这一点主流 DB 都符合。

启动时会创建一个 hash 表，默认 2053 个 bucket，每个 bucket 挂载一条单向链表；发生 hash 冲突时，遍历该 bucket 下的链表直至找到匹配的 item。借助 libpcap 捕获数据包，首先还原成 IP 包 (struct ip)，过滤掉非 TCP 包 (ip->ip_p == IPPROTO_TCP)，并去除只包含控制信息的 TCP 包。tcprstat 会记录每个符合条件数据包的时间戳 tv 以及对应的四元组，并对四元组取模以在 hash 表中定位查找。

对应的数据结构很简单：

```c
struct session {
    uint32_t laddr, raddr;
    uint16_t lport, rport;
    struct timeval tv;
    struct session *next;
};
```

如果该包是入包，将其插入到 hash 表；若对应的 item 已存在，覆盖其已有 tv 值。若该包是出包，根据四元组从 hash 表取出对应 item 并将其从 hash 表移除，将两个包的 tv 相减便得出该请求的处理延时。

tcprstat 将每个请求的处理延时保存到一个长整型数组中，每次输出都要对该数组进行遍历，比如计算 avg：

```c
for (i = 0; i < n; i++)
    avg += stats[i];
avg /= n;
```

而计算 99_avg 时，则先对数组调用 qsort() 进行升序排列，并只计算前 99% 的元素，排除最高的 1%：

```c
n = (n * percentile) / 100;
for (i = 0; i < n; i++)
    avg += stats[i];
avg /= n;
```

计算 min 和 max 同理。

以图形化的方式展现，当应用出现性能问题时可快速排查是否由 DB 引发；只有出现剧烈波动时（如 avg/99_avg 明显上升），才能证明 DB 服务器响应有问题。

## 3、安装

安装很简单，直接将二进制文件下载并复制到 /usr/bin：

示例操作（根据实际二进制名调整）：

```bash
cp -a tcprstat-static.v0.3.1.x86_64/usr/bin/tcprstat /usr/bin/
chmod +x /usr/bin/tcprstat
```

该工具默认直接查询机器的网络接口，在 bonding 模式的网卡下会报错，可改用网卡列表：

示例：

```bash
# 直接按端口过滤
tcprstat -p 3306

# 在 bonding 环境下指定网卡或使用 -l 列表
tcprstat -p 3306 -n 0 -t 1 -l `/sbin/ip a | grep inet | egrep 'eth|lo'`
```

示例输出：

## 4、总结

可以得出如下结论：当 tcprstat 的 avg/99_avg 平稳时，可认为 DB 响应是健康的。虽然作者原本是为了监控 MySQL 开发此工具，但对于 Mongo、Redis 同样适用，只需要修改监控端口即可。
