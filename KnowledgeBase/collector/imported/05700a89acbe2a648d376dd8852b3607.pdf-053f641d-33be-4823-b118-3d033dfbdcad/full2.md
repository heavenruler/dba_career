# Redis凭什么用单线程“干翻”了全世界？

大黄花鱼 · 云同学的技术圈 · 2025年8月24日 · 上海 · 原创

Redis（Remote Dictionary Server，远程字典服务）是一个用 C 语言编写的开源高性能内存键值（Key-Value）数据库。在工程实践中你可能有所接触：作为“单线程”应用，Redis 为什么会这么快？内存是一个重要原因，但不是全部。下文将逐项揭晓原因。

需要说明的是，Redis 所说的“单线程”主要指网络 I/O 和实际的 key-value 读写由一个线程完成，但持久化、后台删除、AOF 重写等由额外线程或后台进程执行。并且从 Redis 6.0 开始，网络 I/O 也可以采用多线程，只有命令执行仍由单线程处理。

## 01 “3个因素”造就了 Redis 的高性能

Redis 以其出色的性能和灵活性著称，常用于缓存、会话存储、消息队列、分布式锁等场景。其高性能可以归结为三点：

1. 内存为王 — 物理层面的优势  
   Redis 将数据存放在内存中，这是它与基于磁盘的数据库（如 MySQL、PostgreSQL、MongoDB）的重要区别之一，访问延迟和吞吐率因此有本质优势。

2. 极致的数据结构  
   Redis 对哈希表、跳表等数据结构进行了专门优化，会根据数据大小和类型动态选择最优编码，借助“空间换时间”或“时间换空间”的策略，减少网络开销和客户端压力。

3. 高效的 I/O 模型  
   Redis 采用 I/O 多路复用机制，使得单个线程也能并发处理大量客户端请求，获得高吞吐率。

前两点较易理解，下面重点讨论 Redis 的 I/O 多路复用模型为什么高效。

## 02 为什么要用 I/O 多路复用？

I/O 多路复用（I/O Multiplexing）允许一个进程或线程监视多个文件描述符（FD），当其中任一 FD 准备好进行 I/O（可读、可写或异常）时通知进程。为理解其价值，看看服务器处理大量连接时的三种选择：

1. 阻塞 I/O + 每个连接一个新进程/线程  
   工作方式：每个新连接创建独立进程或线程，调用 `read()`/`recv()` 阻塞等待数据。  
   缺点：资源消耗巨大（每个线程占用内存栈等）、上下文切换频繁，CPU 用于切换的时间增加，实际业务处理时间减少。

2. 非阻塞 I/O + 忙轮询（Busy-Polling）  
   工作方式：每个连接设为非阻塞，主循环不断轮询所有连接“有没有数据”。  
   缺点：CPU 空转严重，即使大部分连接无事件，循环也不停，造成 100% CPU 使用率和巨大浪费。

3. I/O 多路复用  
   工作方式：应用将要监视的一批 FD 注册给内核，调用阻塞函数（如 `select()`、`poll()`、`epoll_wait()`）后进程“睡眠”，由内核负责监视这些 FD。当某些 FD 就绪时，内核唤醒进程并返回就绪列表，应用只处理这些就绪连接。  
   优势：避免了为每个连接创建线程的资源和切换开销，也避免了忙轮询的 CPU 浪费。一个线程就可以高效管理大量连接。

I/O 多路复用通过内核中的“代理”或事件机制，让单线程在高并发场景下仍能高效工作。

## 03 Linux 上的三种 I/O 多路复用实现

Linux 常见的 I/O 多路复用 API 有 `select`、`poll`、`epoll`。

### 1. select
- 特点：最早、遵循 POSIX，移植性好。
- 工作原理：使用 `fd_set`（位图）表示要监视的 FD；调用 `select()` 时内核将 `fd_set` 从用户态拷贝到内核态，内核遍历所有被监视的 FD，设置就绪位，然后将修改后的 `fd_set` 拷回用户态；用户进程再遍历 `fd_set` 找出就绪 FD。
- 缺点：`fd_set` 大小有限（通常 1024）、每次调用需要往返拷贝、内核和用户态都需线性扫描整个集合，时间复杂度 O(n)。

### 2. poll
- 特点：对 `select` 的改进，使用 `pollfd` 数组替代 `fd_set`，数组可以动态分配，不再有 1024 限制。
- 工作原理：类似 `select`，仍需将整个 `pollfd` 数组从用户态拷到内核态，并在返回后遍历数组寻找就绪 FD。
- 缺点：仍有重复拷贝与线性扫描开销，在连接数很大时性能受限。

### 3. epoll（Event Poll）
- 特点：Linux 对 `select`/`poll` 的重大改进，适合高并发网络服务（Nginx、Redis 等使用）。
- 核心函数：
  - `epoll_create()`：在内核中创建 epoll 实例（事件中心），返回代表该实例的 FD。实例内部通常包含用于查找的红黑树和用于存放就绪 FD 的就绪链表。
  - `epoll_ctl()`：向 epoll 实例添加、修改或删除要监视的 FD。注册后内核会记录该 FD 的状态变化。
  - `epoll_wait()`：阻塞等待直到就绪链表非空，返回就绪 FD 列表。
- 优势：无 FD 数量限制；避免每次拷贝注册列表（注册一次后由内核维护）；`epoll_wait()` 直接返回就绪列表，用户进程无需遍历整个集合，时间复杂度接近 O(1)，与监视的总 FD 数无关，只与活跃的 FD 数有关。

## 04 Redis 的 I/O 模型

### 1. Redis 网络 I/O 处理中的核心步骤

以一个 `GET` 请求为例，Redis 的网络 I/O 大致流程如下：

- `bind()`：将套接字与 IP 地址和端口绑定（本地操作，瞬时完成）。
- `listen()`：把套接字设为被动监听状态，内核维护一个待处理连接队列（backlog）。
- `accept()`：从已完成三次握手的连接队列中取出一个连接，创建已连接套接字，用于后续通信。默认是阻塞的，但在高性能服务器会设为非阻塞并与 epoll 配合使用。
- `recv()`：从已连接套接字读取客户端发送的数据，数据从内核空间进入用户空间。
- `parse()`：解析收到的字节流（应用层），根据协议识别请求。
- `get()`：从 Redis 内部数据结构读取对应值。
- `send()`：将响应数据通过套接字发送回客户端，数据从用户空间进入内核并由协议栈发送。

### 2. 非阻塞模式设置（accept、recv、send 的非阻塞特性）

将套接字通过 `fcntl()` 或 `ioctl()` 设置为非阻塞模式后，所有 I/O 操作都变为非阻塞，这对事件驱动模型至关重要。

- `accept()`  
  阻塞模式：如果没有已完成三次握手的连接，`accept()` 会阻塞。  
  非阻塞模式：若无可用连接，`accept()` 立即返回 `EAGAIN`/`EWOULDBLOCK`，应用可以先处理其他事情，稍后再试。通常监听套接字注册到 epoll，只有在 epoll 通知有新连接时才调用 `accept()`，并在循环中取尽所有待处理连接。

- `recv()`  
  阻塞模式：若内核接收缓冲区无数据，`recv()` 阻塞直到数据到达。  
  非阻塞模式：若无数据，`recv()` 立即返回 `EAGAIN`/`EWOULDBLOCK`。在事件驱动模型中，只有在 epoll 通知套接字“可读”时才调用 `recv()`，避免阻塞单线程。

- `send()`  
  阻塞模式：若发送缓冲区已满，`send()` 阻塞直到有空间。  
  非阻塞模式：若缓冲区满，`send()` 立即返回 `EAGAIN`/`EWOULDBLOCK`，或写入部分数据并返回写入的字节数。通常将待发送数据先放入应用层缓冲区，等待 epoll 通知“可写”时再调用 `send()`。

这些非阻塞特性结合 epoll 的事件通知，允许单个线程高效管理大量慢速或并发的客户端连接，而不会被某个慢客户端卡住。

### 3. Redis 的实现要点

Redis 使用 epoll（或其他 I/O 多路复用实现）来管理套接字。内核持续监视已注册的多个套接字（FD），当某些 FD 有事件时将这些事件放入就绪队列。Redis 的主线程从这个事件队列中取出事件并处理，从而避免了持续轮询造成的 CPU 浪费。

事件驱动回调机制的核心是：内核负责检测 FD 状态变化并在就绪时通知应用，应用仅对就绪事件执行相应处理函数。因此，Redis 不必在等待 I/O 上消耗 CPU，可以持续快速地响应大量客户端请求。

## 05 小结

Redis 成为“性能王者”源自三大要素：内存带来的物理优势、精心设计与优化的数据结构，以及高效的 I/O 模型。I/O 模型方面，Redis 借助内核的 epoll 机制，实现事件驱动的单线程处理——无需关注 I/O 等待、避免阻塞与忙轮询，从而在高并发场景下也能快速响应客户端请求，发挥出色的性能。
