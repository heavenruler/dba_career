# TiDB 数据库在某省妇幼业务系统的应用

作者：我是人间不清醒（2024-03-13）

## 业务背景

某省妇幼健康管理系统建设内容包括：妇幼健康信息数据库；妇幼健康信息采集系统、妇幼健康信息管理及分析系统；母子健康管理 APP、妇幼健康管理与分析 APP 等，共 62 个功能模块。

## 原有数据库架构及痛点

我们最初选择 StarRocks 作为分析层，通过 CloudCanal + DataX 的模式实现实时+离线数据同步。随着业务迭代，这套架构逐渐不适应妇幼业务的发展需要。

架构总体上分为四层，自底向上分别是：

- 数据层：源端数据源主要是以 MySQL 为主的关系型数据库。分别搭建 5 套（10 台）主从，分别承接 14 个子库和 1 个总库，MySQL 子库约 700+ 表。
- 处理层：使用 CloudCanal + DataX 进行实时和离线的数据同步。
  - 离线：将报表、大屏、数据交换服务采用离线方式构建 DM 主题数据集市，使用 DataX。
  - 实时：使用 CloudCanal 将 MySQL 数据 1:1 同步到 StarRocks 中。
- 分析层：保存计算好的指标数据以及用于加速查询的中间结果数据。使用 3 台 32C128G 搭建 SR 集群，分别对应报表业务、大屏业务、数据交换服务、数据查询加速。

主要痛点：

- 业务变动频繁：为满足业务需求，最初每周至少有 2 次数据库结构变更，后经协商控制到每周一次。每次变更需要 DBA 对 14 个地市 MySQL、StarRocks 以及数据调度进行调整。
- 服务器资源浪费：出于数据安全考虑 MySQL 采用主从架构，从库资源没有得到有效利用。
- 应用层更新影响业务：研发以微服务架构开发，研发人员只关注自己的模块，通过 SQL 审计平台提交 DDL，最后由 DBA 执行。DBA 对业务不熟悉，无法判断 DDL 是否满足实际业务，导致上线后出现生产异常（测试环境无法复现）。
- 研发与 DBA 的 SQL 调优能力下降：MySQL 数据 1:1 打到 StarRocks 后，复杂查询全部交给 MPP 处理，导致研发人员不再关注 SQL 优化和合理拆分表结构。等到 MPP 瓶颈出现，往往只能通过升级服务器来解决，无法根本性改进。
- DBA 工作量大：按照每周一次的频率，DBA 常在业务低峰（18:00 后）进行 DDL 操作，需维护 30 套数据库、近 20000 张表。若出现问题，回滚操作影响较大。
- 按地市分库导致跨市业务不兼容：报表需通过打宽表做数据库汇总后展示（宽表越来越多），出现重复档案、跨地市无法查询服务记录等问题。
- 无法方便地应用自动化数据库审计平台：数据库分散、操作复杂，自动化实现难度高。

基于以上痛点，我们寻找更优技术架构并进行评估与测试。

## 数据库合并后表统计

- 表合计：792
- 表中超过 10 万条记录的数量：156
- 表中超过 100 万条记录的数量：58
- 表中超过 1000 万条记录的数量：5

我们关注合并后写入和 Online-DDL 操作的保障，先后测试过 PolarDB、TiDB、OceanBase，最终选择了 TiDB。

## 选择 TiDB 的原因（TiDB 特点）

- 高度兼容 MySQL，大多数情况下无需修改代码即可从 MySQL 迁移至 TiDB，即使是已做分库分表的 MySQL 集群也可通过 TiDB 提供的迁移工具进行实时迁移。
- 水平弹性扩展：通过增加新节点即可实现水平扩展，按需扩展吞吐或存储，应对高并发、海量数据场景。
- 分布式事务：TiDB 100% 支持标准的 ACID 事务。
- 金融级高可用：基于 Raft 的多数派选举协议提供强一致性，在多数副本存在的前提下实现自动故障恢复（auto-failover），无需人工介入。

## TiDB 数据库架构介绍

### 新数据库架构

新架构去掉了 StarRocks 分析层，发挥 TiDB 的 TiFlash 组件的 OLAP 能力，数据交换仍使用 CloudCanal。分析层和业务层进行合并，业务即宽表，统一通过 TiFlash 进行加速。在运维成本和业务扩展上提升明显。

引入 Yearning 自动化 SQL 审计平台以实现自动化 DDL 审核与执行。

### Yearning 自动化 SQL 审计平台功能

Yearning 具有以下功能：

- 自动化 SQL 语句审核，可对 SQL 进行自动检测并执行
- DDL/DML 执行后自动生成回滚语句
- 审核/查询审计功能
- 支持 LDAP 登录、钉钉及邮件消息推送
- 支持自定义审核工作流
- 支持细粒度权限分配

## 关于 SQL 优化

DBA 对监测到的慢查询逐一记录、跟踪反馈并持续优化。通过 SQL 优化后，多数查询性能基本符合预期。

## 删库、删表恢复示例

恢复被 DROP 删除的 test 数据库：
```sql
DROP DATABASE test;
FLASHBACK DATABASE test;
```

恢复被 DROP 删除的 test 数据库并重命名为 test1：
```sql
DROP DATABASE test;
FLASHBACK DATABASE test TO test1;
```

恢复被 DROP 删除的表数据：
```sql
DROP TABLE t;
FLASHBACK TABLE t;
```

恢复被 TRUNCATE 的表数据：由于被 TRUNCATE 的表还存在，恢复后需要重命名被恢复的表，否则会报错表 t 已存在。
```sql
TRUNCATE TABLE t;
FLASHBACK TABLE t TO t1;
```

## 备份/还原应用

TiDB 提供 BR（Backup & Restore）集群快照备份功能，可将快照备份到对象存储（例如 MinIO）。当前某省妇幼采取每天两次快照（0 时、12 时），由于存储限制，临时只保留一天内的快照，尚未实现日志备份。全量快照 + 实时日志备份可以保证数据不丢失。

BR 的还原速度表现良好：300G 数据在 v7.1.3 上还原用时约 20 分钟；官方在后续版本（如 v7.6）对 BR 的还原能力有显著提升。

## 一地两中心尝鲜

考虑到妇幼数据的重要性，在政务云上实施一地两中心。通过 TiCDC 实现主集群实时将数据同步到从集群，从集群承担报表业务以及研发测试库环境的负载。这降低主库风险并满足业务与合规需求。

## pt-kill 应用

pt-kill 用于捕获正在运行且消耗资源过多的 SELECT/ALTER 等查询，过滤后可以选择性地杀死这些查询，并通过邮件/微信报警给 DBA 和相关开发，避免慢 SQL 长时间运行对数据库造成影响。示例后台运行命令：

```bash
nohup php pt-kill.php -u username -p pwd -h 10.10.159.0 -P 3306 -B 10 -I 15 --match-info='select|alter' --match-user='dbuser' --kill --mail --weixin --daemon 1 &
```

被杀的慢语句会在后台日志文件中保留，后续可据此对 SQL 进行优化。

## 新架构效果说明

### 服务器资源合理利用

对比原架构与新架构的资源分配：

- 原架构：数据层 10 台、处理层 2 台、分析层 3 台
- 新架构：数据层 11 台、处理层 1 台、分析层 0 台

通过整合与 TiDB 的扩展能力，服务器资源得到更合理利用。

### 人力成本释放

原架构中，DBA 在数据层和处理层需根据发布周期对数据库进行 DDL 操作、调整和调度。新架构省去了部分调度维护工作，并引入 SQL 审计平台实现自动化 DDL。但 DBA 需更多关注 TiDB 的各项指标和集群运维。

### 运维成本降低

TiDB 部署不依赖大数据组件，部署和运维相对简单。TiDB 兼容 MySQL 生态，业务系统可以直接使用 MySQL JDBC 连接，减少 SQL 语法差异带来的迁移成本。

## 未来规划

目前我们保留两套数据架构：MySQL + StarRocks 和 TiDB。这两套架构各有优势，也可以结合使用。将根据事业部需求和不同业务线的特性决定采用哪套架构或混合方案。
