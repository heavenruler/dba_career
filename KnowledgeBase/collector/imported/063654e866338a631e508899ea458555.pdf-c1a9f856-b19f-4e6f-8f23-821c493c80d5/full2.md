# 别再让IT背锅了！数据质量的第一责任人是业务

作者：张志明

## 前言
在数字化转型领域，好书不多，《华为数据之道》绝对是一本。华为是传统企业做数字化转型最成功的案例之一，这本书由华为的数据管理部撰写，干货满满。

但这本书很奇怪，它花费了极多的篇幅阐述一些明显会过时的具体技术，比如数据湖和数据地图，却把真正的精华用寥寥数笔带过。

## 全书最重要的几句话
- 数据工作的责任主体（Data Owner）是业务部门。  
- 数据是业务运营的产物，数据的质量应由产生数据的业务部门负责。  
- IT 部门是“赋能者”和“平台提供者”，负责提供工具、平台和专业知识，协助业务部门履行其数据责任，而非替业务部门承担责任。

### 权责利统一的管理原则
从管理上说，制度设计的一个原则就是：权、责、利统一。业务部门是数据的使用者，同时也是受益者，所以要对数据质量负责。

## 事例与问责
某公司销售部门委托技术部研发了销售提成的计算平台。由于对提成规则理解不一致，导致给销售发放的提成超出标准，给公司造成了损失。公司要对该事件进行追责，哪个部门应当承担主要责任？

- 如果由技术部承担主要责任：  
  既然类似的事情再次发生，被问责的还是技术部，那么销售部门对于需求的态度可能还是写一个大概，然后假定具体的细节“属于常识”。

- 如果由业务部门承担主要责任：  
  作为系统的需求方，业务部门具有拒绝质量不合格的数据通过验收的权力；业务部门有权要求 IT 部门作为建设方对质量问题进行整改；业务部门有权对 IT 部门的交付质量进行评价；业务部门有权要求 IT 部门提供相应的数据质量监控工具，帮助其履行职责。

## IT 部门责任制的弊端
将数据质量责任全部落在 IT 部门，会产生多方面的负面影响：

- 业务部门将自己定位为“提需求”和“等报表”的用户。  
- 由于无需为错误后果负责，业务部门提交的需求质量会模糊、随意并频繁变更。  
- IT 部门沦为背锅侠和救火队，工作重心从“数字化赋能”转向“自证清白”。  
- IT 部门被迫投入大量精力去“理解”原本应该由业务部门明确定义的规则。  
- 制度往往决定结果；从主观上我相信业务部门同样想要改进数据质量，但不合理的责任划分会阻碍改进。

## 合理的职责划分与复盘机制
合理的职责划分和复盘改进方式应包括：

- 业务部门负责“做什么”和“对不对”：撰写准确的业务规则，并验证交付物是否符合需求。  
- IT 部门负责“怎么做”和“快不快”：严格按照规则进行代码开发。  
- 如果业务部门在验收环节发现问题，应立刻拒绝接收并要求 IT 部门进行整改。  
- 如果在验收环节没有发现问题，而将有问题的数据应用到了业务中，业务部门首先承担主要责任。  
- 在之后的复盘中，必须定位出问题的详细环节，并做出相应的整改。

在这个机制下，业务部门会认真梳理业务规则，IT 部门则处于业务部门的考核之下，也会努力提升代码质量和优化平台工具。

## 结论与建议
如果你的组织对数字化转型持严肃态度，强烈建议学习华为的制度：业务部门是数据质量的第一责任人。这并不是出于我个人“技术人”的偏见，而是经过实践检验、切实有效的制度设计。

你是否同意这个观点？欢迎读者在组织内部讨论并推动相应的制度建设。
