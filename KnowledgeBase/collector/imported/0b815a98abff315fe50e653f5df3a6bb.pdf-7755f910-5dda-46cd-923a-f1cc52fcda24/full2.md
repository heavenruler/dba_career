## 1. MySQL 慢的诊断思路

一般从三个方向来诊断 MySQL 慢的问题：
- MySQL 内部的观测
- 外部资源的观测
- 外部需求的改造

### 1.1 MySQL 内部观测
常用的内部观测手段从上到下包括：
- PROCESSLIST：查看哪些 SQL 压力不正常
- EXPLAIN：分析执行计划
- Profiling：如果能复现，执行 profiling
- performance_schema / sys_schema（MySQL 5.7+）：使用视图快速获取各种信息
- innodb_metrics：更深入的引擎诊断

这些是常规的 MySQL 内部状态观测思路，MySQL 也提供了一些更底层的观测方案，但学习成本较高，未必在实践中广泛形成套路。

### 1.2 外部资源观测
参考“60 秒快速巡检”的十条常用命令可快速排查服务器外部资源：
前五条（查看总体与进程维度）：
1. uptime：机器运行时间与平均负载  
2. dmesg -T | tail：系统日志报错  
3. vmstat 1：虚拟内存、页面换入/换出、swap 使用  
4. mpstat -P ALL：各 CPU 核心压力是否均匀  
5. pidstat 1：各进程资源占用情况

后五条（IO、内存、网络与进程概览）：
6. iostat -xz 1：IO 统计  
7. free -m：内存使用  
8. sar（按设备维度）：设备/网卡等消耗状况  
9. sar（TCP 维度）：TCP 使用率与错误率  
10. top：进程/线程概览

这十条命令覆盖了服务器常见的外部资源问题。

### 1.3 外部需求改造
有时问题出在需求/SQL 的写法上。MySQL 官方文档有“Examples of Common Queries”一章，展示如何把高代价的关联子查询改写为更高效的非关联查询。背后的理论是关系代数：SQL 能被等价地转换以降低执行成本。因此，如果诊断得出 SQL 成本太高，可以考虑改造需求或改写 SQL，使其更容易被数据库高效执行。

总结：
1. 先用 MySQL 提供的观察手段自上而下诊断 SQL 或服务器问题。  
2. 从服务器的角度，按巡检脚本检查有限的资源类型即可。  
3. 如果实在解决不了，建议让需求方按数据库更易接受的方式重写 SQL，往往能快速降低成本。

## 2. 系统观测工具介绍

介绍什么是系统观测工具、数据来源、采集过程和如何展示数据（图或表）。

三个方面：
1. 数据源：Linux 提供的内核态与用户态观测点。MySQL 早期也提供了一些用户态观测点。  
2. 数据采集：常见工具包括 perf、ftrace、sysdig 等，从操作系统中抽取数据。  
3. 数据处理前端：如何呈现和交互，比如 perf 输出的是树形数据，可交互查看函数耗时等。

对比四类常见系统观测工具（ftrace、perf、eBPF、SystemTap）：
- ftrace：通过 sysfs 的“桩”将需要观测的函数签名写入，内核提供函数运行状况。采集端与处理端基于 ftrace。
- perf：通过系统调用将数据写到内核缓冲区，客户端抽取并展示，常用于函数级性能分析（on-CPU）。
- eBPF（今天重点介绍）：将一段代码作为可安全执行的程序插入到操作系统内核中的某个位置运行，运行时采集数据并放到内核空间供用户态读取。与前两种方案不同，它可以将自定义小程序插到内核以实现灵活高效的观测。
- SystemTap：将 C 代码编译为内核模块并加载到内核，侵入性较强但功能强大。

侵入性与系统影响比较：
- 系统调用（perf）伤害最轻（系统承诺的服务）。  
- ftrace 次之（文件系统层的交互口）。  
- eBPF 侵入性更强（将代码嵌入内核运行，但有安全保护）。  
- SystemTap 侵入性最大（加载内核模块，较不稳定）。

eBPF 架构概述：eBPF 程序先编译为字节码，插入内核指定位置，内核执行并将采集到的数据放到内核空间，用户态程序读取并展示。关键是内核对插入代码的安全校验和保护，这也是其复杂之处。

## 3. bcc（eBPF 脚本集）使用举例

bcc 是一个开源的 eBPF 脚本集，提供许多开箱即用的脚本，能帮助快速诊断 MySQL 与系统层面的问题。下面列出十个常用例子与说明。

1. MySQL 请求延迟分析  
   - 输出通常包含三列：延迟区间（指数级增长，单位微秒）、命中次数、分布图（图形与数值）。  
   - 可看到请求分布，识别单峰、双峰情形（例如缓存命中与未命中形成两个峰）。  
   - 可用于环比分析（今天 vs 昨天同一时间的延迟分布），用于监控延迟是否在变差。

2. MySQL 的慢查询（采集方式不同于 MySQL 慢日志）  
   - 可以低成本地获取少量慢查询、特定模式的慢查询或某个用户的慢查询。  
   - 适用于线上环境只需要精准抓取若干慢查询样本或多租户场景下针对单租户分析。

接下来几个例子涉及 IO 堆栈分析，基于 Linux IO 堆栈（用户态 → VFS → 文件系统 → 块设备 → 调度器 → SCSI → 物理设备）：

3. VFS 延迟分析  
   - 可用于分析数据库写压力在 VFS 层的延迟分布，常见双峰代表命中文件系统缓存与直接写穿设备两类延迟。

4. Ext4 文件 IO 延迟分析（按文件维度）  
   - 可以看到哪个文件（如数据文件或日志文件）的 IO 延迟高。日志通常为顺序写，延迟低；数据文件通常随机写，延迟高。

5. 块设备延迟分析（直接下到设备）  
   - 去除操作系统缓存影响，查看真实下到设备的延迟分布。判断设备是否需要更换时，应基于环比或统计样本量而非单次绝对值。

6. MySQL 线程对文件的 IO 压力汇总（按线程统计）  
   - 输出包含线程 ID (TID)、写入大小、文件名等，便于发现是哪个线程造成 IO 压力（例如由 general_log 导致大量写操作）。

7. 短生命周期的临时文件检测  
   - MySQL 在某些情况下会创建大量短生命周期的临时表（外存临时表），这些短命文件对 IO 造成短时但显著的压力。脚本可检测并展示此类临时文件实例。

8. 短连接分析  
   - 在没有连接池或业务频繁建立短连接的情况下，短连接会给数据库带来大量连接/断开压力。脚本可定位短连接来源（服务器、端口），便于处理。

9. 长连接分析  
   - 分析那些长时间保持连接并持续读写的客户端，帮助发现某些业务端对数据库的持续负载来源。

10. CPU off-CPU 消耗分析（offcpu）  
    - on-CPU 表示线程占用 CPU 执行；off-CPU 表示线程被阻塞（IO、睡眠、锁等）而不占 CPU。off-CPU 分析能帮助理解线程为何被阻塞以及持续时长。  
    - 输出通常为“冷火焰图”（off-CPU 的 flame graph），由采样的调用栈合并产生。火焰图可帮助识别占用时间较多的调用栈（例如某个索引读取路径占了 25%），从而判断是否通过更换磁盘、扩展 buffer pool 等措施提高性能。

bcc 提供了一整套观测操作系统各方面的脚本（包括 OOM、内存泄露检测等），是实战中的宝库，很多分析可以直接使用这些脚本完成。bcc 底层使用的是 eBPF。

## 4. eBPF 使用方法与限制

如果 bcc 脚本不能满足需求，可以自写脚本。一般需要两段程序：
- 内核态的 eBPF 程序：插入内核观测点，记录事件（如记录开始时间戳和结束时间戳，计算延迟并保存到统计容器）。
- 用户态的外壳程序：负责加载/插入内核程序到特定的观测点，并从内核读取/打印结果。

示例：实现 MySQL 延迟分布统计需要在查询开始和结束处分别记录时间戳，计算差值并统计延迟。内核态程序负责时间戳的记录与统计，用户态程序负责加载与输出。实际脚本核心部分很短（几十行），错误处理等部分会增加行数，通常可以基于现有脚本改造实现所需功能。

主要限制：
1. Linux 内核版本限制：eBPF 在 Linux 4.4 引入，但早期内核有统计 bug，建议使用 Linux 4.9+，部分高级功能在 4.13+ 才可用。换言之，需要较新的内核版本才能在生产环境稳定使用部分功能。
2. MySQL 编译参数限制：MySQL 早期提供了 dtrace 等观测点，但官方发布的二进制包往往未开启这些观测点编译，因此有时需要自行编译 MySQL 并开启相应编译选项才能利用某些观测点。
3. 若无法满足上述要求，可考虑 SystemTap：SystemTap 更早（Linux 2.6），通过内核模块实现观测，功能全面但相对不太稳定。SystemTap 为稳定性加入了若干限制（内核中可用内存、采集频率、性能影响阈值等），在关闭这些限制以支持更高级功能时会影响内核稳定性，因此 SystemTap 更适合测试环境而非生产环境。

除此之外，市面上还有许多其他观测工具，选择时需根据需求、环境与稳定性权衡并尝试验证。
