## 1. 什么是成本

我们知道，查询会选择成本最低、或代价最低的方式去真正执行。MySQL 的查询成本分为两个部分：

- I/O 成本  
  表使用的存储引擎（如 MyISAM、InnoDB）通常将数据和索引存储到磁盘上。当查询表中的记录时，需要把数据或索引加载到内存，这个从磁盘到内存的加载过程损耗的时间称为 I/O 成本。

- CPU 成本  
  读取以及检测记录是否满足对应搜索条件、对结果集进行排序等操作的时间称为 CPU 成本。

对于 InnoDB 存储引擎来说，页（page）是磁盘和内存之间交互的基本单位。MySQL 规定默认成本常数，例如读取一个页面默认是 1.0，读取以及检测一条记录是否符合搜索条件的成本在某版本中默认是 0.1（不同版本的默认值有所调整）。不管读取记录时需不需要检测是否满足搜索条件，访问一条记录都计为访问一页的常数代价。

### 1.1 如何查看 MySQL 的成本常数

通过查询 mysql.server_cost、mysql.engine_cost 表可以查看成本常数（需要相应权限）：

示例：
```sql
SELECT * FROM mysql.server_cost;
SELECT * FROM mysql.engine_cost;
```

常见的一些成本常数（示例为 MySQL 5.7.22 测测值）：

- server_cost:
  - disk_temptable_create_cost: 20.0（测出 40.0） — 创建基于磁盘的临时表的成本。
  - disk_temptable_row_cost: 0.5（测出 1.0） — 向基于磁盘的临时表写入或读取一条记录的成本。
  - key_compare_cost: 0.05（测出 0.1） — 两条记录做比较的成本（排序时常用）。
  - memory_temptable_create_cost: 1.0（测出 2.0） — 创建基于内存的临时表的成本。
  - memory_temptable_row_cost: 0.1（测出 0.2） — 向基于内存的临时表写入或读取一条记录的成本。
  - row_evaluate_cost: 0.1（测出 0.2） — 检测一条记录是否符合搜索条件的成本。

- engine_cost:
  - io_block_read_cost: 1.0 — 从磁盘上读取一个块（页）的成本。
  - memory_block_read_cost: 0.25（或 1.0 in some versions） — 从内存中读取一个块的成本。

注意：不同版本（例如 8.0+）对成本常数的设定和测算会改进，因此在不同 MySQL 版本查看执行计划时，选择的实际索引可能有所不同。

## 2. 单表查询的成本

下面基于一个示例表分析单表查询成本和优化器为何选择某个索引。

示例建表（用于演示）：
```sql
CREATE TABLE demo_info (
  id INT NOT NULL AUTO_INCREMENT,
  key1 VARCHAR(100),
  key2 INT,
  key3 VARCHAR(100),
  key_part1 VARCHAR(100),
  key_part2 VARCHAR(100),
  key_part3 VARCHAR(100),
  common_field VARCHAR(100),
  PRIMARY KEY (id),
  KEY idx_key1 (key1),
  UNIQUE KEY uk_key2 (key2)
);
```

### 2.1 基于成本的优化步骤

在真正执行一条单表查询语句之前，优化器会找出执行该语句所有可能使用的方案，并对比后找出成本最低的方案（执行计划）。大致步骤如下：

1. 根据搜索条件，找出所有可能使用的索引（possible_keys）。
2. 计算全表扫描的代价。
3. 计算使用不同索引执行查询的代价（先分析唯一索引，再普通索引）。
4. 对比各种执行方案的代价，选出成本最低的方案。

下面通过具体 SQL 示例分析优化器如何计算成本：

示例查询：
```sql
EXPLAIN
SELECT * FROM demo_info
WHERE key1 IN ('a','b','c')
  AND key2 > 10 AND key2 < 1000
  AND key3 > key2
  AND key_part1 LIKE '%hello%'
  AND common_field = '123';
```

分析步骤与结论如下。

1. 根据搜索条件，找出所有可能使用的索引（possible_keys）

- key1 IN (...) 可以使用普通索引 idx_key1。
- key2 范围条件可以使用唯一索引 uk_key2。
- key3 > key2 因为没有常数比较，不能建立合适的扫描区间，不能用索引。
- key_part1 LIKE '%hello%' 因为以通配符开头，不能使用索引。
- common_field = '123' 如果没有索引则不会用到索引。

所以可能使用的索引有：idx_key1、uk_key2。

2. 计算全表扫描的代价

全表扫描需要把聚集索引（InnoDB 的主键聚簇索引）对应的页加载到内存，然后检测每条记录是否符合搜索条件。计算全表扫描代价需要：

- 聚集索引占用的页面数（pages）
- 该表中的记录数（rows）

可以通过 `SHOW TABLE STATUS LIKE 'demo_info'` 查看这些统计信息。示例中查询结果（示例值）：

- Rows（估计）: 100548（注意 InnoDB 的 Rows 是估计值）
- Data_length: 5783552 字节（示例），页面大小默认 16KB，因此聚簇索引页面数约为 5783552 ÷ 16KB ≈ 353 页。

使用成本常数（示例）计算：

- I/O 成本 = 353 × 1.0 + 1.1 = 354.1
 （353 为聚簇索引页数，1.0 为读取一个页面的成本常数，1.1 为微调值）
- CPU 成本 = 100548 × 0.1 + 1.0 = 10055.8
 （100548 为 Rows 估计值，0.1 为访问一条记录的 CPU 成本常数）
- 总成本 = 354.1 + 10055.8 = 10409.9

因此，全表扫描总成本约为 10409.9（示例值）。

3. 计算使用不同索引执行查询的代价

优化器会分别计算使用 uk_key2（唯一二级索引）和 idx_key1（普通索引）的成本，并考虑是否可能使用索引合并（Index Merge）。

先分析使用 uk_key2（key2 范围索引）：

- 搜索条件：key2 > 10 AND key2 < 1000 → 区间 (10, 1000)
- 非聚集索引扫描区间数量：1（即一个范围区间）
- 优化器粗暴地认为读取索引的一个扫描区间的成本相当于读取一个页面的成本：1 × 1.0 = 1.0（I/O 成本）
- 需要回表（查主键并回聚集索引）的记录数：通过统计/估算得出该区间有 883 条二级索引记录（示例）。
  - 读取这 883 条二级索引记录的 CPU 成本 = 883 × 0.1 + 0.01 = 88.31（0.01 为微调）
  - 回表 I/O 成本（回聚簇索引） = 883 × 1.0 = 883.0（优化器认为每条回表相当于一次页面访问）
  - 回表后检测其它搜索条件（如 key3 > key2、key_part1 LIKE '%hello%'、common_field = '123'）的 CPU 成本 = 883 × 0.1 = 88.3

合计使用 uk_key2 的成本：
- I/O 成本 = 1.0 + 883.0 = 884.0
- CPU 成本 = 88.31 + 88.3 = 176.61
- 总成本 ≈ 1060.61

使用 idx_key1 的分析（key1 IN ('a','b','c')，3 个单点区间）：

- 扫描区间数量 = 3 → I/O 成本读取二级索引 = 3 × 1.0 = 3.0
- 每个单点区间对应的二级索引记录数示例：
  - 'a' 对应 99861 条
  - 'b' 对应 0 条
  - 'c' 对应 1 条
  - 总计回表记录数 = 99861 + 0 + 1 = 99862
- 读取这些二级索引记录的 CPU 成本 = 99862 × 0.1 + 0.01 = 9986.21
- 回表 I/O 成本 = 99862 × 1.0 = 99862.0
- 回表后检测其它搜索条件的 CPU 成本 = 99862 × 0.1 = 9986.2

合计使用 idx_key1 的成本：
- I/O 成本 = 3.0 + 99862.0 = 99865.0
- CPU 成本 = 9986.21 + 9986.2 = 19972.41
- 总成本 ≈ 119837.41

比较三种方案：
- 全表扫描: 10409.9
- 使用 uk_key2: 1060.61
- 使用 idx_key1: 119837.41

显然优化器选择使用 uk_key2 执行（总成本最低且远低于其他方案）。

注意点：当采用索引扫描（非聚集索引 + 回表）时，优化器在估算成本时往往将每次回表视为一次页面访问（即每条回表记录相当于读一个页面）。另外在某些访问方法上，优化器对回表成本设置了上限，防止回表成本过高。

如何验证？可以用 EXPLAIN FORMAT=JSON 来查看优化器估算的 query_cost，例如：
```sql
EXPLAIN FORMAT=JSON
SELECT * FROM demo_info
WHERE key1 IN ('a','b','c')
  AND key2 > 10 AND key2 < 1000
  AND key3 > key2
  AND key_part1 LIKE '%hello%'
  AND common_field = '123';
```
返回的 JSON 中会包含 cost_info.query_cost 字段，可与手动计算做对比。

## 3. 连接查询的成本（作为开发人员必看）

### 3.1 连接查询的准备

为了讨论连接查询的成本，我们构造一个与 demo_info 完全相同结构的表 demo_info2。为方便说明，称 demo_info 为 d1，demo_info2 为 d2。

### 3.2 条件过滤（Condition filtering）

MySQL 的嵌套循环连接算法中，驱动表（outer）只访问一次，被驱动表（inner）可能访问多次。因此两表连接查询的总成本由两部分构成：

- 单次访问驱动表的成本
- 多次访问被驱动表的成本（每查询到一条驱动表记录，就会去查询被驱动表，查询次数取决于驱动表结果集的条数）

我们把驱动表的结果集条数称为驱动表的扇出（fanout）。扇出越小，被驱动表被访问的次数越少，总成本越低。因此优化器在计算连接查询成本时，需要估算驱动表的扇出值。扇出估算有时很容易（如没有过滤条件或只用索引区间能精确计算），有时则需要“猜测”，这就是所谓的 condition filtering 或启发式估算。

常见情形：

- 如果驱动表采用全表扫描，优化器需要猜测满足全部搜索条件的记录数。
- 如果驱动表采用索引扫描，优化器需要猜测在索引扫描区间内满足剩余其他条件的记录数（即 condition filtering）。

启发式规则（heuristic）和统计信息都会用于这类估算。

### 3.3 两表连接的成本分析（对开发人员写 SQL 很重要）

连接查询总成本的公式：
连接查询总成本 = 单次访问驱动表的成本 + 驱动表扇出 × 单次访问被驱动表的成本

对于外连接（LEFT/RIGHT JOIN），驱动表位置固定（LEFT JOIN 的驱动表是左表），只需为驱动表和被驱动表分别选择成本最低的访问方法即可。

对于内连接，驱动表和被驱动表可互换，因此优化器需要考虑不同的连接顺序并选择成本最低的那一种。通常计算内连接成本更复杂：要为每种可能的驱动表选择最优访问方法，并计算驱动表扇出和被驱动表访问成本。

示例 SQL：
```sql
SELECT * FROM demo_info AS d1
INNER JOIN demo_info2 AS d2
ON d1.key1 = d2.common_field
WHERE d1.key2 > 10 AND d1.key2 < 1000
  AND d2.key2 > 1000 AND d2.key2 < 2000;
```

可选连接顺序：
- d1 → d2（d1 作为驱动表）
- d2 → d1（d2 作为驱动表）

优化器会分别评估两种顺序下的最优访问成本，然后选取成本更低的顺序与访问方法。

要点总结：
- 连接查询成本的“大头”通常是：驱动表扇出 × 单次访问被驱动表的成本
- 优化目标是：尽量减小驱动表扇出（将小表或经过过滤变小的表放为驱动表）；尽量降低被驱动表的访问成本（为被驱动表的连接列建立合适索引）
- 被驱动表的连接列最好是主键或唯一索引，这样被驱动表可以用高效的 ref/eq_ref 访问方法，显著降低访问成本
- 判定“大表/小表”时应基于条件过滤后的记录数，而不是表的总记录数

### 3.4 多表连接的成本分析（选读）

多表连接有多种可能的连接顺序。表的数量越多，可能的顺序数呈阶乘增长（n!）。MySQL 优化器采用若干策略来减少必须评估的连接顺序数量：

- 提前结束某种顺序的成本评估：在搜索过程中维护当前最小连接成本，如果正在评估的顺序成本已超过当前最小值，就中止该顺序的进一步评估。
- 系统变量 optimizer_search_depth：用于控制在表数量较多时是否对全部可能顺序进行穷举。若表数小于该值则可穷举，否则只对指定数量的表进行穷举。
- 启发式规则（optimizer_prune_level）：某些连接顺序因不满足启发式规则而直接忽略，减少分析工作量（但可能错失最优计划）。

这些机制权衡了查询优化时间与生成优良执行计划之间的关系。

### 3.5 外连接和内连接等同的情况（外连接可转为内连接）

在某些情况下，外连接（LEFT/RIGHT JOIN）与内连接（INNER JOIN）等价，可相互转换。如果在 JOIN 条件或 WHERE 子句中指定被驱动表的某列为 NOT NULL（或其他明确排除 NULL 的条件），则外连接可以转换为内连接，从而扩展优化器可考虑的连接顺序。

示例：
```sql
-- LEFT JOIN，但 WHERE 中排除了被驱动表的 NULL
SELECT * FROM demo_info AS d1
LEFT JOIN demo_info2 AS d2
  ON d1.key1 = d2.common_field
WHERE d2.key_part2 IS NOT NULL;

-- 等价于：
SELECT * FROM demo_info AS d1
JOIN demo_info2 AS d2
  ON d1.key1 = d2.common_field
WHERE d2.key_part2 IS NOT NULL;
```

这种“空值拒绝”（null-rejecting）条件允许优化器把外连接当作内连接处理，从而在评估表连接顺序时有更多优化空间。
