别再纠结线程池大小/线程数量了，没有固定公式
================================

作者：京东保险 蒋信  
来源：京东云开发者社区

摘要：讨论线程数、线程池大小与 CPU 利用率的关系。通过简单测试与分析，说明没有放之四海而皆准的固定公式，给出理解方法与实践流程，并提供一个常用的线程数估算公式及其局限性。

线程数和 CPU 利用率的小测试
------------------

先说一个基本的直观理论（不深入操作系统或计算机原理，仅为便于理解）：

- 一个 CPU 核心在单位时间内只能执行一个线程的指令。
- 理论上，一个“极端”的线程（不停执行计算指令）就能把一个核心跑满。

下面通过几个 Java 小程序来测试不同场景下的 CPU 利用率。

示例 1：纯死循环（CPU 密集）
```java
public class CPUUtilizationTest {
    public static void main(String[] args) {
        // 死循环，什么都不做
        while (true) {
        }
    }
}
```
运行后可以看到某个核心被跑满。将线程数增加到与逻辑核心数相当（例如 12 个线程），可以把所有核心跑满。如果继续增加到 24 个线程，CPU 利用率仍然是 100%，但系统 load average 上升（说明上下文切换更多，CPU 更“繁忙”且线程任务无法及时执行）。

解释：当同时运行的线程数超过核心数时，操作系统通过时间片调度多个线程共享核心。频繁的线程切换会带来代价（寄存器保存/恢复、页表切换等），如果切换开销过大，会浪费大量 CPU 时间在上下文切换而非实际执行。

示例 2：含模拟 I/O 的线程
```java
public class CPUUtilizationTest {
    public static void main(String[] args) throws InterruptedException {
        for (int n = 0; n < 1; n++) {
            new Thread(new Runnable() {
                @Override
                public void run() {
                    while (true) {
                        // 每次空循环 1亿 次后，sleep 50ms，模拟 I/O 等待、切换
                        for (int i = 0; i < 100_000_000L; i++) {
                        }
                        try {
                            Thread.sleep(50);
                        } catch (InterruptedException e) {
                            e.printStackTrace();
                        }
                    }
                }
            }).start();
        }
    }
}
```
单线程运行时，某个核心的利用率约 50%（因为有 50ms 的 sleep）。将线程数调到 12 个后，每个核心约 60% 利用率；调到 18 个后，单核心利用率接近 100%。说明：当线程中有 I/O 或等待时，CPU 空闲时间会被操作系统用于执行其他线程，从而并行执行更多线程。

如果将计算量减半（将循环次数从 100_000_000 降为 50_000_000），在相同线程数下每个核心利用率会下降（例如降到 ~70%），这进一步表明 I/O/等待时间越多，CPU 空闲越多，可以并发更多线程。

线程数和 CPU 利用率的小总结
------------------

- 一个极端（纯计算）线程就能把单个核心跑满。多核 CPU 最多能同时执行等于核心数的“极端”线程。
- 如果每个线程都很“霸道”，且同时执行的线程数超过核心数，会导致频繁切换，性能反而下降。
- 含 I/O 或其他等待的线程在等待期间不占用 CPU，可以让操作系统调度 CPU 执行其他线程，从而提高整体 CPU 利用率并允许更多并发线程。
- I/O 事件越频繁或等待时间越长，CPU 的空闲时间越多，能够并发的线程数也越多。

线程数规划的公式
----------------

《Java 并发编程实战》给出一个常用的估算公式：

N_threads = N_cores * U_target * (1 + W / C)

其中：
- N_threads：需要的线程数
- N_cores：CPU 逻辑核心数
- U_target：目标 CPU 利用率（0~1）
- W：平均等待时间（例如 I/O 等待）  
- C：平均计算时间（占用 CPU 的时间）

例如：12 个逻辑核心，目标利用率 0.9，假设每次循环计算耗时与等待/睡眠时间相同（C = 50ms，W = 50ms）：
N_threads = 12 * 0.9 * (1 + 50/50) ≈ 22

反过来，也可以根据线程数估算实际利用率：
U = N_threads / (N_cores * (1 + W / C))

公式的局限性：
- 在真实程序中，很难精确获得 W 和 C，因为代码往往包含内存读写、计算、I/O、系统调用等混合操作。
- 运行环境上还存在其他进程或线程（如 Tomcat、连接池、JVM 后台线程、垃圾回收线程等）会占用 CPU。
- 上下文切换、锁竞争、缓存命中/未命中等因素都会影响实际性能。

真实程序中的线程数实践
------------------

结论：没有固定的线程数答案。推荐的方法是先设定目标指标（例如目标 CPU 利用率、可接受的 GC 频率/暂停、执行效率等），然后通过测试不断调优。

实践流程建议：
1. 分析当前主机上是否有其他进程会干扰（会占用 CPU）。
2. 分析当前 JVM 进程中已有的线程（应用线程池、容器线程、连接池线程、JVM/GC/编译线程等）。
3. 设定目标：
   - 目标 CPU 利用率：最多能容忍 CPU 飙到多少？
   - 目标 GC 频率/暂停时间：多线程可能会导致更频繁的 GC，能容忍多少？
   - 执行效率：例如批处理场景下单位时间内需要处理多少并发任务。
4. 梳理链路关键点，找出可能成为瓶颈的资源（数据库连接池、第三方接口限流、中间件承载能力等），因为线程数过多时可能在这些点大量等待。
5. 通过增加/减少线程数做压力测试，按最高要求（最苛刻场景）测试，最终得到“满足要求且稳定”的线程数配置。

不同场景下的差异：
- Tomcat 的 maxThreads 在阻塞 I/O 与非阻塞 I/O 下最佳设置不同。
- 类似 Dubbo 会区分 I/O 线程池与业务线程池，I/O 线程一般不需太多，业务线程更容易成为瓶颈。
- Redis 自 6.0 起部分 I/O 多线程化，但“业务”处理（如命令执行）仍主要是单线程化设计。

实用建议（简单场景）
------------------

对于许多内部业务系统或简单异步场景，不需要追求极致性能时，一个保守而实用的起点是：
- 将业务线程池大小设置为 CPU 核心数（逻辑核心），然后通过观察和测试逐步调整。

附录
----

Java 获取 CPU 核心数：
```java
Runtime.getRuntime().availableProcessors(); // 获取逻辑核心数，如 6 核 12 线程返回 12
```

Linux 获取 CPU 核心数（示例命令）：
```bash
# 总核数 = 物理 CPU 个数 × 每颗物理 CPU 的核数
# 总逻辑 CPU 数 = 物理 CPU 个数 × 每颗物理 CPU 的核数 × 超线程数

# 查看物理 CPU 个数
cat /proc/cpuinfo | grep "physical id" | sort | uniq | wc -l

# 查看每个物理 CPU 中 core 的个数(即核数)
cat /proc/cpuinfo | grep "cpu cores" | uniq

# 查看逻辑 CPU 的个数
cat /proc/cpuinfo | grep "processor" | wc -l
```
