# 看完这篇，你的 API 服务设计能力将再次进化!

引言  
在当今快速演变的技术场景中，服务设计不仅是遵循通用设计规范和最佳实践的问题，更涉及如何在满足这些标准的同时，确保服务能够灵活适应未来变化、满足用户期望。本文探讨在遵循通用设计规范之外，服务设计过程中需要考虑的关键因素，包含路径设计、请求方式、出入参、业务处理建议、异常处理、依赖分级、监控日志、降级点、过时服务处理以及服务与信息安全等方面的实践建议。

## 1、服务路径和模块
- 服务路径从域名后开始，应有统一的一级路径，如 `domain/api/`。这样可以将所有 API 请求集中到明确路径，简化安全控制（认证、授权、日志、限流、监控等），增强 API 的安全性。
- 业务分组建议：例如 `order/api/order`、`product/api/product`，让调用方清楚接口所属业务模块。
- 服务粒度需结合业务场景考虑：例如商品价格是否独立为一个服务，还是在商品详情中直接返回？在保证业务场景完整度的前提下，尽量提供可灵活组装的服务以实现解耦。

## 2、服务请求方式
- 大多数服务应支持 AJAX 请求，固定数据格式（JSON/XML 等），便于后续扩展并适配绝大部分客户业务。
- 对于特殊场景（如收银台、登录/登出），可能需要非 AJAX 请求，由服务端做后续的转发或跳转处理。

## 3、服务出入参
- 出入参设计要优先考虑扩展性，避免后续因扩展受限而新增重复接口或产生多个相似版本。
- 数字字段优先设计为 Long 或直接字符串，防止后端类型扩展（例如 Integer → Long）带来对调用方的改造成本。
- 区分“未设置”和“零值”：字段可使用包装类型（例如 Java 的 Long、Integer）以用 null 表示“未设置”。
- 出参字段尽量少暴露非必要字段；对于敏感字段（地址、手机号等），需慎重，必要时做脱敏或加密处理。
- 字段定义及说明要详细且合规，常见说明包括：
  - 字段长度限制
  - 数字字段范围
  - 业务域内相同含义字段名保持一致
  - 日期/时间字段格式说明
  - 枚举类型所有可能值及含义说明
- 已发布服务新增出/入参时需考虑调用方影响：若调用方反序列化使用严格模式，随意新增返回字段可能导致反序列化失败。可以通过在入参中增加扩展字段让调用方在传入指定数据时，服务返回新的出参。
- 出参格式统一示例，建议包含处理结果、状态码和业务数据，便于调用方做业务与异常处理，例如：
```json
{
  "success": true,
  "resultMessage": "success",
  "resultCode": "0000",
  "result": {}
}
```

## 4、服务内业务处理建议
- 尽量支持批量数据处理，减少调用方为高并发调用服务而承担的复杂度（如订单/商品查询、发票开具等）。
- 做水平越权校验，防止调用方查询到不属于自己的数据，校验严格程度依据业务而定。
- 对于重点业务逻辑需要多线程处理时，尽量使用独立线程池，防止共享线程池被其他业务抢占影响关键业务。

## 5、异常处理
- 对开放服务（尤其写操作），需要定义明确的异常码。调用方依赖错误码作自动化处理（重试、告警、忽略或中断业务）。
- 在出现业务异常时，使用统一的响应格式包装合适的异常码和异常信息。对未知异常（如空指针、依赖超时等），在最外层通过拦截器做兜底处理，避免直接将异常抛给调用方。

## 6、服务的强弱依赖
- 把外部依赖分为强依赖和弱依赖：
  - 强依赖：服务运行不可或缺，若异常应短路中断以避免错误传播或数据不一致。
  - 弱依赖：对主流程影响较小，可容忍一定异常，发生异常时可忽略并继续执行核心流程，同时触发业务告警以便追踪处理。
- 此策略旨在保证整体可用性并确保问题可追踪。

## 7、服务的监控和日志记录
- 日志记录：对外服务尤其要记录出入参日志，特别是涉及资金、订单、支付等写操作。一方面可以监控调用方使用了哪些字段，评估新增、下线或修改字段的风险；另一方面可用于数据分析和业务告警，还能在出现数据不一致时作为溯源依据。
- 调用数据收集：对开放平台流量做统计分析，用于流量治理和安全（识别刷单、爬虫等恶意调用），并为限流和安全防护提供数据支持。

## 8、合理设置降级点
- 在分布式和微服务架构下，服务降级是保障系统稳定的关键。当某些模块遇到性能瓶颈或故障时，可主动减少或关闭这些组件的某些功能，以保障核心运作。
- 在设计阶段识别关键/非关键模块，并在非关键模块嵌入降级逻辑，支持手动或自动执行降级，确保关键服务持续可用，减少故障对用户的影响。

## 9、处理过时服务
- 随着业务发展，早期简单设计的服务可能不再满足需求，会新增服务覆盖旧能力，久而久之可能产生几乎无客户使用的过时服务，影响代码排查或触发业务漏洞。处理办法：
  - 若逻辑兼容，可将旧服务路由到新服务。
  - 下线处理但不直接删除代码：统一拦截需要下线的路径，返回统一错误码和提示信息。一旦监控到拦截告警或调用方反馈，可先停止拦截并与调用方沟通后续方案。

## 10、不要相信任何调用方和依赖的第三方
- 对所有依赖的第三方服务保持不信任：对其返回数据、超时时间、异常返回等进行细致处理，考虑各种可能的失败场景，防止单个依赖问题影响全局。
- 同样不应完全信任调用方：无法保证调用方行为符合预期，因此在上线前要构建多种防御机制，包括但不限于限流、严格的输入校验和精细权限控制，提升系统的稳定性和安全性，保证面对不可预测外部因素时系统的弹性与韧性。

# 保障服务和信息安全的措施

## (1) 敏感字段
在 B2B 场景中，企业对用户敏感数据（手机号、用户名、住址等）要求较高，接口出参时应对这些数据进行加密或脱敏。加密可以在不同层面实施：
- 传输层加密：使用 SSL/TLS 对客户端与服务端之间通信进行加密。
- 应用层加密：在发送前对特定敏感字段用 AES、RSA 等算法加密，为传输提供额外保护。
- 数据库层加密：对存储在数据库中的敏感字段加密（常用于密码等信息）。

注意事项：
- 选择业界常用、经过验证的加密标准与算法。
- 安全管理加密密钥（生成、存储、交换、销毁）。
- 考虑性能影响，加解密会增加计算开销，需评估对服务性能的影响。

## (2) 系统准入
部分对安全要求高的客户会要求对非法调用方进行拦截，防止第三方获取身份后以客户身份调用导致资损或数据泄露。常用方式为 IP 黑白名单：
- IP 白名单：只有列入白名单的 IP 被允许访问。优点：高安全性，可在客户维度上限制访问；缺点：管理麻烦，动态 IP 不友好。适用于访问者数量有限且 IP 固定的环境。
- IP 黑名单：列入黑名单的 IP 被拒绝访问。优点：简单，便于列出已知恶意 IP；缺点：安全性较低，无法阻止未知或变换 IP 的攻击者。适合作为其他安全措施的补充。

## (3) 接口出入参防篡改
确保接口数据在传输过程中未被修改，常见方法：
- 使用 HTTPS：加密传输，降低截取与篡改风险。
- 数字签名：发送方使用私钥对数据（或数据哈希）签名，接收方用公钥验证签名以确保数据未被修改。
  - 流程：发送方签名 → 发送数据与签名 → 接收方验证签名。
- 消息认证码（MAC）：使用共享密钥和消息内容生成 MAC 值，接收方用相同密钥生成并比对。
- API 密钥 + 时间戳：时间戳防重放攻击，API 密钥确保只有授权用户可以发送请求。

## (4) 出入参加解密
出入参加解密指客户端请求到服务器（入参解密）与服务器响应到客户端（出参加密）过程的加/解密操作，常见于对数据安全要求非常高的客户（银行、央企）。这类客户通常会指定加密方式，可能是通用算法，也可能提供自定义 SDK。

实践做法示例：
- 搭建能加载不同客户加密 SDK 的隔离平台（例如 ECI），按客户维度配置加解密方法，在拦截器层统一处理入参解密与出参加密，做到对业务代码无侵入。
- 平台隔离加载各客户 SDK，避免某个 SDK 问题影响全部客户。

