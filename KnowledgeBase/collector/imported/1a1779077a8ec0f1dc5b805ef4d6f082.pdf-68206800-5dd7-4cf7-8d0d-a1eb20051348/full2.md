# 分布式系统不可靠时钟问题

作者：疾风先生

今天结合《设计数据密集系统》来聊分布式系统的另一个话题：不可靠的时钟问题。

## 同步网络与异步网络时钟对比

在前文我们讲到分布式系统网络不可靠的原因之一是数据中心服务通信采用的是异步网络实现。同样地，同步网络与异步网络在时钟控制上也存在差异：

- 同步网络存在全局时钟控制。
- 异步网络中每个节点拥有自己的时钟独立运行，并通过 NTP 协议来协同纠正时钟不同步的问题，但仍受网络延迟影响。
- 同步网络发送消息需要先同步脉冲后再传递数据信息；而异步网络通过开始位和结束位标志进行数据同步以保证数据的完整性。

对比总结（时钟精度与控制）：
- 同步网络：全局时钟控制，时钟一致性更易保证但需要额外硬件或同步机制。
- 异步网络：每节点本地时钟，依赖 NTP 等协议校准，受网络延迟与设备差异影响较大。

## 单调时钟与日时钟的不可靠性

应用程序中常用两类时间含义：

- 持续时间（duration）：描述事件从开始到结束经历的时间，例如请求的 p99 耗时。通常用来测量持续时间，需要准确的时间差值。
- 时间点（time instant）：事件发生的某一时刻，例如日志中记录异常发生的时间戳。

现代系统中的时钟可以分为两类：

日时钟（wall clock）
- 根据日历返回当前日期和时间，例如 Java 的 System.currentTimeMillis()，用于表示时间点。
- 通常与 NTP 同步，理想情况下不同机器的时间戳含义相同，但由于硬件时钟与 NTP 的不稳定性，不适合用于测量持续时间。

单调时钟（monotonic clock）
- 主要用于测量持续时间，例如超时或服务响应时间，例如 Java 的 System.nanoTime()。
- 单机单进程下差值有意义。单调时钟可能是自系统启动以来的纳秒计数或类似任意基点的增量数值。
- 单调时钟不需要与 NTP 同步，但 NTP 若检测到本地晶振偏差，会通过微调影响单调时钟的频率（即微调时钟向前走的频率），并且进程暂停（如 GC、SafePoint、fsync 等）会导致观察到的时间停顿。

单调时钟的优点是保证单方向递增，因此在单进程内计算持续时间优先使用单调时钟而非日时钟。日时钟不适合测量持续时间的原因包括：
- 石英晶体钟漂移，可能发生向后跳跃；
- 本地时钟与 NTP 差距过大时被重置；
- NTP 配置错误；
- NTP 同步受网络延迟影响。

因此在分布式系统中，设计应用时必须充分考虑日时钟的不可靠性。

## 分布式系统依赖时钟带来的问题

### 依赖时间戳进行事件排序导致乱序

如果两个客户端向分布式数据库写入数据，谁先到达并最终生效？采用最后写入者胜出（LWW）策略会出现问题。示例（来自《设计数据密集系统》）：

![示意图：不同节点时钟偏移导致写入顺序冲突](image_placeholder)

事件逻辑：
1. ClientA 在 42.004 秒向 Node1 发起 `set x = 1`，Node1 向 Node2、Node3 复制写操作。
2. ClientB 在 42.003 秒向 Node3 发起 `set x = 2`，Node3 向 Node1、Node2 复制写操作。

在 Node2 上，`set x = 1` 的时间戳是 42.004，而 `set x = 2` 的时间戳是 42.003。若使用 LWW（采用时间戳较新的写覆盖旧写），Node2 会丢弃 `set x = 2`，从而造成数据“丢失”。

产生问题的根本原因是不同节点物理时钟（即日时钟或单调时钟）之间存在偏差。解决方案之一是使用逻辑时钟（基于全局递增计数器而非晶振），逻辑时钟不测量绝对时间，仅测量事件的相对顺序，从而为 LWW 或并发写冲突提供更安全的依据。

需要注意，逻辑时钟依赖因果关系实现，且通常需要持久化防止丢失，具体实现更为复杂。

### STW（Stop-The-World）引发的数据写入安全问题

假设一个数据库每个分区只有一个主节点（master）能接受写操作，主节点通过租约（lease）与副本达成共识。主节点持有一个带过期时间的租约，在租约有效期内处理外部写入并复制到其他节点，直到租约过期。

伪代码示例（示意）：
```text
// 每个 Node 的伪代码示意
if (current_time < lease_expiry) {
  accept_writes();
} else {
  try_acquire_lease();
}
```

该方案存在两个主要问题：
- 判断租约过期依赖外部节点的时间戳。若日时钟不同步超过几秒，可能误判为已到期从而发起续约或重新选举 master，选举期间对外服务不可用，写请求被拒绝；若时钟回退，也可能导致多个节点同时认为自己拥有租约，出现脑裂（split-brain）。
- 即便使用单调时钟也存在问题。进程内部发生 GC、进入 SafePoint、执行 fsync 等会暂停进程，导致其他副本误判 master 不可用并重新选举，然后两个节点可能同时接受客户端写操作，造成数据不一致或损坏。

因此在设计分布式系统时必须假定任何节点在任意时刻都可能被暂停一段时间（甚至在函数执行中间），只有基于这个假定来设计系统，才能保证数据的可靠性与完整性。

## 全局快照的同步时钟

为了保证跨多台机器甚至多数据中心的事务顺序性，需要某种全局单调递增的计数器。这在多数据中心环境下极为困难，因为跨区通信需要协调，直接使用全局互斥会严重影响性能。

许多数据库通过快照隔离（snapshot isolation）在性能与并发读取方面做折中：支持小型、快速的读写事务，并允许大型、长时间运行的只读事务。快照隔离与事件顺序性的关系是：如果事务 B 能读取事务 A 的数据，那么事务 B 的 ID 必须高于事务 A，否则快照就是不一致的。

谷歌 Spanner 就是一个基于日时钟的跨数据中心快照隔离实现。Spanner 采用 TrueTime API 来明确报告本地时钟的置信区间（即一个区间 [最早时间, 最晚时间]，而不是单一时间点）。基于以下观察：
- 对于两个提交时间戳 a 与 b，假设对应的置信区间为 A 和 B。
- 若 A 与 B 不重叠且 A 的最晚时间 < B 的最早时间，则 A 一定发生在 B 之前；反之亦然。
- 若 A 与 B 有重叠，为了反映因果关系，Spanner 在提交读写事务时会特意等待一段置信区间长度的时间。为了尽量缩短等待时间，Spanner 需要尽可能减小时钟的不确定性，因此在每个数据中心部署 GPS 接收器或原子钟，使时钟误差降至约 7 ms 内。通过这种方式，它可以确保任何可能读取该数据的事务都发生在足够晚的时间点，从而保证一致性。

可以将逻辑时钟实现与 TrueTime 在几个维度上对比：逻辑时钟关注事件顺序与因果关系，TrueTime 在保留物理时间语义的同时，通过报告不确定性区间并等待来保证跨节点顺序性。

## 分布式系统问题小结

搭建分布式系统时需要建立清晰的系统模型，并在面临组件/服务故障、网络延迟、时钟依赖以及节点暂停等情况下进行设计权衡，以保证数据可靠性。关键要点包括：
- 区分时间点与持续时间的用途，分别使用适当的时钟（日时钟用于时间点，单调时钟用于单机持续时间测量）。
- 意识到物理时钟在分布式环境下的不可靠性，避免直接依赖绝对时间进行全局决策。
- 在需要全局顺序与快照隔离时，考虑逻辑时钟、TrueTime 或混合方案，并权衡复杂性与性能。
- 设计时要假定节点可能随时暂停，确保租约、选举与复制协议能容忍暂停带来的副作用。
