# B 站轻量级容灾演练体系构建与业务实践

分享嘉宾：刘昊

内容亮点：
- 了解 B 站如何轻量级在组织内部常态化实施容灾演练
- 了解 B 站容灾演练的技术演进路线、组织搭建模式和运营机制流程，掌握体系化的解决方案

以下为演讲整理内容。

大家下午好，今天我要讨论的议题是「B 站轻量级容灾演练体系的构建与业务实践」。关键点主要在于“轻量级”，也就是如何以低成本进行容灾演练。过去两年，整个行业都在开展降本增效工作，同时提出要确保业务质量不受损。而容灾演练是推动业务质量建设和有效性验证的最好手段，但这会提升整体工作负载。因此，探讨如何轻量化、低成本开展容灾演练非常有意义。

我简单介绍一下自己：我加入 B 站较早，见证了 B 站从“小破站”发展到拥有亿级用户的演进过程。近几年我主要负责稳定性相关工作，包括应急响应、变更防控、红蓝攻防演练，以及围绕运维稳定性的元数据建设。

今天的分享分为四个部分：首先介绍当前新形势下面临的稳定性挑战以及如何引导我们构建轻量级演练体系；接着详细阐述体系如何构建与实施；然后讨论体系落地的效果，特别是业务场景演练实践；最后做总结与展望。

## 新形势下稳定性挑战

随着信息技术快速进步与数字化转型深入，从集中式架构到分布式、微服务、云原生，以及行业形态的变化，软件系统复杂度和多样性大幅增长，稳定性面临前所未有的挑战。基础设施层面的故障和隐患日益增多，其影响往往具有全局性和长期性（例如数据中心火灾、光纤被剪断事件等），这些都揭示出基础设施故障可能导致大范围服务中断的风险。

行业内广泛强调高可用性服务架构（如同城双活、异地多活），但这些理论和设计在常规运行中未必经历过足够的实战验证。当外部挑战来临时，我们的抵抗能力如何，需要通过演练来检验。

对于 B 站，稳定性挑战主要体现在两方面：一是用户群体活跃，系统故障会被迅速放大并引发流量冲击；二是网络专线异常或业务代码变更可能导致大面积影响。历史上我们也经历过严重中断，因此持续推进多活架构与业务优化，但这些改造是否真实有效需要通过实际运行中的故障验证来检验。

### 如何破局？主动出击，以演带练

为了验证技术改造和功能更新的有效性，需要建立故障演练和验收平台。我们面临的挑战包括强弱依赖故障、应用依赖不合理、可用区切换问题以及降级策略和兜底策略的有效性验证。演练要尽可能模拟真实环境的故障情景，激发研发团队主动性，使验证常态化而非一次性任务，从而持续推动系统稳定性建设。常态化验证不仅能持续检测系统可靠性，还能在降低成本的同时优化资源配置，这促生了轻量化演练的必要性。

B 站早期引入混沌工程，并借助开源工具（如阿里、PingCap 的开源产品）封装成简易平台用于故障注入和验证。但这带来了重复建设、个性化需求和产品化能力不足等问题。混沌工程不能随意注入故障，它必须遵循规则和逻辑。大事业部有人力负责混沌演练，较小事业部则缺乏支持。过去多集中于特定时期（如大型赛事）做演练，日常演练关注度不足，因此需要常态化、轻量化的故障演练平台。

在 2022 年我们整合了多个故障演练平台，目标是增强基础能力、去除冗余，并以混沌工程五大核心原则为基准，优化产品化设计，提升使用体验和稳定性。去年推进演练 3.0 阶段建设，目标实现常态化并确保真实有效，提出从混沌演练到容灾演练的全覆盖方案，强调演练全面性和常态化，从组织、流程、技术等维度持续优化。

在平台优化上，我们关注如何低成本实现广泛覆盖、自动计算输入参数、自动圈定演练链路，降低演练复杂性，提升效率和效果。当前处于演练 3.0 阶段，核心目标是轻量化，推动演练流程推广到全公司，涵盖更多部门和角色。为此需要固化和标准化组织内的演练流程与章程、明确参与部门与职责；技术流程要规范化，提升平台产品化能力，减轻人员心智负担，目标是让用户可以自主发起和执行演练而不依赖 SRE 或专门演练团队。

演练执行中还需增强观测指标能力、拓展可注入对象和范围、推动自动化，量化预期结果并自动收集分析演练数据，异常时提供自动防护，确保演练不会引发真正生产故障。组织间协同机制和内部认知也很关键，需要把演练作为提升系统可靠性和应对能力的手段，通过培训降低学习成本。基于这些挑战我们设定关键目标：让演练平台充当“演练专家”，大幅降低操作和学习成本，通过标准化流程规范演练行为，确保演练在可控、安全范围内进行，推动演练常态化。

## 轻量级容灾演练体系

演练体系的底层是原子故障库，包含 CPU 异常、网络异常、延迟、丢包等基础故障模拟。这些原子故障构成演练核心元素，为各类场景创建和演练提供基础。

在原子故障库基础上封装两大类场景：
- 通用场景：如 CPU 高负载、容器异常、网络丢包，简化用户使用流程，用户只需填写参数或调用封装好的场景。
- 特化场景：针对公司内部特定业务形态（如双活、多活策略、自研中间件）定向封装，满足个性化需求。

进一步构建混沌实验层，关注实验设计与分发，包括如何在不同载体中注入故障，模拟和测试系统在真实故障场景中的表现。演练过程中通过观测能力实时监控注入与恢复效果，不仅验证恢复的有效性，还用稳态指标观察业务稳定性并及时发现异常。

经验沉淀方面，通过故障组合模拟复杂业务场景，多次演练后形成演练模板，帮助服务团队快速创建演练（如网关、SLB、跨机房、跨服务等复杂场景的模板）。体系最上层是产品层的演练管理，涵盖演练编排、故障生成、结果分析和报告生成等功能，便于管理全过程与事后分析。

发展路线是从工具到平台再到一站式服务，最终实现轻量化目标。核心在于技术支撑，但技术搭建不足以确保落地，必须与现有流程和平台无缝集成以发挥最大价值。

在研发过程中计划将故障注入能力深度整合到持续集成（CI）与回归测试流程中，尤其在强弱依赖演练方面希望在制品库出包前或发布准备阶段自动模拟故障并进行检测，及早发现潜在问题，确保发布后的稳定性与容灾能力。大型容灾演练涉及跨机房测试、多活演练与降级兜底方案演练，用以验证容灾方案在真实环境中的有效性。技术落地后要实现容灾演练能力的跨领域赋能，与周边平台协同形成生态，例如：应急响应演练用于故障溯源与定位、攻防演练用于增强抗压能力，以及借助内部大模型团队对演练内容复盘和梳理，基于应急中发现的问题识别薄弱点并调整演练计划，进一步强化容灾能力。

## 组织阵型搭建

组织架构与制度完善对容灾演练成效至关重要。我们从技术、流程与组织三个维度推进，但逐渐认识到组织架构的明确性对效果决定性影响。项目初期通过组织架构优化与虚拟项目协同推动，确保高效实现演练协同目标。

成立“安全生产委员会”，成员由一线核心领导组成，负责高层稳定性决策、关键演练路径规划、制定规则并设立基线与红线，同时提升公司应急协同能力和应急意识。制度方面已发布多部白皮书，规范容灾架构设计、演练章程与操作规范。

在文化宣导上实施“安全生产月”活动，通过易拉宝、专题分享等方式推广稳定性改进经验与最佳实践，并建立在线学习平台定期发布培训视频，帮助新员工快速掌握容灾演练相关知识与技能。推出“孤岛行动”专项，通过机房级别的断网等真实演练促使业务线自诊自保，围绕目标驱动推动演练效果落地。技术分享方面每周通过内部公众号发布容灾演练实践与经验，确保技术经验在公司内部传播与应用。

## 运营机制流程

流程支撑分为两个核心模块：演练执行流程与演练全生命周期流程，用以确保业务高效开展日常演练并为平台功能设计提供依据。

演练执行流程涵盖演练前、中、后三个阶段：
- 演练前：梳理业务场景、明确场景类型与评估方案及影响，决定采用染色流量演练、线上不切流演练还是先切流量后注入故障。
- 演练中：明确范围与环境、划定集群与机器边界、配置规则、模拟故障并确定监控关键指标，设计保护机制以便快速拦截与恢复，同时执行应急预案。验证场景时设置合理指标并通过断言验证预期。
- 恢复阶段：确保故障恢复后业务达到预期状态，因为有时故障撤除后业务仍可能异常。

演练后需规划常态化演练、评估是否开展突袭演练，并根据演练中发现的问题制定待办事项，分配责任并推进落实。

演练全生命周期流程关注将演练在非生产环境验证后再推进到生产环境。非生产环境可直接进行跨网络全局演练，无需流量染色；生产环境需要考虑流量染色策略以及与应急响应流程联动，一旦触发故障要迅速启动应急响应。推动常态化演练周期并引入突袭演练机制，确保演练覆盖多种突发情况并在真实环境中充分验证。

## 容灾产品能力

在组织与流程基础上引入技术手段，确保组织协同和流程标准落地，实现六化目标：
- 演练协议标准化
- 故障引擎自研化
- 实验参数精简化
- 演练覆盖全面化
- 演练过程可视化
- 演练流程自动化

### 演练协议标准化

最初依赖开源工具（如 ChaoBlade）降低成本，但多个团队各自开发工具导致命令行参数与接口协议差异，需通过模型抽象实现统一：标准化的演练描述模型（演练对象、范围、载体类型、演练参数、预检要求等）和演练交互模型（注入、注册、执行、恢复、审计、查询等功能）。使用 Golang 接口进行行为抽象，兼容原有工具并平滑迁移到新工具。

### 故障引擎自研化

依赖开源工具在实践中暴露问题，促成自研决策，主要原因包括：
- 故障注入可靠性不足，影响演练成功率（如 DNS 干扰演练通过修改 /etc/hosts 失效、网络不可用注入因命令过长无法执行、CPU 负载注入效果不足等）。
- 演练编排与流程控制过于简易，缺乏状态管理与回溯能力，无法满足可控性与可追溯性要求。
- 缺乏服务/资源鉴权，增加演练安全风险，需要更精细的权限管理以满足安全合规。
- 观测手段不足，难以排查问题；在容器与 Kubernetes 调度场景中链路复杂且缺乏实时反馈。

自研并提升观测平台以覆盖从外部请求到内部故障代理调度的全链路监控，实时反馈每个环节状态，提高演练成功率与效率。

### 实验参数精简化（以网络演练为例）

为实现系统轻量化，需简化实验参数配置，降低用户与运维负担。把过程式参数（具体操作步骤与命令）转为声明式参数（目标与效果描述），用户只需声明期望的故障效果而无需关心实现细节。比如将原来复杂的 TC 命令配置抽象为对 IP、端口、影响范围等声明式描述，系统自动完成具体操作。这种设计降低心智负担并提高开发效率。

在跨可用区断网容灾演练中，服务级别测试步骤通常包括：流量切换到其他路径、模拟某 AZ 的专线断网、验证服务功能。准备阶段需识别哪些服务会受影响（服务 Pod IP、主机 IP），并确定需要断开或不应断开的服务。对不应断开的服务分为预期内不断开（如 MySQL 等需跨机房通信的服务）与预期外不断开（如未完成改造的依赖）。

核心任务是通过平台自动梳理并识别服务间的网络依赖关系，自动列出服务依赖的数据库、缓存等组件，精准判断哪些服务应断开或保留，从而简化容灾演练操作，用户只需关注需要豁免的服务并做配置。通过自动化服务依赖梳理与豁免管理，跨服务、跨可用区容灾演练得以简化与闭环。

### 演练覆盖全面化

针对不同规模的服务架构（包括海外服务、CDN 节点等），采取三项措施：
- 多样化注入对象类型：抽象为容器与服务器两种核心载体，结合 CMDB 获取集群元信息以便选择演练对象。
- 大规模注入载体管理：在 K8s 等大规模容器环境中实现自动化打标与撤标，故障注入组件仅在演练期间激活，演练结束自动撤除，降低误操作风险。
- 多样化环境支持：覆盖机房内、机房外及各区域部署，通过公有云与 Proxy 管理实现跨网络指令调度，基于内部统一应用信息映射简化跨区域调度。

### 演练过程可视化

可视化依赖标准化监控指标与模型抽象。对于未遵循 OpenTelemetry 标准的业务，也需纳入监控，否则无法提供演练期间的监控数据。制定监控指标标准，关注基础行为抽象：如何获取对象数据、明确服务对象定义、围绕对象的关键指标以及如何实时获取指标数据。系统在选择演练对象时自动加载相关指标供用户勾选，完成后可实时监控指标变化。

演练架构以演练对象为核心，展示基于 Trace 的动态调用链和基于 CMDB 的静态拓扑。故障注入后可以直观看到对上下游服务及自身的影响，这种可视化有助于理解演练复杂性并确保可控性与有效性。

### 演练流程自动化

实现自动化关注三点：
- 场景自动验证：能否自动验证演练场景有效性。测试团队提供 UI 自动化测试能力，可直接与演练流程对接以验证多场景和单应用结果。
- 演练自动执行：包括定时调度器（定期触发演练）、编排自动化（自动推进演练流程）与护栏机制（实时获取可观测指标并通过断言与规则引擎进行保护以避免偏离预期）。
- 预期自动验证：基于指标判断演练后场景是否符合预期或是否触发反结果，并追踪未按预期发生故障的链路补偿机制，确保演练结果精准可靠。

## 业务场景演练实践

### 演练协同一体

目标是将线下协同演练转为线上协同，通过平台化将多方协作移至同一演练面板，实现演练人员、系统、业务服务与关键数据的实时汇聚与管理。演练全过程在同一页面完成，阶段包括：
- 演练配置：定义演练对象、测试范围、编排流程并配置原子实验，设定观测指标与护栏机制。
- 演练执行：所有操作集中在同一页面，简化流程并提升效率。
- 演练恢复：确保系统与服务恢复到正常状态。
- 报告生成与验收：生成演练报告并验收，评估效果和业务影响，为未来改进提供数据支持。

### 可用区级断网容灾演练

公司级别的大规模可用区演练通过在交换机上实施策略切断机房专线与公网入口，测试机房故障时的逃生与容灾能力。演练节奏通常在年度规划中设定宏观目标并细分关键节点、角色与交付物，进行现状评估与白名单排除后执行并复盘。总体思路包括三阶段：
1. 断开机房间专线，检查双活/多活架构实施效果。
2. 切断南北向流量（公网入口）测试系统承载能力。
3. 完全脱网模拟（同时断开专线与公网）测试极端故障恢复能力。

具体流程如先进行公网流量切换验证承载能力，再切断专线保留少量流量（如 1%），逐步排查与引导剩余流量，最终模拟完全脱网并观察全流程影响。

### 集成测试演练

把故障注入能力移植到测试环境，成为测试团队日常工作一部分。平台提供 OpenAPI 能力，测试/QA 团队可将测试用例集成到 CI 流程中，代码提交后自动触发执行以验证新代码不会引入不合理依赖或潜在故障。关键是识别适合快速验证并能在流水线中高效运行的测试场景（例如断网演练通常不适合在 CI 中执行）。充分利用 UI 自动化与接口测试能力进行系统验证。

### S 赛演练

以《英雄联盟》赛事为例，基于分级规则对赛事进行等级划分并制定赛事保障场景地图，规划并应用前述能力进行系统摸排。专项摸排旨在深入挖掘代码层面的强弱依赖与降级方案合理性，发现的关键问题对保障赛事期间服务稳定性与用户体验非常重要。

## 总结与展望

### 演练收效
- 演练从规划到落地周期从周级缩短至天级，显著降低时间与资源成本。
- 演练覆盖由核心部门扩展到全公司，支持跨部门协同。
- 演练频次从季度提升到每周一次，业务团队参与度提升。
- 演练有效性验收实现自动化并复用测试场景，产生技术建设的复利效应。

### 实践反思与建议
- 明确演练目标：开始前设定清晰目的与预期成果，便于评估与改进。
- 量身定制推广方案：根据各业务团队特性定制推广方案，推动实施以确保落地。
- 优先高密度、高关注、高收益的演练类型：如强弱依赖演练、应急预案演练与多活演练。
- 演练自动化关键在于过程护栏有效性与演练效果验证自动化，确保全过程受控并能自动验证效果。
- 完善自动化用例：针对演练场景补充自动化用例，确保关键场景与操作被覆盖。
- 接口自动化优化：利用流量录制工具捕获真实业务流量并转为自动化测试用例，增强现实性与覆盖面。
- 分阶段实施混沌工程实验：针对不同成熟度系统采用不同类型混沌实验，逐步提升系统稳定性与容错能力。
