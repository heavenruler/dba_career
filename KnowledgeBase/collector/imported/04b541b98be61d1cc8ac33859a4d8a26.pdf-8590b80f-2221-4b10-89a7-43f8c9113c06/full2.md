# 二级缓存架构极致提升系统性能

作者：木将（阿里云开发者）  
日期：2024-09-11

阿里妹导读  
本文详细阐述了如何通过二级缓存架构设计提升高并发下的系统性能。

## 前言

随着 k8s 成为云端常用平台，容器成为众多用户弹性的利器，因此容器的创建天生具备高并发特性。  
在高并发、大数据量下，为了提供更好的容器弹性体验，笔者通过二级缓存的设计，成功优化了系统性能、资源消耗和系统容量。

但持续压榨性能的道路是曲折的。各种缓存方案需要考虑非常多因素，包括缓存的多级架构、预热、击穿、刷新、运维等。  
下面先展示优化效果，然后详细介绍带来性能提升的二级缓存架构，以及各种缓存方案的设计、比较与落地。

## 优化结果

笔者对系统进行了多次压测，观察了二级缓存等优化手段上线前后，在高 QPS 下的资源消耗、RT 和系统容量。

1. 资源消耗大幅下降  
   上线二级缓存等优化手段后，相同 QPS 下，CPU 使用率大幅下降。

2. RT 和系统容量优化  
   由于计算资源消耗下降，同 QPS 下的平均 RT 也大幅下降。优化前随着 QPS 提升，RT 明显变慢且幅度很大；优化后，RT 变慢的幅度明显减缓，这意味着系统容量也得到了显著提升。

## 系统瓶颈在哪？

持续压榨性能的前提是知道瓶颈所在。笔者通过 cpuProfile 火焰图定位瓶颈。  
在高 QPS 下，分析系统 CPU 火焰图，发现 50%+ 的 CPU 瓶颈在大量的业务数据处理，主要有两块：

1. 从 Redis 取业务 pojoList 时的 fastjson array 反序列化。背景是与 Redis 交互的数据量特别大，缓存 value 曾经做过压缩，在解压缩后的 string 转为 List<Pojo> 的过程中，实际上是用时间换空间，消耗大量 CPU 时间。
2. 调用 Redis 获取数据后的 alibaba.cachejson 的 POJO 解析。

在高并发、大数据量的业务背景下，这些过程的耗时被笛卡尔积般放大。既然分布式缓存的 CPU 资源消耗是瓶颈，那么引入本地缓存，就能缓解这个问题。另外，本地缓存还能提供服务容灾能力。

## 数据属性和缓存选型

### 数据属性（业务分析）

- 符合 key -> value 的数据模式
- 读多写少
- 变更频率低
- 数据一致性要求不高

这类属性是缓存的绝佳使用场景。接下来在本地缓存、分布式缓存与二级缓存之间做匹配分析。

### 本地/分布式缓存特性

本地缓存
- 优点
  - 访问速度快
  - 减少网络开销：不存在 Redis 流量瓶颈（之前由于数据 valueList 量过大，导致 Redis 网络流量超过实例 SLA，还做过一次 gzip 压缩）
- 缺点
  - 可用性相对差：如果应用实例宕机/重启，缓存数据会丢失
  - 资源限制：受限于单机内存大小，不适合大规模数据缓存（但本业务数据量相对较小且较为静态，可放入内存）

分布式缓存
- 优点
  - 高可用：数据共享，不会因为业务机器重启丢失
  - 大容量：可以水平扩展存储容量
- 缺点
  - 序列化带来的 CPU 瓶颈（在大数据量、高调用量下尤其明显）
  - 网络开销：存在 Redis 流量瓶颈

本地 + 分布式二级缓存
- 优点
  - 结合两者优势，层次化加速
- 缺点
  - 对代码侵入较大，实现较复杂
  - 额外的缓存层级会占用更多计算和存储资源

## 缓存方案比较

结合业务场景和系统现状，评估了三种方案，从多级架构、预热、击穿、刷新、运维等角度比较。

### 方案1：仅本地缓存 + Guava refreshAfterWrite

1. 架构  
   仅本地缓存，根据业务场景设置失效时间。

2. cache miss 处理与刷新  
   使用 Guava Cache 的 refreshAfterWrite + 异步 reLoader 机制刷新缓存，保证时效性。超过 TTL 后，Guava 会在 reLoader 中拉取最新的 <key, value> 更新缓存。

   在本业务场景下，缓存的 key & value 设计有两种方案：  
   a. <key, value>：Guava cache loader 只能按 key 更新。如果在单地域有 N 台机器，每台查 M 个 key，缓存刷新时最大会有 M*N 个查询请求打到下游实时数据系统，无法利用批量查询能力，且 M 数量非常大，下游压力会翻倍。  
   b. <"all_data"(hardCode), Map<key, value>>：可以利用批量查询能力，但当某些 <key, value> 在本地缓存缺失时，Guava 的 loader 无法识别 cache miss 并从下游系统拉数据。

3. 预热  
   启动时全量预热。

4. 运维  
   暴露 dubbo/http 服务，按主机运维或通过重启机器。

优点
- 无 Redis 相关的 CPU、网络消耗
- 直接使用 Guava cache 原生 loader 机制

缺点
- 对下游压力翻倍

由于对下游压力太大，放弃此方案。

### 方案2：二级缓存（本地 + 分布式）+ 刷新 Job

1. 架构  
   本地 + 分布式二级缓存。本地缓存失效时间设为无限（即以本地为主，确保高命中率）。

2. 预热  
   启动时全量预热。

3. cache miss 处理  
   不使用 Guava cache loader。cache miss 时业务调用分布式缓存（Redis），再 miss 则调用下游服务。

4. 刷新  
   使用 Spring 单机 job 定时全量刷新缓存，保证一定时效性（数据变动频率很低，因此 job 频率设得很低即可）。由于系统在引入分布式定时任务框架（例如 Quartz）时，并没有直接支持单机 job（Quartz 本身是支持的），因此额外使用了一个 Spring 单机 job 框架，这会导致任务管理框架不统一。

5. 运维  
   本地缓存按主机运维（暴露 dubbo/http 服务或重启机器）；分布式缓存通过 Redis 服务管理。

优点
- 本地缓存命中率高，基本不会发生 Redis 瓶颈

缺点
- 业务代码需自行 load 并 write back
- 使用了新的定时任务框架（单机），使系统任务管理框架不统一

### 方案3：二级缓存 + Guava expireAfterAccess

1. 架构  
   本地 + 分布式二级缓存。根据业务设定本地缓存失效时间（expireAfterAccess）。

2. 预热  
   启动时全量预热。但这会导致应用重启后缓存预热完成时，部分本地缓存 TTL 可能批量到期失效，后续请求到来时可能大量击穿到下游服务，形成典型的缓存雪崩场景。  
   而且，当本地缓存雪崩或大量 miss 时，即使命中分布式缓存，也会导致 Redis 相关序列化的 CPU 瓶颈，从而造成系统性能长尾。

3. cache miss 处理  
   不使用 Guava cache loader。cache miss 时业务调用分布式缓存，再 miss 则调用下游服务。

4. 刷新  
   无刷新 job。

5. 运维  
   本地缓存按主机运维；分布式缓存由 Redis 管理。

优点
- 不需要额外刷新 job
- 不强依赖运维方案（本地和分布式缓存都有失效时间）

缺点
- 存在缓存雪崩风险
- Redis 相关序列化造成 CPU 瓶颈的概率仍然较大

## 落地方案

根据上述优缺点与改造量评估，最终基本按照方案 2 执行。

1. 架构  
   本地 + 分布式二级缓存，以本地缓存为主，保证本地缓存极高命中率。

2. Cache Miss 处理  
   业务自行处理，保持现有链路一致：localCache -> Redis -> Dubbo，这样能够快速上线并降低变更风险。

3. 预热  
   应用启动时在提供服务前进行预热。

4. 刷新  
   基于 Spring Cron Job 实现定时刷新。

5. 运维  
   按主机运维，提供 dubbo/http 服务用于管理与排查。

此外，对 Guava Cache 进行了包装，各个业务场景的本地缓存统一存储在一个容器 Map<${prefix}, Cache<${key}, ${value}>> 中。对现有本地缓存也进行了重构，通过缓存包装层统一交互与统一管理，提升可复用性并简化代码。

## 写在最后

本文对多种缓存方案在架构、预热、击穿、刷新、运维等方面进行了比较分析，并完成了工程落地，极大提升了容器场景下高并发、大数据量业务的系统性能。后续的优化点包括：

1. 其他系统瓶颈：当前压测模式较单一，缓存可能掩盖了其他依赖服务的性能瓶颈。未来会采用更全面的压测模式继续挖掘系统性能极限。  
2. 批量刷新能力：针对方案 1 的批量刷新能力缺陷，Caffeine 实际上有相关特性，可参考其 Bulk refresh 相关讨论/issue。  
3. 运维方案：缺乏集群批量 invalid 能力，需建设完善。  
4. 架构统一：期望将分布式与单机 Job 整合到同一个任务管理框架中。
