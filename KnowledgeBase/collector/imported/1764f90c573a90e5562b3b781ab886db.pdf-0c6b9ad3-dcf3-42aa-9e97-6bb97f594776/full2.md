# 关于“稳定性建设”的一些思考

作者：大袤（大袤宏图）  
时间：2023-11-05；地点：北京；原创

## 01 概述

技术方案模板解耦合

系统的稳定性是系统提供良好服务的基础，稳定性建设是一个系统化工程，贯穿软件生命周期的全过程。残酷的墨菲定律提示我们不要对自己系统的可用性过于乐观——“If something can go wrong, it will.” 并且通常发生在最糟糕的时间。我们不能把碰运气当成战略。

## 02 全景

做好稳定性建设如同盖一座坚固的房屋，需要坚实的根基和强壮的支柱。从时间维度划分，稳定性建设可分为不同阶段；从空间上看，稳定性建设应是结构化、系统性的整体工程。

### 2.1 使命

技术服务于业务，保证业务的稳定、高效、安全、持续增长是首要任务。为落地实施，需要一套目标体系来度量稳定性建设的效果。

### 2.2 目标

- 服务质量指标（SLI：Service Level Indicator）
- 服务质量目标（SLO：Service Level Objective）
- 服务等级协议（SLA：Service-level agreement）

SLA 包括 SLO 及当 SLO 未达成时的后果（如经济赔偿、信用措施等）。SLA 可以理解为 SLO 达成部分 + SLO 未达成部分的约束与责任。

#### 2.2.1 SLI（VALET 指标集）

推荐使用 VALET 指标体系，从不同层级定义稳定性指标：
- 容量（Volume）：服务承诺的最大容量（如 QPS、TPS、流量、连接数、吞吐等）。
- 可用性（Availability）：
  - 时间维度：Availability = MTBF / (MTBF + MTTR)，其中 MTBF 为平均无故障间隔（Mean Time Between Failure），MTTR 为平均修复时间（Mean Time To Repair）。
  - 请求维度：Availability = 成功请求数量 / 请求总数量。
- 延迟（Latency）：服务响应速度，关键百分位（如 P95、P99）是否满足预期。
- 错误率（Errors）：如 HTTP 5xx 的占比或其他业务错误率。
- 工单（Tickets）：工单数量、响应时间、处理时间、解决率、重复率、满意度、知识库更新频率等。

#### 2.2.2 SLO（示例与时效指标）

SLO 是对 SLI 的具体目标化。示例：
- 容量：支持峰值 10w QPS。
- 延迟：TP99（请求响应时间的第 99 百分位）应 ≤ 100 ms（或具体业务要求）。
- 可用性：4 个 9（99.99%）等。
- 响应与恢复目标示例：1 分钟发现、5 分钟定位、10 分钟恢复。

关键时间类指标：
- MTTI（Mean Time To Identify）：平均故障发现时间，从故障发生到开始感知/识别/响应的时间。
- MTTK（Mean Time To Know）：平均故障定位时间。
- MTTF（Mean Time To Fix）：平均故障恢复（解决）时间，从定位到采取恢复措施的时间。
- MTTV（Mean Time To Verify）：平均故障修复验证时间，验证服务真正恢复所需时间。

工单类 SLO 示例：月度工单≤xxx，工单平均响应时间≤xx，等（根据实际场景设定）。

#### 2.2.3 SLA（收益与战略意义）

SLA 的价值包括：
- 增强运行态的确定性：让产品从“能正常对外提供服务”提升到“能主动控制服务状态”。
- 帮助产品明确短板，以最优资源投入产出比提升服务质量（例如评估将可用性从 99.9% 提升到 99.99% 的成本与收益比）。
- 提高突发情况应对能力：合适的 SLI 能指导故障时的决策（在资源不足时选择放弃哪些服务）。
- 提供差异化服务，节省成本：明确服务级别使客户在构建时权衡风险与成本。
- 建立用户预期：SLA 可以帮助用户对服务质量有合理预期，避免过度依赖。

错误预算（Error Budget）：用于平衡稳定性与创新迭代速度。错误预算 = 1 - SLO，例如 99.99% 可用性对应 0.01% 的错误预算。

在确立目标后，下面从“根基”和“支柱”两方面介绍如何推进稳定性建设。

## 03 根基

### 3.1 文化建设

文化建设是稳定性建设之基。系统不断迭代、人员流动，如果没有良好文化传承，无法保证稳定性投入和效果。良好的文化像稳定平衡的生态，帮助团队成员以有效方式学习和成长，保障业务健康发展。文化建设可分为“形”“信”“行”三个阶段（理念、信任、落地执行）。

### 3.2 稳定性建设的要素

稳定性建设离不开“人、管理、技术”三要素。

#### 3.2.1 “人”

个人成长：
- 稳定性建设是全员参与的过程。参与者包括研发、测试、运维、安全、产品等，主导方一般为研发、测试和运维。
- 负责人需具备责任心与单兵作战能力。
- 学习金字塔理论表明团队学习、主动学习和参与式学习效果更好。提升稳定性能力的有效方式是成为稳定性布道者。

团队建设：
- 团队活力比单兵能力更可持续。参考高效能团队模式（支持快速交付的组织架构）：
  - 流动式团队（Stream-aligned Team）：面向价值流，快速交付。
  - 赋能团队（Enabling Team）：具备专业技能，指导流动式团队。
  - 复杂子系统团队（Complicated-Subsystem Team）：维护需要专业知识的子系统。
  - 平台团队（Platform Team）：面向底层服务，降低认知负荷，建设基础设施与工具。

#### 3.2.2 “管理”

稳定性工作需要各方配合，应从管理角度制定覆盖工单、监控、预警、故障、应急、变更、演练、压测、度量、问责等的规范，并通过机制保证执行，形成全生命周期管理流程。

规范（静态、明确）示例：
- 方案设计规范（设计原则如 SOLID、KISS 等）
- 编码规范（安全性、容错、可维护性）
- 测试用例设计规范、测试数据管理规范、测试环境管理规范、缺陷管理规范、测试报告规范、自动化测试规范、安全性测试规范
- 变更申请、评估、审批、实施、文档、沟通与安全规范

机制（动态、过程性）示例：
- 评审机制：方案评审、代码评审、COE Review 等，用于发现潜在问题、提高质量、减少风险、促进知识共享、改进流程。
- 值班机制：制定日/周/月值班流程，提高服务连续性与责任感，优化资源分配。
- 复盘机制：故障和项目复盘，形成改进闭环。
- 记账机制：业务与资源登记，避免遗忘。
- 例会机制：定期稳定性会议（周会、月报）用于协调与沉淀。
- 权限管控机制：高危操作双重检查，防止未授权操作。
- 应急响应机制：建立应急预案、响应团队与通报机制。
- 培养与考核机制：专项训练与考核相结合，提升团队能力与积极性。

#### 3.2.3 “技术”

技术是实现稳定性的手段，涉及多个领域：
- 研发流程域：迭代管理、代码扫描等
- 测试仿真域：单元测试、灰度环境、自动化测试、性能压测等
- 运维变更域：代码发布、配置管理等
- 数据域：采集、预处理、存储、计算、分析
- 监控巡检域：监控系统、巡检、舆情监控等
- 事件流程域：事件管理、流程审批、工单系统、故障管理等

## 04 支柱

支柱部分细化稳定性在不同生命周期的具体工作：预防、发现、定位、恢复、复盘。

### 4.1 预防

稳定性建设的大部分工作是预防。真实业务场景与环境多变，采取积极措施预防或减轻潜在风险损失非常重要。

#### 4.1.1 梳理摸底

梳理是高性价比工作，帮助识别系统瓶颈、风险和提升空间。梳理包括系统梳理与业务梳理，回答“现状是什么？”与“不足（风险）是什么？”两个核心问题。

业务梳理流程示例：
- 确定梳理目标（如了解流程、发现痛点、优化）
- 收集流程信息（流程图、说明、相关表单）
- 挖掘问题（流程不顺畅、效率低、错误率高）
- 制定改进措施（优化流程、简化操作、提高自动化）

#### 4.1.2 容量保障

容量是稳定性的关键。容量保障包括容量需求分析、容量调度和容量供给。

容量需求（行业峰值场景举例）：
- 电子政务：政策发布、大型活动
- 出行：节假日、春运
- 电商：大促（双十一等）
- 直播：秒杀、直播活动
- 物流：促销季、展会
- 金融：政策、市场波动、年终结算
- 能源、零售等的高峰场景

容量调度要点：
- 业务流量预估：基于历史数据与实时监控预测冲击流量。
- 系统容量评估：按业务场景计算所需服务节点数与资源。
- 压力测试与决策支持：通过压测验证弱点并支持容量决策。
- 调度策略：弹性调度、分时调度、优先级与隔离策略。
- 风险管理：注意容量调度器配置风险与安全保障。
- 核算与资源管理：动态分配资源避免浪费，提高利用率。

容量供给渠道：
- 私有云：内部维护，适合对安全与控制要求高的场景。
- 公有云：第三方按需付费，弹性强。
- 混合云：结合公有云与私有云的优势，兼顾扩展性与控制。

#### 4.1.3 可观测建设

可观测性通过指标（Metrics）、日志（Logging）、链路（Tracing）来衡量系统状态。建设目标是全面收集、分析与利用运行数据，实现实时监控、快速定位与优化改进。

可观测包含：
- 指标采集与告警（关键指标、阈值、聚合）
- 日志收集、存储与检索（保证日志覆盖、规范、可追溯）
- 链路追踪（TraceId 贯穿请求以还原调用链）
- 可视化大盘与运行态分析

#### 4.1.4 预案与灾备

预案要点：
- 触发条件：明确何时执行预案（监控、告警或用户反馈触发标准）。
- 执行步骤：角色分工明确（如 Owner 执行回滚、其他人定位并查看监控）。
- 观察效果：配置观察大盘并记录结果。

灾备：通过灾备技术保证业务不中断、数据不丢失（备份策略、异地容灾、切换流程等）。

#### 4.1.5 专项治理

专项治理用于发现并解决系统突出问题（性能瓶颈、安全漏洞、数据不一致等），提升系统稳定性与用户体验。示例项目：
- 日志治理：覆盖、规范化、存储、分析、搜索、可视化、审计与监控。
- 监控治理：确保监控覆盖度、准确性与有效性；采用分层监控减少噪声。
- 服务治理：慢 SQL 优化、幂等设计、解耦、读写分离、缓存策略等。
- 数据安全与资金安全：从业务场景、流转环节、风险视角构建安全体系。
- 代码清理专项：降低技术负债，提高可读性、可测试性和可维护性。

### 4.2 发现

及时发现问题是稳定性的关键环节。

#### 4.2.1 常规巡检

检查内容示例：
- 硬件检查：电源、风扇、温度、磁盘空间、网络接口等。
- 软件更新与安全补丁：操作系统与应用补丁、杀毒与防火墙规则更新。
- 数据库与备份：数据库状态、备份流程验证与恢复性测试。
- 对账：主动发现数据差异。
- 安全性与访问控制：身份认证、授权机制与漏洞修复。

#### 4.2.2 监控告警

建立全面的监控告警矩阵，覆盖业务链路、系统资源、容量指标与异常检测，支持多级告警与告警去噪。

#### 4.2.3 客诉响应与舆情感知

舆情从客诉发展，需构建自动化感知流程：
- 数据收集：自动采集新闻、社交媒体、论坛、博客、视频平台等数据源。
- 事件监测：关键词监测与 NLP，快速发现热点与危机。
- 情感分析：识别文本情感倾向并触发预警。
- 竞品分析：跟踪竞争对手舆情变化，支持决策。
- 画像分析：分析传播路径、节点、用户画像与话题倾向。

#### 4.2.4 智能感知

利用 NLP、目标检测、目标跟踪、行为分析与异常检测等技术，实现自动化监控与预警，提高发现精度与效率，降低人力成本。

### 4.3 定位

定位围绕监控、日志与链路进行，结合统计与建模方法提升异常检测与根因定位能力。

#### 4.3.1 监控、日志、链路分析

指标异常检测方法：
- 阈值法：固定阈值或同环比阈值。
- 统计方法：基于标准差（2/3 倍法）、IQR（1.5 倍）等。
- 建模方法：机器学习/统计模型、模型评估、优化与部署，持续监控模型表现。

日志：记录离散事件（调试信息、错误信息），是诊断问题的重要依据。

链路追踪：在分布式系统中为每次请求分配 TraceId，记录各调用点的链路日志，通过 TraceId 组装还原请求调用链轨迹，便于定位延迟与错误来源。

#### 4.3.2 场景复现

场景复现流程：
- 收集历史故障数据（类型、现象、影响范围）。
- 分析故障原因，找出根本因素。
- 构建与原始故障相似的环境与配置。
- 在构建场景中复现故障（模拟输入或操作）。
- 验证解决方案并观察效果。

#### 4.3.3 根因分析（RCA）

常用方法：
- 5W1H：Who、What、When、Where、Why、How。
- 鱼骨图（Ishikawa）：将可能因素分门别类分析。
- 5 Whys：反复问“为什么”直至根因。
- 故事线分析：按时间线拆解事件。
- 树状图分析：建立问题与影响因素的关系图（指标、告警、变更关联）。
- DMAIC：Define、Measure、Analyze、Improve、Control（六西格玛方法）。

### 4.4 恢复

故障处理是对系统与团队能力的一次检验。快速有序的恢复依赖于前期的预案、演练与工具链支持。

### 4.5 故障改进

#### 4.5.1 故障复盘

海因里希法则指出：每一起严重事故背后往往伴随多次轻微事故与隐患。通过故障复盘挖掘深层原因，能够将风险扼杀在萌芽中。复盘应形成改进清单并跟踪执行。

#### 4.5.2 改进验收

改进时关注：
- 及时性：尽快改进以防复发。
- 针对性：措施要对准根本原因。
- 可行性：考虑实施难度与成本。
- 预防性：采取预防措施避免类似问题。
- 持续性：将改进纳入长期治理与监控。

#### 4.5.3 故障演练

建立故障库：每次故障的发生、处理与复盘都应沉淀为演练素材，通过演练提高系统与团队能力。

演练目标：
- 检查预案的完整性与有效性。
- 锻炼队伍与提升响应效率。
- 磨合跨部门联动机制。

混沌工程：通过有控制的实验验证系统在故障场景下的表现，频繁演练可增强系统的反脆弱性。正如《反脆弱》所述：我们不仅要在不确定性中生存，还要从中获益，使系统能够驯化并掌控不确定性。

至此，本文从故障生命周期（预防、发现、定位、恢复、复盘）对稳定性建设的各项工作做了综述性介绍。

## 05 总结

稳定性建设是长期且持续的过程，需要不断改进与完善。通过全面分析、科学方案与有效实施，才能确保系统的稳定性与可靠性，为业务发展提供有力保障。道阻且长，行则将至。
