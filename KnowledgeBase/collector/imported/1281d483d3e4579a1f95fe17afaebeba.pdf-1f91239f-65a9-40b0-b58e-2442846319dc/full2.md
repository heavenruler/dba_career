# 盛天网络 TiDB 落地复盘：从 MySQL 瓶颈到高效运营的架构升级之路

作者：吴志国｜盛天网络 DBA

导读  
泛娱乐行业中，业务运转对数据处理的依赖日益加深，数据库的性能表现、稳定程度与扩展能力，直接决定了业务响应速度与用户体验。盛天网络在游戏社交业务推进中，遭遇数据量快速增长与统计分析需求复杂化的挑战，经过评估，最终选择并落地 TiDB。本文根据作者在 TiDB 地区活动的分享，梳理 TiDB 的选型逻辑、架构设计细节、实战问题解决方法、版本升级过程与实际使用收益，并给出游戏行业数据库选型建议，供技术实践参考。

## 一、盛天网络业务与 TiDB 应用场景

盛天网络长期深耕泛娱乐领域，业务覆盖数字媒体内容运营、游戏开发与发行、娱乐社交平台搭建等。TiDB 的应用重点集中在游戏社交板块，尤其是“带带”产品线的统计分析场景——涉及用户背包物品信息、礼物赠送与消费记录、钱包收支流水、活动派奖明细以及订单生成与履约情况等核心交互数据，均需通过 TiDB 实现高效存储与查询。

## 二、盛天网络选择 TiDB 的核心逻辑

在数据库架构演进过程中，最初使用 MySQL。随着业务快速扩张、数据量累积，MySQL 的性能瓶颈逐渐显现：复杂查询耗时严重、查询超时或导致业务系统抖动，影响用户体验与运营效率。面对数据持续增长与统计分析需求深化，盛天网络综合考量业务需求、开发体验、运维保障与生态活跃度后，选择 TiDB。

主要考虑因素包括：

- 业务场景需求：数据量持续增加和复杂统计分析查询（例如按用户维度统计特定时间段订单量、礼物消费总额）在 MySQL 下耗时长、容易超时。TiDB 的分布式架构能应对大规模数据存储与高并发查询，缓解性能瓶颈。
- 高可用性：游戏社交业务需 7x24 小时运行，TiDB 原生分布式高可用设计（多副本存储、节点故障自动检测与恢复）可确保在单节点下线或网络短暂中断等异常情况下稳定运行，保障 RTO 在分钟级、RPO 接近 0。
- 兼容性：TiDB 高度兼容 MySQL 协议、功能与语法，迁移成本低，开发团队通常只需修改 JDBC 连接地址即可切换，大幅降低改造工作量，实现平滑过渡。
- 生态与社区：TiDB 提供 DM（Data Migration）工具支持 MySQL 到 TiDB 的全量+增量数据同步；集成 Prometheus 与 Grafana 进行监控告警；社区活跃，便于在遇到数据同步或性能问题时快速获取解决方案。

## 三、TiDB 系统架构与实战经验

### （一）系统架构设计

盛天网络的 TiDB 系统由 MySQL 集群、DM 集群与 TiDB 集群三部分构成，形成从数据产生到同步、存储、查询的完整链路。

- MySQL 集群：作为业务生产库，采用微服务架构下的分库设计，按业务模块拆分为用户库、订单库、活动库等。
- DM 集群：连接 MySQL 与 TiDB，承担数据流转任务。关键组件包括：
  - DM-ctl：DM 集群的命令行管理工具，负责创建数据迁移任务、配置数据源信息以及查看任务运行状态。
  - DM-master：协调者，管理 DM 集群拓扑，监控 DM-worker 运行状态，确保各节点协同工作。
  - DM-worker：数据同步的执行者，通过模块化实现全量+增量同步。
- TiDB 集群：数据存储与查询分析的核心，由 TiDB 服务器、TiKV 存储节点与 PD 节点组成。
  - TiDB 服务器：接收应用层 SQL 请求，解析 SQL 并生成执行计划。
  - TiKV：分布式 KV 存储引擎，按 Region 分片存储数据，支持分布式事务，保证数据读写一致性。
  - PD：负责集群元数据管理（如 Region 分布信息）与调度决策（如节点负载高时自动迁移 Region），保障集群高效运行。

### （二）实战经验分享

在落地过程中，团队遇到并解决了若干具体问题，积累了可复用经验。

1. 分表合并同步  
   由于 MySQL 采用按“年月”分表策略，落地时需定期清理历史分表，并在 TiDB 中仅同步某年某月之后的分表以避免无用历史数据占用存储。团队通过配置 DM 的 block-allow-list 正则规则实现精准匹配，从而仅同步需要的分表。

2. 同步 DDL 报错  
   在一次数据同步中，DM 报错并中断。排查发现问题源于 TiDB 与 MySQL 在字符串校验规则处理上的差异：MySQL 允许默认值“0”以非 UTF8MB4 格式存储，而 TiDB 要求默认值格式与字段字符集一致。该 DDL 语句不影响核心业务数据，团队选择跳过该语句以恢复同步。针对 DM 6.0 及以后版本，执行如下命令快速恢复同步流程：

   ```bash
   tiup dmctl --master-addr 10.0.8.183:8261 handle-error task_mysql2tidb skip
   tiup dmctl --master-addr 10.0.8.183:8261 binlog skip task_mysql2tidb
   ```

## 四、TiDB 升级历程

随着业务对性能与稳定性的要求提高，盛天网络对 TiDB 版本进行升级，目标是优化架构、提升资源利用率并保障数据安全。

升级原因  
- 优化架构并提升资源利用率：此前需全量同步 MySQL 的 1000 多张表，导致网络带宽与磁盘 IO 负载过高。升级后可以根据业务需求只同步必要表，降低资源消耗。  
- 满足数据安全合规需求。

升级步骤（概要）  
- 下载离线版本包及 TiUP、TiUP Cluster 等工具安装包。  
- 根据业务调整配置文件参数。  
- 检查当前集群运行状态，排除基础故障。  
- 停止数据同步进程与写入操作，避免升级中数据冲突。  
- 启动升级程序并实时监控进度，确保过程可控。  
- 验证数据同步是否恢复正常。  
- 执行核心业务 SQL 查询，全面确认系统功能与性能稳定性。

升级注意事项（五项要点）  
1. 研读官方升级文档并明确版本差异，确认中间版本是否有必要的配置调整；若使用 TiFlash，需关注其升级特殊要求。参考文档： https://docs.pingcap.com/zh/tidb/v7.5/upgrade-tidb-using-tiup/  
2. 根据业务特性选择升级方式：停服升级可缩短时间（约 2 小时）但会中断业务；在线升级耗时约 6–8 小时但可保障业务连续性。团队最终选择在凌晨 2 点至 8 点的业务低峰期执行在线升级。  
3. 提前处理兼容性变更：例如 v6.5.9 中部分 v5.4 版本默认启用的参数被调整为默认禁用，需要预先核对并处理。  
4. 升级期间严禁执行 DDL 与备份操作：在确认无 DDL 语句执行且无备份任务运行后才启动升级，防止写入与升级冲突导致集群故障。  
5. 在测试环境完成全流程验证：TiDB 在原有服务器上升级通常不支持回退，需在测试环境模拟升级流程并验证关键业务功能（如订单查询、礼物赠送）、性能指标（如查询耗时）与数据一致性，确认可行后再在生产环境执行。

## 五、TiDB 使用收益与体验

迁移至 TiDB 并完成版本升级后，盛天网络在系统性能与运维效率上均有显著提升，解决了此前的业务痛点。

- 系统性能飞跃：查询耗时大幅降低。以游戏社交中“大神用户”订单统计场景为例，某 SQL 在 MySQL 中返回 100 条结果需 34.03 秒，而在 TiDB（8 核 16G 虚拟机）中仅需 3.48 秒，性能提升近 10 倍。  
- 运维效率提升：DDL 操作性能显著优化。升级前，对 1100 万行表执行 DDL 需 30 多分钟；升级至 TiDB v6.5.9 后，对 3 亿行数据的“订单表”执行相同的加索引操作仅需 7 分钟，且不会影响业务连续性。这意味着运维团队无需再仅在业务低峰期执行 DDL，可在白天正常工作时段处理，大幅降低运维压力。

## 六、游戏行业数据库选型建议

结合实践经验与游戏行业业务特点，总结 TiDB 相较于 MySQL、PostgreSQL 等传统开源 RDBMS 在游戏关键业务场景中的优势：

- 原生自愈高可用设计，简化架构并降低故障风险。  
- 灵活的水平扩展能力，适配游戏业务流量波动。  
- 高效的 OLAP 实时查询能力，满足游戏运营的数据分析需求。

选型建议要点：  
- 以业务需求为根本，优先保障可靠性与稳定性（数据不丢失、服务稳定运行是首要考量），再评估性能与功能。  
- 关注产品生态完善度与社区支持能力：完善生态应包含周边管理工具以降低开发运维成本，活跃社区能帮助解决特殊技术问题。  
- 关注开源协议的友好性以避免后续商业风险。  
- 版本选择上建议“稳定优先”，优选市场验证过的版本，因为 bug 少、稳定性更有保障。  
- 结合自身业务规模（如 DAU、数据量）与技术团队能力（是否熟悉分布式架构）做出适配决策，并在落地过程中注重测试与优化，以充分发挥数据库价值。

结语  
TiDB 在盛天网络的落地实践，不仅解决了游戏社交业务的数据库痛点，也为泛娱乐行业的分布式数据库应用和架构升级提供了参考案例。随着数据量和业务复杂度的持续增长，分布式数据库将成为越来越多企业的选择，但选型需结合自身实际并做好充分的测试与优化，才能真正支撑业务的持续发展。
